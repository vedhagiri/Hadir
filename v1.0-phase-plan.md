# Hadir — v1.0 Phase Plan

**Purpose:** Step-by-step Claude Code prompts to build Hadir v1.0 on top of the delivered pilot. This file lives at the repo root alongside `pilot-plan.md` (now historical) and is the authoritative sequence for v1.0.

**Audience:** Suresh and any future Claude Code session resuming mid-build.

**Scope source:** BRD §7–§15, Appendix A (deferred-from-pilot list), and PROJECT_CONTEXT.md §8 and §13.

**Last updated:** Day 0 of v1.0 (pilot delivered, `v0.1-pilot` tagged).
**Status:** Ready to execute.

---

## Branching model

- `release/pilot` — maintenance branch. Tracks `v0.1-pilot`. Only touched for hotfixes Omran reports against the pilot.
- `main` — active v1.0 development. All phases below commit here.
- Milestone tags: `v1.0-m2`, `v1.0-m3`, `v1.0` at the end.
- Optional: `release/v1.0-m2` and `release/v1.0-m3` branches if you want a pre-hardening safety net.

To demo the pilot at any point: `git checkout v0.1-pilot`. To return to work: `git checkout main`.

---

## How to run these prompts

1. One prompt per Claude Code session. Don't paste two in a row.
2. Every prompt ends with "stop and show me". No auto-continue.
3. Commit after every approved session on `main`. Conventional messages: `feat(P5): per-tenant branding with curated slots`.
4. Review before committing: run the app, check DB state, walk the review checklist.
5. Keep `CLAUDE.md` current — every prompt instructs Claude Code to update it at the end.
6. If a session runs long, stop at the logical checkpoint and re-prompt. Don't balloon to 6 hours.

---

## Red lines (same as pilot, reinforced for v1.0)

These override any prompt that would compromise them.

- **Tenant isolation is real, not decorative.** In multi mode, no query may run without a resolved `tenant_id`. Cross-tenant reads are a security bug, full stop.
- **Biometric data encrypted at rest** — face crops and embeddings. RTSP credentials Fernet-encrypted. Passwords Argon2.
- **Audit log is append-only** at the DB grant level (`hadir_app` gets INSERT + SELECT only). This survives from pilot P2; never relax it.
- **Super-Admin cross-tenant access is always audit-logged.** Every session-switch writes an entry with actor + target tenant + scope.
- **No speculative design changes.** The design archive in `design-reference/` is authoritative. Copy CSS verbatim. Don't "improve" the look.
- **No scope creep outside this plan.** If a prompt tempts Claude Code to expand beyond its stated goal, stop and push back. Add the extra to a later phase.
- **Entra ID claims do not imply roles.** Roles are assigned inside Hadir. OIDC gives identity, not authorization.

## M2 — Core (Weeks 2–7)

### P1 — Multi-tenant routing switch (~3 hours)

**Goal:** The same codebase runs in `single` mode (pilot behaviour, one `main` schema) and `multi` mode (schema-per-tenant). In multi mode, tenant is resolved per-request from the session claim.

**Prompt:**

> Session **P1 — Multi-tenant routing switch**. Read `CLAUDE.md`, `v1.0-phase-plan.md`, and `docs/phases/P0.md`.
>
> In this session:
>
> 1. The `HADIR_TENANT_MODE` env var was introduced in pilot P1 as `single`. Wire it up to actually branch behaviour:
>    - `single`: all queries run in schema `main` (no change from pilot).
>    - `multi`: queries run in the schema named for the active tenant (e.g. `tenant_omran`, `tenant_demo`).
> 2. **Session claim:** extend `user_sessions.data` JSONB to carry `tenant_id` and `tenant_schema`. On login, resolve the user's tenant and write both into the session.
> 3. **Request middleware:** add `TenantScopeMiddleware` that reads the session, resolves schema, and sets a `SET search_path TO {schema}, public` on the connection for the duration of the request. Use SQLAlchemy events or an explicit dependency injection — whichever is cleaner. Document the choice in `backend/CLAUDE.md`.
> 4. **Super-Admin override hook:** if the session has `impersonated_tenant_id`, use that instead. (The UI for this arrives in P3; this just prepares the hook.)
> 5. **Test setup:** add a `multi`-mode pytest fixture that provisions two schemas (`tenant_a`, `tenant_b`) with minimal data and verifies that a query scoped to one never returns the other's rows. This test is the canary — if it fails in any later phase, tenant isolation is broken.
> 6. Backward compatibility: all existing pilot tests must still pass in `single` mode.
>
> Red lines: no query may execute without a resolved tenant. If a code path reaches the DB without `search_path` set, that's a bug. Crash loud in development; fail closed in production.
>
> When both modes boot and the isolation test passes — commit as `feat(P1): multi-tenant routing`, update `docs/phases/P1.md`, **stop, and show me**. Do not start P2.

**Review checklist:**
- `HADIR_TENANT_MODE=single` boots identically to pilot
- `HADIR_TENANT_MODE=multi` with two seeded schemas isolates queries
- Isolation pytest passes
- No query in logs runs without a `search_path`

### P2 — Per-schema Alembic + tenant provisioning CLI (~3 hours)

**Goal:** Adding a new tenant is a single command. Migrations run per-schema. No manual SQL.

**Prompt:**

> Session **P2 — Per-schema Alembic + tenant provisioning CLI**. Read `CLAUDE.md` and confirm P0–P1 are complete.
>
> In this session:
>
> 1. **Alembic per-schema:** configure Alembic to track a version table per schema. On `alembic upgrade head`, iterate all tenant schemas (discovered from the `tenants` table in the `public` schema) and apply pending migrations to each. The `public` schema holds only the `tenants` table and a global `alembic_version`.
> 2. **Move `tenants` table** from `main` schema to `public` schema (new migration). This is the only globally-visible table. Everything else (users, employees, cameras, etc.) lives per-tenant.
> 3. **Provisioning CLI:** `backend/scripts/provision_tenant.py`
>    - Args: `--slug tenant_omran --name "Omran" --admin-email hr@omran.om --admin-password <prompted>`
>    - Creates the schema
>    - Runs all migrations on the new schema
>    - Seeds default roles (Admin, HR, Manager, Employee), default departments, one default Fixed policy
>    - Creates the first Admin user with Argon2-hashed password
>    - Prints summary; exits non-zero on any failure with cleanup
> 4. **Deprovisioning CLI:** `backend/scripts/deprovision_tenant.py --slug tenant_x --confirm` — drops the schema after a text confirmation. Document in `backend/CLAUDE.md` that this is destructive and should never run in production without a DB backup taken in the same session.
> 5. **Migration authoring rule:** going forward, all new migrations must be schema-agnostic (no hardcoded schema names). Enforce via a pytest that inspects each migration file for forbidden patterns.
>
> Red lines: the `public` schema holds `tenants` and nothing else. Deprovisioning must take a backup first in production (script refuses otherwise when `HADIR_ENV=production`).
>
> When you can provision a new tenant from scratch with one command and the isolation test from P1 still passes — commit as `feat(P2): per-schema alembic + tenant provisioning CLI`, update `docs/phases/P2.md`, **stop, and show me**.

**Review checklist:**
- `python -m backend.scripts.provision_tenant --slug tenant_demo --name "Demo Co"` creates a working tenant
- `alembic upgrade head` applies to all tenant schemas
- Deprovisioning refuses without `--confirm` and without a fresh backup in production

### P3 — Super-Admin role + console (~4 hours)

**Goal:** MTS staff can log in as Super-Admin, see all tenants, "Access as" any tenant for support, every access audit-logged.

**Prompt:**

> Session **P3 — Super-Admin role + console**. Read `CLAUDE.md` and confirm P0–P2 are complete.
>
> In this session:
>
> 1. **New role `SuperAdmin`** — only valid in `multi` mode. Seed via provisioning: one MTS staff user lives in a reserved `mts_staff` table in the `public` schema (not per-tenant). Authenticates separately.
> 2. **Super-Admin login** at `/super-admin/login` — separate from tenant login. On success, session has `is_super_admin=true` but no `tenant_id` until impersonation begins.
> 3. **Super-Admin console** at `/super-admin/*` (different layout from tenant shell — red accent bar at top to make it obvious this is privileged context):
>    - **Tenants list:** name, slug, created, admin count, employee count, status (active/suspended). Port layout from `design/pages.jsx` where applicable.
>    - **Tenant detail:** branding (placeholder, wired in P4), admin users, schema stats, recent audit events involving this tenant.
>    - **"Access as" button:** on click, set `impersonated_tenant_id` in the session, redirect to the tenant shell `/`. A persistent banner at the top of the tenant UI reads "Viewing as SuperAdmin — click to exit impersonation". Exiting clears the impersonation.
>    - **Provision tenant** button: web UI wrapper around the P2 CLI. Validates slug format (lowercase, hyphens, no spaces). Calls the provisioning code in-process.
> 4. **Audit:** every Super-Admin action writes an audit entry — login, logout, tenant viewed, "Access as" start and end, tenant provisioned, tenant suspended. Tenant-context audit entries write to both the tenant's `audit_log` AND a global `public.super_admin_audit` table. This duplication is intentional: tenants must see that they were accessed.
> 5. **Never** allow a Super-Admin session to write to tenant data without the "Access as" impersonation flow. Reads outside impersonation are OK for the console views (summary counts, etc.). Writes require impersonation and are audited both places.
>
> Red lines: every cross-tenant touch audit-logged. The red banner during impersonation is not cosmetic — it's a safety feature. Never remove it "because it's ugly".
>
> When you can log in as Super-Admin, list tenants, provision a new one from the web UI, "Access as" into it (with banner visible), and see the audit entries in both places — commit as `feat(P3): super-admin role + console`, update `docs/phases/P3.md`, **stop, and show me**.

**Review checklist:**
- Super-Admin login separate from tenant login
- Tenant list and detail render; provisioning from UI works
- "Access as" switches context with red banner
- Audit entries appear in both the tenant log and `public.super_admin_audit`

### P6 — Entra ID OIDC (~4 hours)

**Goal:** Users authenticate via Microsoft Entra ID in production. Local email+password still works as break-glass. Role assignment remains in Hadir, not derived from Entra groups.

**Prompt:**

> Session **P6 — Entra ID OIDC**. Read `CLAUDE.md` and confirm P0–P5 are complete.
>
> In this session:
>
> 1. **Per-tenant OIDC config** (Alembic migration, per-tenant schema):
>    - `oidc_config` (tenant_id PK, entra_tenant_id, client_id, client_secret_encrypted, enabled, updated_at)
>    - `client_secret` encrypted with Fernet (same key as RTSP from pilot P7, or a separate auth-scoped key — use `HADIR_AUTH_FERNET_KEY`, new env var).
> 2. **Authlib integration** in `backend/hadir/auth/oidc.py`:
>    - `GET /api/auth/oidc/login` — initiates the OIDC flow, state + nonce in signed cookie
>    - `GET /api/auth/oidc/callback` — handles the redirect, validates state, exchanges code for token, reads `email` claim
>    - Match on lowercased email against `users.email`. If no match, return 403 with message: "Your Microsoft account is not registered in Hadir. Contact your administrator." **Do not auto-provision users from OIDC claims.**
>    - On match, create a session identical to local auth.
> 3. **Tenant resolution:** OIDC sign-in needs to route to the correct tenant. Support two patterns:
>    - Subdomain: `omran.hadir.example.com` → tenant_omran. The middleware reads host and resolves tenant before OIDC kickoff.
>    - Tenant picker: `hadir.example.com/login` shows a tenant slug input first, then redirects to OIDC for that tenant. Subdomain is preferred; picker is fallback.
> 4. **Login page updates:** when the tenant has OIDC enabled, show "Sign in with Microsoft" as the primary action. Local email+password becomes a secondary link labelled "Use local account" — intended for break-glass only. Warn if both are unavailable.
> 5. **OIDC configuration UI:** Admin settings → Authentication tab. Fields: Entra tenant ID, client ID, client secret (write-only, shows `***` on read like RTSP), enable toggle. Save triggers a validation ping to the Entra discovery endpoint — refuse to save if the config is invalid.
> 6. **Audit:** OIDC login (success + failure), OIDC config change, fallback to local auth all audit-logged. Failure audit records email attempted and reason (no user match / invalid state / expired code).
>
> Red lines: never auto-provision from OIDC. Never derive roles from Entra groups. Never log the client secret or the access token. Role management stays fully within Hadir per BRD FR-AUTH-006.
>
> When Omran's Admin can configure OIDC, sign in via Entra, and the session works identically to local login — commit as `feat(P6): entra id oidc`, update `docs/phases/P6.md`, **stop, and show me**.

**Review checklist:**
- OIDC login works end-to-end against a real Entra tenant (use the MTS internal tenant for test)
- Unregistered email → clear 403 message, not auto-created
- Client secret never appears in logs or API responses
- Local auth still works as break-glass

### P8 — Manager assignments UI (~3 hours)

**Goal:** Admin drags employees between manager buckets. Backend stores primary-manager flag per BRD Open Item Q9.
