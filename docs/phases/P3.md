# P3 — Super-Admin role + console

**Phase:** v1.0 P3 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P3 — Super-Admin role + console"`

## Goal

MTS staff log in to a separate console (URL + cookie), see every
tenant in `public.tenants`, "Access as" any tenant for support, and
have every cross-tenant touch audit-logged in BOTH the tenant's own
`audit_log` and a new global `public.super_admin_audit`. The red
visual treatment is part of the safety story — operators must always
know they're in the privileged context.

## What was done

### Migration 0009 — three new globals + `tenants.status`

`alembic/versions/0009_super_admin.py`. Adds three tables in `public`
and one column on `public.tenants`:

* `public.mts_staff` — Muscat Tech Solutions operators. Independent
  of every per-tenant `users` table; CITEXT email; Argon2id
  password_hash; `is_active`. Owner: `hadir_admin`. `hadir_app`
  full CRUD.
* `public.super_admin_sessions` — opaque-token sessions for
  operators. `data` JSONB carries `impersonated_tenant_id` once an
  "Access as" is active. `mts_staff_id` FK CASCADE. Mirrors the
  shape of per-tenant `user_sessions`.
* `public.super_admin_audit` — append-only operator log.
  `super_admin_user_id` FK RESTRICT, `tenant_id` FK SET NULL
  (nullable for actions that don't target a specific tenant —
  login, logout). `hadir_app` has **INSERT + SELECT only**, mirroring
  the per-tenant `audit_log` contract.
* `public.tenants.status` (new column): `active` | `suspended`,
  CHECK-constrained, default `active`. Used by the console and by
  the access-as guard (suspended tenants refuse impersonation).

The migration uses `CREATE TABLE IF NOT EXISTS` everywhere so the
orchestrator's loop (which runs `alembic upgrade head` once per
tenant schema) is a no-op for tenants 2..N once it has applied the
revision against `main`. 0009 is added to the migration-lint
whitelist with a docstring justifying that operating on `public` is
intrinsic to its purpose.

### `hadir/super_admin/` package

* `sessions.py` — opaque-token CRUD on `public.super_admin_sessions`,
  sliding expiry, `set_impersonation(session_id, tenant_id)`. Cookie
  name `hadir_super_session` (distinct from `hadir_session` so a
  browser can hold both at once).
* `audit.py` — direct writer for `public.super_admin_audit` plus a
  `write_audit_dual` helper used at the router-level for actions
  that target a tenant (provisioning, suspend, access-as start/end).
* `dependencies.py` — `current_super_admin` reads the super cookie,
  validates against `mts_staff`, slides expiry, and stashes
  `is_super_admin` + `super_admin_user_id` + `super_admin_session_id`
  + `client_ip` on `request.state`.
* `repository.py` — read helpers used by the console: `list_tenants`
  (returns name, slug, status, admin_count, employee_count;
  per-tenant counts by switching `tenant_context` per query),
  `get_tenant_detail` (admin users + recent operator-audit rows),
  `update_tenant_status` (suspend/unsuspend).
* `router.py` — `/api/super-admin/*` endpoints:
  - `POST /login` / `POST /logout` / `GET /me`
  - `GET /tenants` / `GET /tenants/{id}` (audited as
    `super_admin.tenant.viewed`)
  - `POST /tenants` (provisioning — wraps the P2 CLI's
    `provision()` function in-process; failure path emits
    `super_admin.tenant.provision_failed`)
  - `POST /tenants/{id}/access-as` /
    `POST /exit-impersonation` (audited as
    `super_admin.access_as.start` / `super_admin.access_as.end`)
  - `POST /tenants/{id}/status` (audited as
    `super_admin.tenant.suspended` / `super_admin.tenant.unsuspended`)

Every action emits a `public.super_admin_audit` row carrying the
operator's IP. Provisioning, access-as, and suspend also carry a
`tenant_id` — login/logout don't.

### Middleware + `current_user` integration

`TenantScopeMiddleware` (`hadir/tenants/middleware.py`) gained a
super-admin priority branch:

1. **Super-admin session with active impersonation** → resolve the
   impersonated tenant's `schema_name` from `public.tenants`, apply
   it as the request's tenant context, and set
   `request.state.is_super_admin`, `super_admin_user_id`,
   `impersonated_tenant_id`. Also installs the
   `_super_admin_context` ContextVar so every `write_audit` call
   during this request dual-logs.
2. **Super-admin session without impersonation** → schema = `public`
   so console queries land; tenant_id is `None`, so any per-tenant
   write endpoint will 401 at `current_user`.
3. **Tenant session** (P1 path, unchanged).
4. **Anonymous** (single-mode default to `main`, multi fail-closed).

`current_user` (`hadir/auth/dependencies.py`) now returns a synthetic
`CurrentUser(id=0, roles=("Admin","HR","Manager","Employee"), …)`
when the request has `is_super_admin=True` and `tenant_id` set but
no `hadir_session` cookie. This lets every existing role guard
(`require_role`, `require_any_role`, `require_department`) permit
the operator's request without a per-route change.

### Audit dual-write

`hadir/auth/audit.py` is the load-bearing change. It now:

1. Holds a per-request `_super_admin_context: ContextVar[dict|None]`
   set by the middleware on impersonation.
2. Translates `actor_user_id <= 0` to `NULL` (the synthetic
   Super-Admin's `id=0` would FK-violate `users`).
3. When `_super_admin_context` is set, augments the tenant log row's
   `after` with `impersonated_by_super_admin_user_id` AND inserts a
   paired row into `public.super_admin_audit` carrying the operator
   id, tenant id, and client IP.

Net effect: every existing `write_audit(...)` call site dual-logs
during impersonation without touching any router. Tenant Admins
reading their own `audit_log` see the impersonation marker; the
global operator log holds the cross-tenant history independently.

### `/api/auth/me` impersonation flag

`MeResponse` (`hadir/auth/router.py`) gained
`is_super_admin_impersonation: bool` and `super_admin_user_id: int |
None`. The frontend's tenant shell uses the flag to mount the red
banner.

### `scripts/seed_super_admin.py`

Idempotent CLI for the first MTS staff user. Mirrors `seed_admin.py`
shape (`--email`, `--full-name`, `$HADIR_SUPER_ADMIN_PASSWORD`).
Argon2id-hashed; never echoes the password.

### Frontend (`frontend/src/super-admin/`)

* `types.ts` — wire types mirroring the Pydantic responses.
* `SuperAdminProvider.tsx` — TanStack Query hooks
  (`useSuperMe`/`useSuperLogin`/`useSuperLogout`/`useTenants`/
  `useTenantDetail`/`useProvisionTenant`/`useAccessAs`/
  `useExitImpersonation`/`useUpdateTenantStatus`).
* `SuperAdminLogin.tsx` — `/super-admin/login`, distinct red-bordered
  card on the same neutral background.
* `SuperAdminLayout.tsx` — `/super-admin/*` shell. Red 6px accent bar
  + sticky red-bordered header reading **MTS Operator Console** with
  Tenants and Provision tenant nav.
* `SuperAdminProtectedRoute.tsx` — mirrors the tenant
  `ProtectedRoute` but for the super-session probe.
* `TenantsListPage.tsx` — table with name, slug, created, admin
  count, employee count, status, per-row Access-as button (disabled
  when status='suspended').
* `TenantDetailPage.tsx` — branding placeholder (P4), stats, admin
  users, recent operator-audit table, Access-as + Suspend/Reactivate
  actions.
* `ProvisionTenantPage.tsx` — `/super-admin/provision`. Form fields
  validated against the same regex as the DB CHECK
  (`^[a-z_][a-z0-9_]{0,62}$` — lowercase letters, digits,
  underscores). Wraps the in-process provisioning endpoint. On
  success redirects to the new tenant's detail page.

### Persistent impersonation banner

`frontend/src/shell/ImpersonationBanner.tsx` mounts at the top of
the tenant `Layout` whenever `useMe()` returns
`is_super_admin_impersonation=true`. Full-width red bar reading
**⚠ Viewing as SuperAdmin (operator #N) — click to exit
impersonation**. Click → `POST /api/super-admin/exit-impersonation`
→ redirect to `/super-admin/tenants`.

`Layout.tsx` is the only existing tenant-shell file changed; it now
renders the banner above the existing sidebar/topbar/content
hierarchy.

### Routes

`App.tsx` registers:
- `/super-admin/login` (public)
- `/super-admin` → `Navigate /super-admin/tenants`
- `/super-admin/tenants` (list)
- `/super-admin/tenants/:tenantId` (detail)
- `/super-admin/provision` (form)

All under `<SuperAdminProtectedRoute>` + `<SuperAdminLayout>`.

## Notes for whoever resumes next

- Per-tenant **branding** (P4) lands inside the existing
  `TenantDetailPage`'s "Branding" placeholder card. Wire the same
  form into the tenant Admin's Settings page.
- The middleware's tenant-cookie path still reads sessions from
  `main.user_sessions`. Login from non-`main` tenants needs a
  follow-up to either (a) embed the tenant_schema into the session
  token, or (b) carry the schema as a separate cookie at login.
  Either way, the super-admin flow already routes correctly because
  it uses its own cookie + global session table.
- The "actor_user_id ≤ 0 → NULL" translation in `write_audit` is the
  tax we paid for the synthetic-user shim. If a future phase adds
  a real `super_admin.user_id` column to `audit_log`, that
  translation can go away in favour of writing the operator's id
  directly. For now the impersonation marker on `after` is enough.
- The red banner is **mandatory** when impersonating. Tests do not
  yet enforce it visually; consider a Playwright check that hits
  `/super-admin/tenants/1/access-as`, navigates to `/`, and asserts
  the banner is in the DOM.
