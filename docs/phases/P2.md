**Date (UTC):** 2026-04-25

**Plan reference:** `v1.0-phase-plan.md §"P2 — Per-schema Alembic +

## Goal

A new tenant can be provisioned end-to-end with one command: a fresh
Postgres schema, every per-tenant table materialised inside it,
grants applied, alembic version stamped, defaults seeded, and the
first Admin user created — all inside one transaction with cleanup on
failure. The global `tenants` registry moves from `main` to `public`,
and migration tracking is per-schema so future schema-agnostic
migrations apply uniformly to every tenant.

## What was done

### Migration 0008 — boundary cut

`alembic/versions/0008_tenants_to_public.py`. The one-shot migration
that crosses schemas:

- `CREATE TABLE public.tenants` mirroring `main.tenants` shape from
  P1 (id, name, schema_name, created_at) plus the regex CHECK +
  UNIQUE constraints from migration 0007.
- `INSERT INTO public.tenants SELECT * FROM main.tenants` preserves
  the pilot tenant's id (Omran, id=1, schema_name='main') so existing
  FK column values stay valid.
- `setval(pg_get_serial_sequence('public.tenants', 'id'), MAX(id))`
  advances the new sequence past the highest existing id.
- A `DO $$ … $$` block discovers every FK in
  `information_schema.table_constraints` whose target is
  `main.tenants(id)`, drops each, and re-creates it pointing at
  `public.tenants(id)`. Constraint names are preserved verbatim;
  `delete_rule` is read from `information_schema.referential_constraints`
  and re-applied so RESTRICTs aren't silently flattened to NO ACTION.
  All 14 FKs (one per per-tenant table — users, roles, user_roles,
  departments, user_departments, user_sessions, audit_log, employees,
  employee_photos, cameras, detection_events, camera_health_snapshots,
  shift_policies, attendance_records) rewired in a single statement.
- `DROP TABLE main.tenants` (no CASCADE — every dependent FK was
  rewired, so the drop succeeds cleanly; CASCADE would mask any
  missed constraint).
- `ALTER TABLE public.tenants OWNER TO hadir_admin` +
  `GRANT SELECT, INSERT, UPDATE, DELETE ON public.tenants TO
  hadir_app` + sequence USAGE/SELECT.
- `downgrade()` is the reverse, refusing if any non-`main` tenant
  exists in `public.tenants` (would orphan a tenant_<slug> schema).

### `hadir/db.py` — single MetaData, public-baked tenants

- `tenants` Table is back in the primary `metadata` (single MetaData
  so per-tenant tables can resolve their FK target string against
  it), but with `schema="public"` baked into the Table constructor.
- All per-tenant tables now reference `ForeignKey("public.tenants.id",
  ...)` (was `"tenants.id"` in P1).
- `metadata_global` is a documented alias for `metadata` so
  `alembic env.py` and call sites that conceptually want the "global"
  set have a name for it.

### `alembic/env.py` — per-schema target

- Reads `-x schema=<name>` from the alembic CLI; defaults to `main`
  for backward compat.
- `version_table_schema` is set to that schema, so each tenant gets
  its own `<schema>.alembic_version` cursor.
- Issues `SET search_path TO "<schema>", public` on the connection
  before calling `context.run_migrations()`, so unqualified table
  references in 0009+ migrations land in the active tenant schema.
- Creates the schema with `CREATE SCHEMA IF NOT EXISTS` first so a
  fresh DB run can write its `alembic_version` row without a separate
  bootstrap step.

### `scripts/migrate.py` — orchestrator

Replaces the entrypoint's bare `alembic upgrade head`:

1. `alembic -x schema=main upgrade head` first — advances the pilot
   from 0007 to 0008 (the boundary) and any future legacy upgrades.
   On a fresh DB this also creates `main` + 0001-0007 so that step 2
   has a `public.tenants` to iterate.
2. Reads `public.tenants` (skipping if the table doesn't exist yet
   on first boot before 0008 has applied) and runs
   `alembic -x schema=<name> upgrade head` for every non-`main` row.

Subprocess invocation (rather than `alembic.command.upgrade`) so
each run reads `-x` from a fresh CLI invocation and version state
can't leak across iterations. `entrypoint.sh` switched to
`python -m scripts.migrate`.

### `scripts/provision_tenant.py`

Single transaction, fail-closed. Validates slug against the same
regex the DB CHECK enforces, requires a non-empty admin email +
password (read from `--admin-password`, then
`$HADIR_PROVISION_PASSWORD`, then an interactive `getpass` prompt
with confirmation — never on argv). Sequence:

1. INSERT `public.tenants` row → captures the new tenant_id.
2. `CREATE SCHEMA <slug>` + `SET search_path TO <slug>, public`.
3. `metadata.create_all(bind=conn, tables=[t for t in metadata.tables.values() if t.schema != "public"])`
   materialises every per-tenant table inside `<slug>`. The
   `t.schema != "public"` filter is the one place the global
   `tenants` table is actively excluded — keeping it in the same
   `MetaData` made FK resolution work; filtering keeps `create_all`
   from re-grant-ing it.
4. Mirror the pilot's grants on `<slug>`: `hadir_admin` owner,
   `hadir_app` SELECT/INSERT/UPDATE/DELETE on every table EXCEPT
   `audit_log` (INSERT + SELECT only — the append-only contract is
   preserved per tenant).
5. Seed: 4 roles, 3 departments, one Fixed shift policy, one Admin
   user (Argon2id-hashed password), and one `tenant.provisioned`
   audit row.

After commit, runs `alembic -x schema=<slug> stamp head` as a
subprocess so `<slug>.alembic_version` is at the latest revision and
future 0009+ migrations only apply incrementally. If the stamp
subprocess fails, `_cleanup` drops the schema and removes the
`public.tenants` row before the script exits non-zero. There is no
half-provisioned middle state on disk.

### `scripts/deprovision_tenant.py` — destructive

Refuses without `--confirm`. Refuses in `HADIR_ENV=production`
without `--backup-taken` (no way for the script to verify a backup
was taken — the flag is a hard checkpoint on operator discipline).
Refuses to drop the pilot schema `main`. Without `--yes-i-know`,
prompts the operator to re-type the slug on stdin before proceeding.

`DROP SCHEMA <slug> CASCADE` + `DELETE FROM public.tenants WHERE id =
<id>` in one transaction. Logs the expected encrypted-crops paths
under `/data/faces/<id>/` and `/data/faces/captures/<id>/` so the
operator can clean the volume separately — the script intentionally
doesn't touch the disk volume.

### Migration-authoring lint

`tests/test_migration_lint.py`. Scans every file under
`alembic/versions/` and fails the suite if a migration outside the
whitelist contains any of:

- `schema='main'` / `schema='public'`
- `SCHEMA = 'main'` / `SCHEMA = 'public'`
- `"main".` / `"public".`
- `main.<table>` / `public.<table>` (regex)

Whitelist is a hardcoded `frozenset` of {0001, 0002, 0003, 0004,
0005, 0006, 0007, 0008}. Adding a new entry requires a deliberate
code edit + review — the goal is for the whitelist to never grow.
A second test (`test_whitelist_only_contains_known_legacy_migrations`)
makes sure the whitelist doesn't drift away from the actual files on
disk.

## Notes for whoever resumes next

- v1.0 P3 needs to address the **session→tenant resolution** problem
  P2 deliberately deferred. Right now `TenantScopeMiddleware` reads
  session rows from `main.user_sessions` only. After P2,
  `user_sessions` is per-tenant, but the cookie is opaque so the
  middleware can't tell which schema to look in. The P2 prompt's red
  line ("public schema holds tenants and nothing else") rules out a
  global session-index table in public, so the fix is one of:
  embedding the tenant_schema in the session token / cookie, or
  carrying the schema as a separate cookie set at login. Either way,
  this is a v1.0 P3 (auth/session refactor) task — single-mode pilot
  keeps working until then.
- The migration-lint whitelist is `{0001..0008}`. Going forward 0009+
  must be schema-agnostic. The lint will catch hardcoded `main` /
  `public` literals; if a new genuinely cross-schema migration is
  ever needed, add it to the whitelist with a docstring justifying
  why.
- Provisioning's `_create_per_tenant_tables` filters by
  `Table.schema != "public"`. Any new `Table` definition that
  belongs in the global registry MUST set `schema="public"` on the
  Table itself — otherwise the provisioning CLI will try to recreate
  it inside every tenant schema and the redundant rows will diverge.
