# P8 — Manager assignments UI

**Phase:** v1.0 P8 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P8 — Manager assignments UI"`

## Goal

Admin can directly assign individual employees to managers, on top
of (not in place of) department membership. The Manager's
visibility on the daily attendance list — and every later
Manager-scoped surface — becomes the **union** of their department
employees plus the directly-assigned ones. Exactly one of those
links can be marked "primary" per employee, and the partial unique
index that enforces that rule lives in Postgres so a buggy direct
INSERT can't slip through.

Hard red line (the prompt's own): the primary-manager constraint
is enforced at the DB level (partial unique index), not just
application logic.

## What was done

### Migration 0012 — `manager_assignments`

Per-tenant table:

| Column | Notes |
| --- | --- |
| `id` (PK) | autoinc serial. |
| `tenant_id` | FK to **unqualified** `tenants(id)` so the migration is schema-agnostic. |
| `manager_user_id` | FK to `users(id)`, ON DELETE CASCADE. |
| `employee_id` | FK to `employees(id)`, ON DELETE CASCADE. |
| `is_primary` | bool, default false. |
| `created_at` / `updated_at` | tstz, default now(). |

Constraints:

* `UNIQUE (tenant_id, employee_id, manager_user_id)` — a manager
  appears at most once per employee.
* **Partial unique index** —
  ``CREATE UNIQUE INDEX … ON manager_assignments (tenant_id,
  employee_id) WHERE is_primary = TRUE``. The DB-level enforcement
  the prompt's red line calls out.

Grants (`hadir_app` full CRUD on the table + sequence USAGE) match
the existing per-tenant pattern. Provisioning's
`_APP_CRUD_TABLES` extended.

### `hadir/manager_assignments/` package

* `schemas.py` — Pydantic shapes mirroring the API.
* `repository.py`:
  - `list_managers_with_employees(scope)` — returns every Manager
    user with their assignment chips, including department codes
    inline so the UI can show "Manager Eng (ENG)" without a second
    round-trip.
  - `list_unassigned_employees(scope)` — active employees with no
    rows in `manager_assignments`.
  - `set_assignment(scope, employee_id, manager_user_id, is_primary)`
    — clear-prior-primary then upsert, both in the caller's
    transaction so the partial unique index never sees an
    intermediate violation.
  - `delete_assignment(scope, assignment_id)`.
  - **`get_manager_visible_employee_ids(scope, manager_user_id)`**
    — the load-bearing scope helper. Returns the union of
    (employees in the manager's departments) ∪ (employees in
    `manager_assignments` for that manager).
* `router.py`:
  - `GET /api/manager-assignments` — Admin-only. Returns the
    grouped `{managers: [...], unassigned: [...]}` shape the UI
    consumes.
  - `POST /api/manager-assignments` — Admin-only. Validates the
    manager user is an active Manager + the employee is active.
    Captures the prior-primary id (if any) so the audit row
    records the swap, then calls `set_assignment`.
  - `DELETE /api/manager-assignments/{id}` — Admin-only.

Audit actions (per-tenant `audit_log`):

* `manager_assignment.created`
* `manager_assignment.primary_set` (carries
  `prior_primary_assignment_id` so the demoted row is visible).
* `manager_assignment.deleted`

### Manager scope union into attendance

`hadir/attendance/repository.py::list_for_date` learns an
`employee_ids: list[int]` kwarg. When supplied, results are
restricted to that set; combined with `department_ids` it
intersects, so an Admin-style department filter still narrows
visibility further.

`hadir/attendance/router.py` Manager branch:

* Calls `get_manager_visible_employee_ids` to build the union of
  dept-members + direct assignments.
* Empty union → 200 with `items: []` (the manager genuinely sees
  nothing — no widening to the full tenant view).
* `?department_id=` query param continues to refuse departments
  outside the manager's `user_departments` (no widening), now
  layered on top of the union.

The pilot's "Manager only sees their department" behaviour is
preserved when no direct assignments exist; it only widens when
an Admin links an out-of-dept employee.

### Frontend (`frontend/src/manager-assignments/`)

* `types.ts` — wire types mirroring the Pydantic shapes.
* `hooks.ts` — `useAssignments`, `useCreateAssignment`,
  `useDeleteAssignment` (TanStack Query).
* `ManagerAssignmentsPage.tsx` — Admin-only page. Layout: left
  column "Unassigned employees" + a responsive grid of manager
  cards on the right. Native HTML5 drag-and-drop (no new deps):
  - Each chip is `draggable`; `dataTransfer.setData(MIME, JSON)`
    carries `{employee_id, source, source_manager_id, source_assignment_id}`.
  - Manager cards + Unassigned column are drop targets — they
    inspect `dataTransfer.types.includes(MIME)` in `dragover`,
    flip a hover style, and call POST/DELETE on drop.
  - Drag from Manager A to Manager B → POST under B then DELETE
    the A row inside one effect (visually a move; under the hood
    it's two API calls but ordering keeps the partial unique
    index honest because B's POST defaults `is_primary=false`).
  - Drop on Unassigned → DELETE.
  - Star icon (`★` / `☆`) on a chip in a manager card toggles
    `is_primary` via the same POST endpoint; the
    clear-prior-primary inside `set_assignment` makes the swap
    safe.

Wired into `App.tsx` at the existing `mgr-assign` nav id, dropping
the pilot's Placeholder there. The sidebar nav already exposes
"Manager assignments" for Admin and HR; the page itself remains
Admin-only on the API side (HR users see a 403 if they navigate
in).

### Provisioning + grants

`scripts/provision_tenant.py::_APP_CRUD_TABLES` extends to include
`manager_assignments` so a freshly-provisioned tenant gets the
right grants from day one.

## Notes for whoever resumes next

- The drag-from-Manager-A-to-Manager-B path issues two API
  calls (POST to B, then DELETE of A). The partial unique index
  rules out a race where two primaries co-exist (the new POST
  defaults `is_primary=false`), but if either step fails the
  user is left in a half-applied state. The page invalidates the
  TanStack query on success of either, so a refresh recovers.
  Worth a tiny optimistic-UI pass when we touch this again.
- The seeded matrix's `.test` admin can't drive this UI from a
  browser without loosening EmailStr — same quirk noted in
  P6.md / P7.md. The live smoke above used `*.hadirsmoke.io`.
- Approvals (P15+) and Reports (P17+) will want the same
  Manager-scope union. Both should call
  `get_manager_visible_employee_ids` instead of re-deriving the
  set from `user.departments`.
- Department-only managers behave exactly as in the pilot —
  empty direct-assignments set means the union reduces to
  department membership.
