## Goal

Cross-cutting in-app + email notifications for the six BRD events
(approval_assigned, approval_decided, overtime_flagged,
camera_unreachable, report_ready, admin_override). Per-user
preferences with sensible defaults (both channels on). The email
channel reuses the P18 provider stack; the in-app channel feeds a
topbar bell + a full history page.

## What was done

### Migration 0021

`alembic/versions/0021_notifications.py`. Replaces the P16 stub
`notifications_queue` (dropped — nothing in production depended on
it; M2 still in dev) with two real per-tenant tables:

| Table | Notes |
| --- | --- |
| `notifications` | id, tenant_id, user_id, category (CHECK on the 6 enum values), subject, body, link_url, payload JSONB, read_at, email_sent_at, email_attempts, email_failed_at, email_error, created_at. Indexes on (tenant_id, user_id, read_at) for the bell + (tenant_id, email_sent_at, email_failed_at) for the worker drain. |
| `notification_preferences` | composite PK (user_id, tenant_id, category) + in_app + email + updated_at. Missing row = both true (resolved in code). |

### `db.py` + provisioning

Two new `Table` objects; the P16 `notifications_queue` reference
removed. `_APP_CRUD_TABLES` updated.

### `maugood/notifications/`

* `categories.py` — pure enum + human labels.
* `repository.py` — list / mark-read / mark-all-read /
  preference-resolver / set-preference, plus the worker support
  helpers (`list_pending_email`, `mark_email_sent`,
  `mark_email_failed`, `mark_email_skipped`).
* `producer.py` — typed wrappers operators + producers call:
  `notify_user`, `notify_approval_assigned`,
  `notify_approval_decided`, `notify_admin_override`,
  `notify_overtime_flagged`, `notify_camera_unreachable`,
  `notify_report_ready`. The in-app preference is honoured at
  insert time (no row written if `in_app=False`); the email
  preference is the worker's job.
* `worker.py` — APScheduler 30-second tick. Per active tenant:
  read `email_config` (skip when disabled and stamp the row as
  skipped to keep the queue drained), pull pending rows, **re-
  resolve** the preference per row (so a flip during the tick
  takes effect), build the email via the new
  `notification.html` template, send via the P18 provider, mark
  sent / skipped / failed.
* `camera_watch.py` — sister tick (mounted in the existing P18
  scheduler) that scans `camera_health_snapshots` for cameras
  whose latest snapshot is `reachable=false` AND whose outage
  exceeds 5 minutes. Dedupes via the most-recent
  `camera_unreachable` notification per camera so a long outage
  fires once.
* `router.py` — `GET /api/notifications`,
  `POST /api/notifications/{id}/mark-read`,
  `POST /api/notifications/mark-all-read`,
  `GET /api/notification-preferences`,
  `PATCH /api/notification-preferences`. All
  authenticated-user-scoped.

### Producers wired

* **Request submit (P14)** — `notify_approval_assigned` to the
  primary manager.
* **Manager-decide (P13)** — approve → `notify_approval_assigned`
  to every HR user; reject → `notify_approval_decided` to the
  employee (terminal).
* **HR-decide (P13)** — both decisions are terminal →
  `notify_approval_decided` to the employee.
* **Cancel (P13)** — `notify_approval_decided` to the employee
  with `decider_label="You"` (operator-friendly self-trace).
* **Admin override (P16)** — replaced the P16 stub queue with
  `notify_admin_override` (Manager + HR + Employee, deduped on
  user_id).
* **Attendance recompute (P10/P11)** — new `_maybe_notify_overtime`
  fires `notify_overtime_flagged` only when `overtime_minutes`
  flips from 0 to > 0 (a "first time today" gate via
  `existing_overtime_minutes`). Manager (direct
  manager_assignments rows) + every HR user receive it.
* **On-demand reports (P13/P17)** — both `attendance.xlsx` and
  `attendance.pdf` endpoints fire `notify_report_ready` to the
  caller.
* **Camera unreachable (P8)** — the scheduler tick scans for
  outages > 5 min and fires `notify_camera_unreachable` to every
  Admin user.

### Email template

`backend/maugood/emailing/templates/notification.html` — minimal
inline-styled HTML email, tenant-branded header (accent hex +
name), subject + body + optional "Open in Maugood" button, and a
plain-text unsubscribe note pointing the recipient at Settings →
Notifications (deliberately not a link, per the BRD-friendly
"text-only" copy).

### Frontend `frontend/src/notifications/`

* `NotificationBell.tsx` — topbar bell with unread badge +
  dropdown panel showing the last 20 notifications, "Mark all
  read", and a "See all notifications" link. Clicking a row
  marks it read and follows `link_url`. Polls
  `/api/notifications` every 30 seconds + on focus.
* `NotificationsPage.tsx` — full history at `/notifications`.
* `PreferencesPage.tsx` — Settings → Notifications. Category ×
  channel grid; defaults are surfaced when no row exists.
* Bell mounted in `Topbar.tsx` between the role chip and the
  identity name.
* Routes + tab wired in `App.tsx` and `SettingsTabs.tsx`.

### Lifespan + tests

Backend `main.py` starts the worker on lifespan and stops it on
teardown. `tests/conftest.py` neutralises the worker (and the
existing schedulers) so TestClient(app) lifespan entries don't
spawn 30-second job threads.

## Notes for whoever resumes next

- **Camera-unreachable dedupe.** The watcher fires once per
  outage, identified by "no `camera_unreachable` notification
  exists since the camera's last `reachable=true` snapshot." If
  the camera flickers (alternates reachable/unreachable inside
  one polling minute), each outage longer than 5 minutes fires
  fresh — that matches operator expectations from the BRD.
- **HR auto-routing on approval.** The approve-→-HR producer
  notifies every active HR user. If the tenant grows past a
  handful of HR users, add a department-scoped routing column on
  the request to narrow this — the producer accepts an iterable.
- **Preference UX.** The grid is checkbox-based (in_app + email
  per category). A future "snooze for 24 hours" affordance fits
  cleanly via a per-pref `snoozed_until` column without changing
  the wire shape.
- **Camera-unreachable producer cost.** The scan runs once per
  60-second tick across every tenant. At pilot scale this costs
  one query per camera per minute — fine. If a tenant adds
  hundreds of cameras, swap the per-camera "last reachable"
  lookup for a window-function query that returns the outage
  start in one round-trip.
- **`.test` TLD gotcha.** Same as P11–P19 — the smoke uses
  `.maugood`.
- **Email body fidelity.** The notification template is plain
  HTML inline-styled for Outlook / Gmail compatibility. If a
  category gains a richer payload (attachments, structured
  tables), extend `render.py` with a per-category template
  rather than overloading `notification.html`.
