## Goal

Ship a working backup + restore pipeline, fire it on a
schedule, and prove the restore actually works on a throwaway
host. Drop the result on disk in the form of a per-rehearsal
log + a durable RTO/RPO statement so the next operator can
defend the targets to Omran IT.

## What was done

### `backend/scripts/backup.sh`

Writes a self-describing backup directory under
`${HADIR_BACKUP_ROOT:-/backup}/<UTC-TIMESTAMP>/`:

* `db/<schema>.sql.gz` — `pg_dump --schema=NAME` per schema in
  the source: `public` (global registry + super-admin tables),
  `main` (legacy / pilot tenant), and every row of
  `public.tenants` whose `schema_name <> 'main'`. Each dump
  uses `--no-owner` so the destination cluster doesn't try to
  ALTER OWNER TO a missing role; privileges (the GRANTs that
  let `hadir_app` see the tables) are preserved.
* `data/{faces,attachments,branding,erp,reports}.tar.gz` —
  tarballs of `/data/<sub>/`, skipped when empty so an
  operator without a feature in use doesn't pay the tarball
  cost.
* `manifest.json` — `manifest_version`, `created_at`,
  `pg_server_version`, the schema list, and one entry per
  file with `path`, `size_bytes`, `sha256`. Restore consumes
  only the fields it needs and tolerates extras.
* `_complete` — a marker file written *after* everything else
  so retention can tell complete runs from crashed ones.

Retention: keep the most recent N daily, the most recent N
weekly (highest run per ISO week), the most recent N monthly
(highest run per calendar month). Defaults from the BRD —
`HADIR_BACKUP_RETAIN_{DAILY,WEEKLY,MONTHLY}` override.

Optional off-site upload: `HADIR_BACKUP_S3_URI=s3://bucket/prefix/`
triggers `aws s3 cp --recursive` after the local copy is
verified. Requires `INCLUDE_AWS_CLI=1` at image-build time —
default is OFF so the image stays small for the local-NAS path
(BRD Open Item Q4 default).

### `backend/scripts/restore.sh`

Drives the round trip. Behaviour:

1. **Checksum gate** — every file in the manifest is sha256'd
   and compared to the manifest entry. A single mismatch
   aborts before any destructive SQL is issued.
2. **Blast-radius probe** — counts `public.tenants` rows and
   non-system schemas. A non-empty target triggers either:
   * Typed `RESTORE` confirmation read from `/dev/tty`
     (case-sensitive; a script can't bypass).
   * `--yes-i-have-a-backup-of-the-target` flag for non-
     interactive rehearsals; production refuses the flag
     unless the operator passes it explicitly (the typed
     prompt is the manual path).
3. **Drop in reverse dependency order** — per-tenant schemas
   first (`DROP SCHEMA <name> CASCADE`), then `main`, then the
   Hadir tables on `public`. The schema `public` itself stays
   put because the citext extension lives there.
4. **Restore in forward dependency order** — `public`
   (filtering the dump's `CREATE SCHEMA public` line so it
   doesn't conflict with the existing schema), then `main`,
   then every per-tenant schema alphabetically.
5. **Restore data tarballs** — wipe and extract each subtree.
6. **Probe** — log tenant count + active main user count.

### `ops/backup/`

Container that runs the backup on a schedule:

* **Dockerfile** — `postgres:15-alpine` + `bash`, `coreutils`
  (gnu sha256sum), `tar`, `gzip`, `jq`, `tzdata`, `ca-certs`,
  + `supercronic` (pinned binary, GitHub release URL).
* **crontab** — `0 2 * * * /app/scripts/backup.sh`. Container
  TZ defaults to `Asia/Muscat` so the cron entry's local time
  matches BRD-required 02:00.
* **entrypoint.sh** — runs supercronic by default, OR runs
  `backup.sh` as a one-shot when invoked with `once`. Both
  paths validate the script + crontab presence before exec —
  a typo fails container start, not silently in production.

### `docker-compose.prod.yml`

Adds the `backup` service:

* Builds the `ops/backup/Dockerfile` with the script
  bind-mounted in (so updates ship without an image rebuild).
* Mounts `${HADIR_BACKUP_DIR:-backup_storage}` at `/backup`
  for the destination, and `faces_data:/data:ro` so tar can
  snapshot the live volume read-only (the load-bearing red
  line for the source artifacts — the script never mutates
  them).
* Pulls every backup-tunable env var via compose-level
  interpolation: `HADIR_BACKUP_S3_URI`,
  `HADIR_BACKUP_RETAIN_*`, `HADIR_BACKUP_TZ`. Defaults match
  the script's defaults.
* `restart: unless-stopped` so supercronic comes back after
  host reboots.

### `docs/disaster-recovery.md`

Durable RTO/RPO statement: **RTO 4 hours, RPO 24 hours** for
the pilot defaults. Documents the future-work path —
continuous archiving via wal-g or pgBackRest — that tightens
both numbers to RTO 1 hour / RPO 5 minutes once it ships in
v1.x. Also covers hot standby, encryption-key escrow, and
off-site copy verification.

The doc explicitly is a policy + roadmap; it doesn't quote
unshipped behaviour as delivered.

### Live DR rehearsal — outcomes

Source data shape: 3 tenants, 7 main users, 614 audit rows, 7
demo users, 10 omran users, plus 868 KB of `/data` content
(faces + attachments + reports — branding + erp empty).

Backup wall-clock: **1 second**. On-disk: **~560 KB**
(56 KB DB dumps + 488 KB tarballs).

Restore round trip:

* Stop backend on the throwaway target — 2 s.
* Verify checksums + drop + restore + extract — 1 s.
* Start backend on the restored data — 11 s.
* **Total RTO actual: ~13 seconds.**
* **RPO actual: 0** (backup was just taken).

Post-restore verification: logged in via API on the throwaway
target. `/api/auth/me` returned `preferred_theme=dark` and
`preferred_density=compact` — the P22 user-level preferences
survived the round trip. DB row counts matched the source
exactly (the +1 audit row is the verification login itself).

Three issues surfaced and fixed in the same session:

1. `pg_dump --clean --if-exists` emitted `DROP SCHEMA public`
   which cascaded the citext extension. Fix: dropped the
   flags; restore does its own drops.
2. Per-schema drops failed because cross-schema FKs reference
   `public.tenants(id)`. Fix: restore drops every schema
   upfront in reverse dependency order.
3. `pg_dump --no-privileges` stripped GRANTs, leaving
   `hadir_app` with `permission denied for schema main`.
   Fix: dropped the flag; grants round-trip with the dumps.
   `--no-owner` is kept.

Full transcript in `docs/dr-rehearsal.md`.

## Notes for whoever resumes next

* **Script lives in `backend/scripts/`.** That keeps it
  alongside the code it's tightly coupled to (the schema list
  comes from `public.tenants`; the data dirs are what the
  backend writes to). Updates ship without an image rebuild
  because the path is bind-mounted into the backup container.
* **`--no-privileges` is intentionally OFF.** A future phase
  that introduces operator-specific roles in the cluster
  needs to revisit this — if a role exists on the source and
  not on the destination, the GRANT statement in the dump
  errors out. The clean fix is to make the migration system
  the source of truth for both schema *and* roles, and have
  restore re-run the relevant migration steps before applying
  the dumps. P25 hardening territory.
* **Destructive-confirm is the red line.** The typed `RESTORE`
  is read from `/dev/tty` so a `restore.sh < /dev/null` can't
  silently bypass it. The `--yes-i-have-a-backup-of-the-target`
  flag is a checkpoint, not a shortcut — operators sign in
  blood that they've taken a confirmed backup of the *target*
  before destroying its data.
* **Off-site upload is OFF by default.** Operators on the
  local-NAS path (BRD Open Item Q4 default) get nothing extra;
  operators on S3 set `INCLUDE_AWS_CLI=1` at build time +
  `HADIR_BACKUP_S3_URI` at runtime. The script logs a WARN
  (not a FATAL) when the URI is set but the binary isn't
  available — the local copy is still good.
* **Quarterly rehearsal cadence.** `docs/dr-rehearsal.md`'s
  "Schedule for the next rehearsal" section names
  **2026-07-25** as the next due date. Set a calendar
  reminder; don't trust this comment to surface itself.
* **`pg_server_version` in the manifest.** The manifest
  records the source's server version. Restoring to a
  *different* major version will probably work for one major-
  version jump (Postgres maintains backward-compat for the
  pg_dump format), but rehearse before you commit. v1.x
  hardening should add an explicit cross-version compat
  check.
* **Open critical item carries forward**: Omran HR native-
  speaker review of the Arabic translations from P21 is
  still pending before v1.0 launch.
