## Goal

Backend foundation for the exception + leave request workflow that
P14 (submission UI), P15 (approvals inbox), and P16 (admin override
UI) build on top of. Eight-state machine enforced server-side, with
attendance-side effects firing automatically on terminal approval.

## What was done

### Migration 0016

`alembic/versions/0016_requests.py`. Two per-tenant tables:

| Table | Notes |
| --- | --- |
| `requests` | id, tenant_id, employee_id, type ['exception','leave'], reason_category, reason_text, target_date_start, target_date_end nullable, leave_type_id nullable, status, plus per-stage actor + decision-at + comment trios for manager/hr/admin, submitted_at, created_at, updated_at. CHECK on `type`, CHECK on `status` (only the 8 named values), CHECK that `leave` rows carry a `leave_type_id` and `exception` rows do not, CHECK on date range. | `request_attachments` | id, request_id, tenant_id, file_path, original_filename, content_type, size_bytes, uploaded_at. Schema + grants only — the upload UI lands in P14. |

Schema-agnostic — every FK targets unqualified `tenants(id)` so the
migration runs cleanly under every tenant schema.

### `db.py` + provisioning

Two new `Table` definitions added in `hadir/db.py`. Provisioning's
`_APP_CRUD_TABLES` extends to include both tables.

### `hadir/requests/state_machine.py` (pure)

Decoupled from FastAPI and SQLAlchemy. Four functions —
`cancel`, `manager_decide`, `hr_decide`, `admin_override` — each
take the current status + a decision and either return the next
status or raise `InvalidTransition`. The router translates that
exception into HTTP 409 with the message verbatim, so the caller
sees a clear "Cannot HR-decide on a request with status
'manager_rejected' — HR can only act after the manager has
approved." rather than a generic 400.

States and transitions (per the BRD §FR-REQ-* and the prompt):

| From → To | Action | Actor |
| --- | --- | --- |
| `submitted` → `cancelled` | cancel | Employee, own only |
| `submitted` → `manager_approved` | manager_decide approve | Manager (assigned only) |
| `submitted` → `manager_rejected` (terminal) | manager_decide reject | Manager (assigned only) |
| `manager_approved` → `hr_approved` (terminal) | hr_decide approve | HR |
| `manager_approved` → `hr_rejected` (terminal) | hr_decide reject | HR |
| any state → `admin_approved` (terminal) | admin_override approve | Admin |
| any state → `admin_rejected` (terminal) | admin_override reject | Admin |

**Manager rejection is terminal — never escalates to HR.** **HR
decision is final except for Admin override, and override comment is
mandatory** (`Field(min_length=1)` plus a `model_validator` that
rejects whitespace-only). Both red lines tested.

### Repository (`hadir/requests/repository.py`)

* `get_request`, `get_request_by_id` join through `employees` and
  optional `leave_types` so the API row is fully populated without
  N+1.
* List helpers per role: `list_requests_for_employee`,
  `list_requests_for_employee_ids` (Manager — union of assignments +
  own row), `list_requests_for_hr` (everything that ever reached HR
  — `manager_approved` plus the four terminal HR/Admin outcomes),
  and `list_all_requests` (Admin).
* `get_primary_manager_user_id` reads the partial-unique-index row
  from migration 0012 — the same primary-manager rule that powers
  the P8 UI. Used to auto-set `manager_user_id` on submission.
* `is_manager_assigned_to` — gate on the manager-decide endpoint so
  a manager can only act on employees they're assigned to (primary
  or not — the primary rule only governs auto-routing).
* `update_status` — single helper that applies the new status +
  per-stage actor + decision_at + comment in one UPDATE.
* `employee_for_user_email` — same lower-cased email shortcut the
  attendance router uses (until v1.0 wires an explicit user↔employee
  join table).

### Router (`hadir/requests/router.py`)

| Method + Path | Role | Notes |
| --- | --- | --- |
| `GET /api/requests` | any | Role-scoped. Admin sees all; HR sees `manager_approved` + terminal outcomes; Manager sees own + assigned employees' rows; Employee sees own only. |
| `GET /api/requests/{id}` | any | Same role scoping; 403 if the row is invisible to the caller. |
| `POST /api/requests` | Employee | Resolves the submitting employee from session email; auto-resolves `manager_user_id` from primary assignment; 400 if the user has no employee row, 400 if the leave_type_id doesn't belong to this tenant. |
| `POST /api/requests/{id}/cancel` | Employee, own only | 409 if status ≠ submitted; 403 if the row isn't yours (Admin can also cancel anyone's). |
| `POST /api/requests/{id}/manager-decide` | Manager (assigned) | 403 if not assigned to this employee; 409 if status ≠ submitted. |
| `POST /api/requests/{id}/hr-decide` | HR | 409 if status ≠ manager_approved (the manager-rejection terminal red line). |
| `POST /api/requests/{id}/admin-override` | Admin | 422 on empty/whitespace comment; allowed from any state, even terminal rows. |

Audit hook on every transition with before/after status, actor,
comment:
`request.{submitted,cancelled,manager.approve,manager.reject,hr.approve,hr.reject,admin.approve,admin.reject}`.

### Attendance integration

On `hr_approved` or `admin_approved`:

* If `type=leave` — insert one `approved_leaves` row covering the
  date range (idempotent: skips if an identical `(employee, type,
  range)` row already exists from a prior approval).
* For both leave and exception — call the new
  `attendance.scheduler.recompute_for(scope, employee_id, the_date)`
  for **every covered date** (start through end inclusive). This
  helper is a single-employee/single-date version of `recompute_today`
  that explicitly handles past dates, since requests routinely
  cover yesterday and earlier. Failures are logged + skipped per
  date — a recompute hiccup never reverses the decision.

### Router wired into `main.py`

`from hadir.requests import router as requests_router` +
`app.include_router(requests_router)`.

## Notes for whoever resumes next

- **P14 (Submission UI)** consumes `POST /api/requests` and
  `POST /api/requests/{id}/cancel`; the form needs a leave-type
  dropdown sourced from `GET /api/leave-types` and an exception
  reason-category list (free-form for now — P14 is the right place
  to seed a small enum if HR asks).
- **P15 (Approvals inbox)** is mostly a list + decide-button page.
  The role-scoped `GET /api/requests` already returns the right
  rows for each role; for the manager view, the list will need a
  per-row "decide" affordance against `POST
  /api/requests/{id}/manager-decide`.
- **P16 (Admin override)** is the smallest UI: a comment box on
  every non-cancelled row + the `POST
  /api/requests/{id}/admin-override` call.
- **request_attachments** has a table + grants but no upload route.
  P14 picks that up and wires a Fernet-encrypted ingest path in
  the same shape as the P6 employee-photo flow.
- **HR auto-routing.** Today HR sees every `manager_approved` row
  in the tenant. If/when HR is sub-departmentalised (Oman HR vs.
  Muscat HR, e.g.), the `list_requests_for_hr` helper is the
  one-line change-point — narrow it on a future `hr_user_id`
  routing rule rather than the status set.
- **Idempotent approvals.** Re-running an approval (re-POSTing the
  same hr-decide / admin-override in the unlikely retry case)
  guards against duplicate `approved_leaves` rows by checking
  existence first. Attendance recompute is naturally idempotent
  (it's an UPSERT). So a flaky network → retry never produces a
  double-counted leave or a stale attendance row.
- **`.test` TLD gotcha.** Same as P11/P12 — Pydantic `EmailStr`
  rejects reserved TLDs, so the seeded `*.omran.test` users can't
  log in via the API. The smoke uses `*.p13.hadir` for the
  one-off Manager/HR/Admin/Employee accounts and the pilot's
  `admin@pilot.hadir` for the bootstrap.
