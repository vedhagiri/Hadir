# P10 — Policy engine: Ramadan + Custom types

**Phase:** v1.0 P10 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P10 — Policy engine: Ramadan + Custom types"`

## Goal

Extend the policy system from "Fixed + Flex" to "Fixed + Flex +
Ramadan + Custom". Ramadan is functionally a Fixed policy with a
calendar date range (one per year). Custom wraps either a Fixed or
a Flex inner shape plus a date range — used for one-off days
(half-day before a holiday, etc.).

The two new types add **two new tiers at the top of the resolver**:

> Custom > Ramadan > employee > department > tenant > legacy

**Only one policy applies per (employee, date) — no stacking.** The
priority is deterministic and now documented in
`backend/CLAUDE.md §"Policy resolution priority"`.

## What was done

### Engine refactor — Ramadan + Custom dispatch

`maugood/attendance/engine.py`:

* `ShiftPolicy` extended with `range_start` / `range_end`
  (`Optional[date]`) for Ramadan + Custom and a
  `custom_inner_type: Optional[Literal["Fixed", "Flex"]]` for
  Custom.
* `_flags_for(...)` dispatch:
  - `Flex` → `_flex_flags`.
  - `Custom` with `custom_inner_type='Flex'` → `_flex_flags`.
  - Everything else (`Fixed`, `Ramadan`, `Custom-Fixed`) →
    `_fixed_flags`. Ramadan reuses Fixed flag math because its
    shape *is* Fixed; the resolver filters by date range, not the
    engine.
* `policy_from_row` reads `shift_policies.config` and inflates
  the right field set per type (parses `start_date`/`end_date`
  via `date.fromisoformat` for Ramadan + Custom).

The engine **stays pure** — no DB calls. It just gets a richer
input and dispatches one extra branch.

### Resolver — Custom + Ramadan tiers

`maugood/attendance/repository.py::resolve_policies_for_employees`:

* New private `_custom_or_ramadan_for_date(conn, scope, *,
  the_date, policy_type)` helper. Loads candidate
  `shift_policies` rows for the given type + tenant whose
  row-level `active_from`/`active_until` covers the date, then
  filters by `config.start_date` / `config.end_date` in Python.
  Multiple matches → most recent `active_from` wins (the
  per-tenant `ORDER BY active_from DESC LIMIT N` does the work).
* The main resolver runs Tier 0a (Custom) → Tier 0b (Ramadan)
  before any of the existing P9 tiers. A hit at Tier 0a/0b
  applies tenant-wide for that date — every employee resolves to
  the same policy. The remaining P9 cascade only runs when both
  Custom and Ramadan are silent.
* Empty `employee_ids` → empty dict (unchanged).

### Pydantic validation

`maugood/policies/schemas.py::PolicyConfig` now models
`start_date` / `end_date` (ISO strings) plus `inner_type`
(`Literal["Fixed", "Flex"]` or `None`). Two new validators:

* `_check_times` parses both date fields with
  `date.fromisoformat` and refuses inverted ranges
  (`start_date > end_date`).
* `PolicyCreateRequest._check_type_fields` enforces the per-type
  required fields:
  - **Ramadan** needs `start`, `end`, `start_date`, `end_date`.
  - **Custom** needs `start_date`, `end_date`, and `inner_type`,
    plus the matching inner-shape fields (Fixed: `start` + `end`;
    Flex: all four window times).
  - Anything missing → 422 at the API.

### Frontend Policies page

`frontend/src/policies/PoliciesPage.tsx`:

* The single table is now segmented into **three** sections:
  Standard policies (Fixed + Flex), Ramadan policies, Custom
  policies. The Standard section keeps the assignment column and
  the per-row chips from P9; Ramadan + Custom omit it because
  those types apply tenant-wide for any date in their range.
* `describeWindow` learned to summarise Ramadan and Custom rows
  inline — Custom carries `[Flex]` / `[Fixed]` plus the inner
  shape and the date range.
* `PolicyForm` extended:
  - The type dropdown gains `Ramadan` and `Custom`.
  - Picking Ramadan pre-fills the 2026 dates (`2026-02-18` →
    `2026-03-19`, 08:00–14:00, 6h required) so the operator can
    just adjust the year.
  - Picking Ramadan or Custom shows a calendar range picker.
  - Custom adds an `inner_type` dropdown (Fixed / Flex) that
    switches the rest of the form between the Fixed and Flex
    field sets.

`types.ts::PolicyConfig` adds the four new optional fields.

## Notes for whoever resumes next

- **Hijri labelling on the Ramadan editor** — the prompt suggests
  "labelled with the Hijri year if easy". I left it at the
  Gregorian display since (a) Maugood stores Gregorian throughout
  and (b) Python's stdlib has no Hijri converter. If the operator
  asks for it, `hijri-converter` is the standard library and a
  small per-component formatter would do the job — but the
  storage stays Gregorian.
- **Multiple Custom on the same date** — the resolver picks the
  one with the most recent `active_from`. If a tenant ever ships
  two Custom rows covering the same date, the audit trail on the
  rejected one shows up in the operator log; the live resolver
  just picks the newest. If we ever need a hard guarantee of "at
  most one Custom per date" we'd add a partial unique index
  similar to the manager_assignments `is_primary` rule.
- **Ramadan vs department-scoped Flex**: a department might
  legitimately want a *different* Ramadan-style adjustment (e.g.
  Operations stays at 9-hour shifts). P10 doesn't support that —
  Ramadan is tenant-wide. The right way to model that case today
  is: Ramadan policy at tenant level + an employee-scope override
  for the affected employees. Per-department Ramadan would be a
  follow-up.
- **Custom day stacked with leave**: out of scope for P10 — the
  leave system arrives later and integrates at the engine's
  ``leaves`` arg.
