## What was done — runbook trace

### Step 1: prereqs — clean

Already had docker installed. Repo clone was no-op (worked in
the existing checkout). The runbook step is straightforward
and matches a real Ubuntu install path.

### Step 2: generate secrets — **GAP #1** (CRITICAL → FIXED)

The runbook said `python3 -c "from cryptography.fernet import
Fernet; ..."`. A fresh Ubuntu 22.04 host doesn't ship
`cryptography` in stdlib, so the operator gets
`ModuleNotFoundError`.

**Fix landed in `docs/deploy-production.md §2`**: replaced
the cryptography invocation with a stdlib-only equivalent
(`base64.urlsafe_b64encode(secrets.token_bytes(32))`)
producing the exact Fernet-compatible key shape. Also added
the three DB / Grafana password lines that were implicit
before.

### Step 3: write `.env` — **GAP #2 + #3** (medium → noted/fixed)

* Gap #2 (low, deferred B-3): the literal
  `<HADIR_APP_DB_PASSWORD>` placeholder syntax in the
  DATABASE_URL sample is ambiguous — a careless paste-and-edit
  operator could leave it.
* Gap #3 (medium, **fixed in `docs/deploy-production.md §6`**):
  the runbook never explained `HADIR_TENANT_MODE`. A SaaS
  operator deploying multiple tenants needs `multi`; an
  Omran-pilot-style deploy stays on `single`. Added a
  pick-a-path callout at the top of §6 and a §6b multi-tenant
  flow with the `provision_tenant.py` invocation.

### Step 4: TLS cert — clean

The mkcert / self-signed dev path worked first try.
Operator-provided cert path is the default — that's the
right call for Omran's PKI workflow.

### Step 5: boot the stack — **GAP #4** (CRITICAL → FIXED)

`docker-compose.yml` hard-coded `POSTGRES_PASSWORD: hadir`,
but the runbook tells operators to set
`HADIR_ADMIN_DB_PASSWORD` and bake it into the
`HADIR_ADMIN_DATABASE_URL`. Result on first boot: the
postgres bootstrap superuser `hadir` has password `hadir`
(the docker-compose default), Alembic tries to authenticate
with the random password, fails immediately:

```
psycopg.OperationalError: FATAL: password authentication
failed for user "hadir"
```

The two values were *distinct things* the runbook conflated.

**Fix landed in `docker-compose.yml`**: `POSTGRES_PASSWORD:
${HADIR_ADMIN_DB_PASSWORD:-hadir}`. The dev default of
`hadir` keeps unconfigured local-dev `docker compose up`
working; production deployments using the runbook now have
their `hadir` superuser and their `HADIR_ADMIN_DATABASE_URL`
agree from boot zero.

### After step 5 boot — **GAP #5** (CRITICAL → FIXED)

`HADIR_TENANT_MODE=multi` got the migrations to apply, then
crashed during the FastAPI lifespan:

```
RuntimeError: no tenant schema in scope — refusing to issue
queries (HADIR_TENANT_MODE=multi)
```

`capture_manager.start()` ran `engine.begin()` to enumerate
cameras — without a tenant context. The fail-closed listener
(P1 red line) refused.

**Fix landed in `backend/hadir/capture/manager.py`**: in
multi-mode, `start()` now opens `tenant_context("public")`,
reads `public.tenants` for active tenants, then for each one
opens `tenant_context(schema)` to enumerate that tenant's
cameras and spawn workers. Single-mode falls back to the
existing pilot path. `_spawn_locked` accepts optional
`tenant_id` + `schema` so the multi-mode caller can pass
both through cleanly.

### Step 6: provision Demo Co — clean (per the new §6b)

```
[provision] provisioned tenant_id=2 schema=tenant_democo
admin_user_id=1 admin_email=demoadmin@demoapp.local
```

### Login as Demo Co admin — **GAP #6** (low, deferred B-4)

`email-validator` rejected `demoadmin@demoapp.local` with
"special-use or reserved name." Operators on real public
TLDs never hit it; the dev recipe for `.local` does. Patched
the seeded admin's email to `@example.com` and noted on the
backlog (`docs/v1.x-backlog.md` B-4).

### Configuring Demo Co — **GAP #7** (medium, deferred B-1)

There's no API endpoint to create non-Admin users. Demo Co's
operator can stand up the tenant, log in as the seeded admin,
and then has no UI/API path to create HR / Manager / Employee
users. Worked around with `docker compose exec backend
python ...` raw SQL. Logged on backlog (B-1).

### Branding (navy + lato), policies, holidays, cameras — clean

* `PATCH /api/branding {primary_color_key:"navy",font_key:"lato"}`
  → 200, branding applied.
* `POST /api/policies` for the Flex policy required
  `in_window_start`, `in_window_end`, `out_window_start`,
  `out_window_end`, `required_minutes`, `grace_minutes` — the
  schema is fussier than the example-free flow lets on. The
  Settings → Shift Policies UI surfaces the right shape; only
  API-direct callers feel the friction. Acceptable.
* Three holidays inserted via `POST /api/holidays`.
* Two cameras created via `POST /api/cameras` with throwaway
  RTSP URLs. The capture worker tries to connect, logs
  "could not open stream", flips `reachable=false` on the
  per-minute health snapshot, and the
  `hadir_camera_reachable{tenant="2"}` gauge ticks to 0.
  Working as intended.
* Three employees (Anya, Bilal, Carla) inserted via raw SQL
  (workaround for B-1). Manager assignments wired via raw
  SQL (a Settings → Manager Assignments UI exists in the
  frontend; an API path opens up via the same fix).

### Walkthrough — **GAP #8** (medium, deferred B-2)

Calling `recompute_today(TenantScope(tenant_id=2))` from a
shell logged "no active policy resolves for tenant 2 on
2026-04-26" — but a direct probe of
`resolve_policies_for_employees` returned 3 policies. Root
cause: `TenantScope(tenant_id=2)` defaults
`tenant_schema='main'`, and `recompute_today` wraps the
inner work in `tenant_context(scope.tenant_schema)`. The
queries hit `main.*` (Omran's data) and silently returned
empty.

Worked around by calling `_recompute_today_inner` directly
inside an explicit `tenant_context("tenant_democo")` — got 3
upserts. Logged on backlog (B-2). Multi-mode background-job
entry points need a `TenantScope.for_tenant(engine,
tenant_id)` classmethod.

### Metrics + dashboards — **GAP #9** (CRITICAL → FIXED)

Prometheus targets showed `hadir-backend health=down
lastError: server returned HTTP status 421 Misdirected
Request`. P26 wired the HTTPS gate to refuse plain-HTTP
everywhere except `/api/health`. Prometheus scrapes
`http://backend:8000/metrics` over the private docker
network in plain HTTP (TLS terminates at nginx, not the
backend) — the gate 421s every scrape. Result: every
dashboard panel goes empty in production.

**Fix landed in `backend/hadir/security.py`**: added
`/metrics` to `_HTTPS_EXEMPT_PATHS`. Internal-only is
preserved (nginx still doesn't proxy `/metrics`). New
regression test `test_https_gate_exempts_metrics_for_prometheus`
in `test_p23_https.py`.

After the fix:

```
hadir-backend health= up   lastError= 
prometheus    health= up   lastError= 
```

`hadir_active_sessions{tenant="2"} 4.0` rendered in
`/metrics` (the four sessions I'd opened during the
walkthrough). The Hadir — Operations dashboard's "Active
sessions" panel would render `tenant 2` once Grafana's
30-second refresh fired.

  reverted the host to the dev stack via
  `docker compose up -d` (Omran's `main` data untouched
  throughout). To bring Demo Co back up: re-run
  `docker compose --env-file /tmp/p28-demo.env -p hadirdemo
  -f docker-compose.yml -f docker-compose.prod.yml up -d`.

  P28 fixes are backwards-compatible with a single-mode
  deployment. A future migration of Omran's data into a
  per-tenant schema (`tenant_omran`) is a v1.x exercise —
  the orchestrator from P2 already handles that shape.

  "stand-up-from-scratch" rehearsal should run **every
  major release** with a fresh operator who hasn't touched
  the code. The runbook + this phase doc + the v1.x backlog
