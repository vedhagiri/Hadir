# P23 — HTTPS + reverse proxy

**Phase:** v1.0 P23 (M3 hardening — first phase)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P23 — HTTPS + reverse
proxy"`

## Goal

Stand up a production deployment story: nginx fronts the backend
on a private docker network and terminates TLS, the backend
refuses to boot in production unless every TLS prerequisite is
in place, and the cookie / CORS / forwarded-header plumbing all
honours the proxy. The dev stack stays plain HTTP — operators
tune to taste via the optional mkcert recipe in the deployment
runbook.

## What was done

### Backend hardening

`backend/hadir/security.py` (new). Three pieces:

* **`check_production_config(settings)`** — fail-fast guard
  called at the top of `create_app()`. When `env=production`
  it refuses to boot unless the cookie-secure flag is on,
  `behind_proxy=true`, `allowed_origins` is non-empty, the
  OIDC redirect base URL is HTTPS, and every placeholder
  secret (`HADIR_SESSION_SECRET`, `HADIR_FERNET_KEY`,
  `HADIR_AUTH_FERNET_KEY`, `HADIR_REPORT_SIGNED_URL_SECRET`)
  has been rotated. Failures collect into a single
  `ProductionConfigError` that lists every missing piece, so
  an operator sees the whole punch-list in one container log
  line.
* **`HttpsEnforceMiddleware`** — refuses plain-HTTP requests
  with HTTP 421 (Misdirected Request). `request.url.scheme`
  is what Starlette resolves *after*
  `ProxyHeadersMiddleware`, so a request that arrived over
  TLS at nginx and was proxied as HTTP on the private network
  still reads as `https`. `/api/health` is exempt so an
  internal load-balancer probing over HTTP isn't bounced.
* **`SecurityHeadersMiddleware`** — stamps HSTS,
  X-Content-Type-Options, X-Frame-Options, Referrer-Policy,
  and a minimal Permissions-Policy on every response. nginx
  ships the same headers; the backend's copy is defence in
  depth so the API stays safe even when fronted by a
  different proxy.

### Settings additions

`backend/hadir/config.py` gains:

* `allowed_origins_raw: str` (CSV, exposed via the
  `allowed_origins -> list[str]` property — a property
  because pydantic-settings 2.5.x lacks `NoDecode` and would
  otherwise try to JSON-parse the env value).
* `behind_proxy: bool = False`.
* `forwarded_allow_ips: str = "*"`.
* `hsts_max_age_seconds: int = 31_536_000`.

### Middleware ordering

`create_app()` now layers the middleware stack as (outer →
inner):

```
ProxyHeadersMiddleware           (only when behind_proxy=true)
HttpsEnforceMiddleware           (only when env=production)
CORSMiddleware                   (only when allowed_origins non-empty)
SecurityHeadersMiddleware        (when env != dev)
TenantScopeMiddleware            (always)
```

Order matters — proxy headers MUST run first so the HTTPS gate
sees the rewritten scheme; CORS must precede the security
header stamp so OPTIONS responses get HSTS too.

### nginx

`ops/nginx/hadir.conf.template` — a single `envsubst`-rendered
template. Reads `${HADIR_PUBLIC_HOSTNAME}`, `${HADIR_TLS_CERT}`,
`${HADIR_TLS_KEY}`. Listens on :80 (with an ACME-challenge
carve-out + a 301 to https for everything else) and :443.
Modern TLS profile — TLS 1.2 + 1.3 only, Mozilla "intermediate"
cipher list, OCSP stapling implicit via the cert chain.

`ops/nginx/entrypoint.sh` — runs `envsubst` to produce the
final config, validates with `nginx -t`, then exec's nginx in
the foreground. A missing cert or key kills the container with
a readable error message rather than nginx silently 502'ing.

`ops/nginx/Dockerfile` — multi-stage build. Stage 1 runs
`vite build` to produce the production frontend bundle; stage 2
copies the bundle into nginx's webroot and adds the templated
config. The result is a single image that serves both the
static bundle and proxies `/api/*` to the backend.

### Compose overlays

* `docker-compose.prod.yml` — overlays the dev compose:
  attaches every service to the new `hadir-internal` network,
  removes the host port mapping on backend (only nginx reaches
  it), removes the dev frontend service entirely (the bundle
  ships inside the nginx image), and adds the nginx service
  itself.
* `docker-compose.le.yml` — optional Let's Encrypt overlay.
  Adds a `certbot` sidecar that uses HTTP-01 against the
  webroot nginx already serves. Persists the issued cert to
  `ops/certs/{fullchain,privkey}.pem`, the same paths nginx
  reads from. Run once on first deploy + weekly via cron for
  renewal — there is intentionally no long-running daemon.

### Deployment runbook

`docs/deploy-production.md` — full runbook from a fresh Ubuntu
22.04 server to a serving stack. Covers Docker install, secret
generation, env file, both cert paths (operator-provided default
+ optional Let's Encrypt), boot sequence, first admin seed,
backups, smoke checklist, and the optional mkcert recipe for
HTTPS in dev.

`.env.example` documents the new production-only variables.

### Tests

`backend/tests/test_p23_https.py` (+16):

* Production config gate: passes when everything is set; skips
  when env=dev; rejects each individual failure mode (cookie,
  proxy, origins, http OIDC base, every placeholder secret).
* HTTPS gate: rejects plain HTTP with 421; accepts when
  `X-Forwarded-Proto: https` is present (via
  `ProxyHeadersMiddleware`); exempts `/api/health`.
* Security headers: HSTS, X-Content-Type-Options,
  X-Frame-Options, Referrer-Policy, Permissions-Policy all
  present on every response.
* End-to-end: `create_app()` boots when env vars are valid;
  raises `ProductionConfigError` when the cookie-secure flag
  is off in production.

Total: **425 passing** (409 prior + 16 P23). All tests use a
``prod_env`` fixture that flips the relevant env vars via
``monkeypatch.setenv``, so no test leaks production-shaped
config into the wider suite.

## Notes for whoever resumes next

* **HTTPS in dev is off by default.** The prompt says so, the
  runbook says so, the tests say so. If a future phase wants
  HTTPS by default in dev, lift the `mkcert` recipe out of the
  runbook and into a `docker-compose.dev-https.yml` overlay
  rather than touching the base dev compose.
* **`forwarded_allow_ips="*"` is intentional.** The backend
  port is internal-only inside the docker network, so `*`
  means "trust forwarded headers from any peer" but the only
  peer is nginx. If a future deployment exposes the backend
  port directly (don't), tighten this to nginx's IP — there's
  a one-line change site in `create_app()`.
* **HSTS preload.** The HSTS header includes `preload` because
  we ship `includeSubDomains`. Submitting the public hostname
  to <https://hstspreload.org> is operator discretion — the
  header is correct for it but the listing is irreversible.
  Don't preload a hostname you might later want to serve over
  HTTP.
* **Let's Encrypt rate limits.** ACME registers a cert per
  hostname per week with a 5-cert weekly cap. If you bring up
  and tear down the stack repeatedly during testing, use the
  ACME staging endpoint (`--server https://acme-staging-v02
  .api.letsencrypt.org/directory`) by overriding the certbot
  command — otherwise you'll hit the rate limit and have to
  wait it out.
* **Cert renewal monitoring.** The certbot run-once design
  doesn't ship its own monitoring. Set up a cron-job email or
  prometheus ssl_exporter against
  `${HADIR_PUBLIC_HOSTNAME}:443` so cert expiry doesn't sneak
  up on you. The 30-day buffer that certbot's
  `--keep-until-expiring` enforces is generous; the 5-day
  buffer that browsers warn at is not.
* **OIDC + HTTPS.** When an operator changes
  `HADIR_OIDC_REDIRECT_BASE_URL` (e.g. on first prod deploy or
  when adding a vanity hostname), they must also update the
  redirect URI list inside the Entra ID app registration —
  otherwise OIDC callback fails with `redirect_uri_mismatch`.
  The operator-facing error message in
  `hadir.auth.oidc.callback` is clear enough to surface this.
* **The 421 status code.** `HttpsEnforceMiddleware` returns
  421 ("Misdirected Request") rather than 426 ("Upgrade
  Required") because clients almost always retry 421 on the
  same path against a different scheme, while 426 has poor
  client-library support. If a client misbehaves under 421,
  swap the constant — both interpretations are RFC-blessed.
