# P5 — Second-tenant non-prod smoke

**Phase:** v1.0 P5 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P5 — Second tenant non-prod smoke"`

## Goal

This phase is a **verification gate**, not a feature build. Two
tenants exist in the dev DB at the same time; each has its own
Admin, employees, branding, and audit history; the API surface
provably refuses every cross-tenant read attempt; CI runs the
isolation suites on every push so a future regression fails the
build before it lands.

Red line: any cross-tenant leak surfaced here is a P0. The CI gate
makes failures impossible to overlook.

## What was done

### Multi-tenant login (small, targeted)

P3 left a known gap: the middleware looked sessions up only in
`main.user_sessions`, so a tenant Admin outside the pilot couldn't
log in. P5 fixes it minimally:

- `POST /api/auth/login` now accepts an optional
  `tenant_slug` (the schema name). When present, the handler
  resolves the slug → `(tenant_id, schema_name)` from
  `public.tenants`, refuses suspended tenants, and runs the user
  lookup + session creation under `tenant_context(schema_name)`.
- Login sets two cookies: `maugood_session` (the opaque token, as
  before) and a new `maugood_tenant=<slug>` cookie. Logout clears
  both.
- `TenantScopeMiddleware` reads `maugood_tenant` (validated against
  `_TENANT_SCHEMA_RE` defensively) and applies that schema before
  looking up `user_sessions`. Falls back to `main` when the cookie
  is absent so the pilot's single-tenant flow stays untouched.
- Unknown slugs return 401 (don't oracle tenant existence) and
  invalid regex matches return 400.

### `tests/test_two_tenant_isolation.py` (the P5 suite)

Module-scoped fixture provisions `tenant_smoke_omran` and
`tenant_smoke_demo` via `scripts.provision_tenant.provision()`,
yields, and deprovisions both at teardown. 9 tests in declared
order share the state:

1. `test_provision_creates_two_isolated_tenants` — both rows in
   `public.tenants`; each schema seeded with ENG/OPS/ADM
   departments.
2. `test_each_admin_logs_in_against_their_own_tenant` — login
   succeeds with each Admin's own slug; reusing Omran's password
   against Demo's slug returns 401.
3. `test_omran_admin_creates_employee_and_uploads_photo` —
   `OM-SMOKE-001` + a tiny synthetic JPEG (`_TINY_JPEG`) goes
   through `/api/employees/{id}/photos`. The session-wide
   `_NoopAnalyzer` keeps InsightFace out of the loop.
4. `test_demo_admin_creates_distinct_employee` — `DM-SMOKE-001`
   under the Demo Admin's session.
5. `test_omran_admin_cannot_see_demo_employees_via_list` — the
   `/api/employees` list contains `OM-SMOKE-001` and explicitly
   not `DM-SMOKE-001`.
6. `test_omran_admin_cannot_see_demo_employees_via_search` —
   searching for Demo's email, code, and full name all return
   `total: 0` from Omran's session.
7. `test_omran_audit_log_excludes_demo_actions` — Omran's audit
   log API never surfaces Demo's `entity_id`. Belt-and-braces
   DB-level read confirms each schema has its own
   `employee.created` row.
8. `test_demo_admin_sees_their_own_data` — symmetric assertion
   from Demo Co's side.
9. `test_super_admin_access_as_both_tenants` — Super-Admin login,
   Access-as Omran (verifies `is_super_admin_impersonation=true`
   on `/api/auth/me`), Access-as Demo (same), exit. Asserts both
   tenant ids land as `super_admin.access_as.start` rows in
   `public.super_admin_audit`.

127 tests pass total (118 prior + 9 new isolation).

### CI gate — `.github/workflows/isolation.yml`

GitHub Actions workflow that runs on every push to `main` and on
PRs touching `backend/`. Boots a Postgres 15 service, installs
backend deps, runs `python -m scripts.migrate` to bootstrap the
schema, then runs:

- `tests/test_migration_lint.py` (3 tests)
- `tests/test_multi_tenant_isolation.py` (7 tests, the P1 canary)
- `tests/test_two_tenant_isolation.py` (9 tests, this phase)

`CLAUDE.md` documents that **isolation-suite failures are P0
blockers** — fix the leak before merging anything else, do not
punt to a known-issue list.

## Manual walkthrough (replayable)

These commands run against the dev compose stack. Substitute your
own super-admin credentials (the seed script is idempotent).

### 1. Seed an MTS Super-Admin (one-time)

```sh
docker compose exec -e MAUGOOD_SUPER_ADMIN_PASSWORD='SuperAdmin!42' backend \
    python -m scripts.seed_super_admin \
    --email super@mts.example.om --full-name 'MTS Super Admin'
```

### 2. Provision both smoke tenants

```sh
docker compose exec -e MAUGOOD_PROVISION_PASSWORD='OmranSmoke!42a' backend \
    python -m scripts.provision_tenant \
    --slug tenant_smoke_omran --name 'Omran Smoke' \
    --admin-email admin@omran.smoke --admin-full-name 'Omran Admin'

docker compose exec -e MAUGOOD_PROVISION_PASSWORD='DemoSmoke!42a' backend \
    python -m scripts.provision_tenant \
    --slug tenant_smoke_demo --name 'Demo Smoke Co' \
    --admin-email admin@demo.smoke --admin-full-name 'Demo Admin'
```

Each prints `[provision] provisioned tenant_id=N schema=tenant_smoke_X
admin_user_id=1 admin_email=…` on success.

### 3. Switch Demo Co's branding to navy + lato (operator-side)

```sh
# Login as super-admin
curl -c sa.txt -X POST :8000/api/super-admin/login \
    -H 'Content-Type: application/json' \
    -d '{"email":"super@mts.example.om","password":"SuperAdmin!42"}'

# Find tenant ids
curl -b sa.txt :8000/api/super-admin/tenants

# Patch Demo Co's branding (replace <demo_id> with the id from above)
curl -b sa.txt -X PATCH :8000/api/super-admin/tenants/<demo_id>/branding \
    -H 'Content-Type: application/json' \
    -d '{"primary_color_key":"navy","font_key":"lato"}'
```

Omran Smoke's branding stays at `teal` + `inter` (the seeded
default).

### 4. Each Admin logs in (separate cookie jars)

```sh
curl -c omran.txt -X POST :8000/api/auth/login \
    -H 'Content-Type: application/json' \
    -d '{"email":"admin@omran.smoke","password":"OmranSmoke!42a",
         "tenant_slug":"tenant_smoke_omran"}'

curl -c demo.txt  -X POST :8000/api/auth/login \
    -H 'Content-Type: application/json' \
    -d '{"email":"admin@demo.smoke","password":"DemoSmoke!42a",
         "tenant_slug":"tenant_smoke_demo"}'
```

Both return `200`. Each cookie jar holds `maugood_session` (the opaque
token) plus `maugood_tenant=<slug>` (the routing hint).

### 5. Each Admin creates a distinct employee

```sh
curl -b omran.txt -X POST :8000/api/employees \
    -H 'Content-Type: application/json' \
    -d '{"employee_code":"OM-WALK-001","full_name":"Omran Walker",
         "email":"walker@omran.smoke","department_id":1}'

curl -b demo.txt  -X POST :8000/api/employees \
    -H 'Content-Type: application/json' \
    -d '{"employee_code":"DM-WALK-001","full_name":"Demo Walker",
         "email":"walker@demo.smoke","department_id":1}'
```

Both return `id: 1` — the per-tenant sequences are independent.

### 6. Cross-tenant reads return zero (the verification step)

```sh
$ curl -b omran.txt ':8000/api/employees?q=DM-WALK-001'
{"items":[],"total":0,"page":1,"page_size":50}

$ curl -b omran.txt ':8000/api/employees?q=walker@demo.smoke'

$ curl -b demo.txt  ':8000/api/employees?q=OM-WALK-001'
{"items":[],"total":0,"page":1,"page_size":50}
```

### 8. Super-Admin Access-as both, observe red banner via API

```sh
# Access-as Omran Smoke
curl -b sa.txt -X POST :8000/api/super-admin/tenants/<omran_id>/access-as

# /api/auth/me carries the impersonation flag
$ curl -b sa.txt :8000/api/auth/me
{"id":0,"email":"super-admin#…","full_name":"Super-Admin (impersonating)",
 "roles":["Admin","HR","Manager","Employee"],"departments":[],
 "is_super_admin_impersonation":true,"super_admin_user_id":…}

# Switch to Demo
curl -b sa.txt -X POST :8000/api/super-admin/tenants/<demo_id>/access-as

# Both starts in the operator log
$ docker compose exec postgres psql -U maugood -d maugood -c "
    SELECT id, action, super_admin_user_id, tenant_id, ip
    FROM public.super_admin_audit
    WHERE action='super_admin.access_as.start' ORDER BY id DESC LIMIT 2;"
 186 | super_admin.access_as.start | … | 12 | 192.168.65.1   # Demo
 185 | super_admin.access_as.start | … | 11 | 192.168.65.1   # Omran

curl -b sa.txt -X POST :8000/api/super-admin/exit-impersonation
```

In the browser: hitting `/api/auth/me` while impersonating returns
`is_super_admin_impersonation=true`, which causes the tenant
shell's `Layout.tsx` to mount the persistent red `ImpersonationBanner`
above the sidebar. The banner is **mandatory** — it's the in-shell
safety signal that operator-mode is active (see P3 red lines).

### 9. Cleanup

```sh
docker compose exec backend python -m scripts.deprovision_tenant \
    --slug tenant_smoke_omran --confirm --yes-i-know
docker compose exec backend python -m scripts.deprovision_tenant \
    --slug tenant_smoke_demo  --confirm --yes-i-know
```

Encrypted face/capture crops under `/data/faces/<id>/` and
`/data/faces/captures/<id>/` are intentionally NOT removed by the
CLI (per its docstring) — that's an operator-volume concern.

## Backward compatibility

- Existing pilot login (no `tenant_slug`) still works — defaults to
  `main`. All 118 prior tests pass unchanged.
- `maugood_session` cookie format unchanged. The new `maugood_tenant`
  cookie is purely additive.
- The pilot's single-mode middleware path (cookie-less anonymous
  request → `main` schema) is untouched.

## Notes for whoever resumes next

- The frontend `LoginPage` does not yet render a tenant slug field.
  For dev we use the super-admin Access-as flow to inspect each
  tenant's shell. v1.0 production will require a host-based tenant
  resolver (e.g. `acme.maugood.io/login` → backend reads the host
  header to fill in `tenant_slug`) so tenant Admins don't have to
  type a slug. Track that under the Auth/SSO phase rather than
  retrofitting it here.
- The 401 on unknown slug intentionally doesn't distinguish from
  401 on bad password — that prevents tenant enumeration. The
  audit row inside the (now empty) tenant context isn't written
  for unknown slugs because we don't have a tenant_id to attach
  the row to. Acceptable for now; if we later add a
  `public.global_audit` we can record these there.
- The two-tenant suite is module-scoped — provisioning runs once.
  If you're iterating on a single test, use
  `pytest tests/test_two_tenant_isolation.py::test_xxx -k xxx -v`;
  the fixture's `try/finally` ensures both smoke tenants are
  deprovisioned even on test failure, so a flaky run doesn't
  pollute the dev DB.
