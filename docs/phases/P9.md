## Goal

Add a **Flex** policy alongside the pilot's Fixed type, with
per-tenant assignment cascade (employee > department > tenant >
legacy fallback). Engine stays pure: the only DB-touching code
is the resolver, which feeds the engine the resolved policy.

Hard red line (the prompt's own + pilot's pilot-plan P10 carry-over):
`maugood.attendance.engine` is side-effect-free. Resolution and the
CRUD API live in `repository.py` / `policies/router.py` so the
engine itself remains plain Python that takes inputs and returns a
value.

## What was done

### Engine refactor ‚Äî Fixed + Flex share the same shape

`maugood/attendance/engine.py`:

* `ShiftPolicy` extended with optional Flex-only fields
  (`in_window_start/end`, `out_window_start/end`). Fixed-only
  fields (`start`, `end`, `grace_minutes`) become Optional too.
  Existing tests construct unchanged.
* New private `_Flags` dataclass ‚Äî `late`, `early_out`,
  `short_hours`, `overtime_minutes`. Both per-type helpers
  (`_fixed_flags`, `_flex_flags`) return it.
* `compute()` is the single public entry point. It does the
  common work ‚Äî events sorting, in/out time extraction,
  total_minutes, absent ‚Äî then dispatches on `policy.type` to
  `_flags_for(...)` which routes to the right per-type helper.
  Ramadan / Custom slots fall through to Fixed flags until
  their respective phases ship.
* `policy_from_row(row)` reads ``shift_policies.config`` and
  inflates the right field set per type.

Per-type rules (Flex):

* `late = in_time > policy.in_window_end` ‚Äî anywhere inside the
  in-window is on time.
* `early_out = out_time < policy.out_window_start` ‚Äî symmetric.
* `short_hours / overtime_minutes` use `total_minutes` against
  `required_minutes` exactly like Fixed.

### Migration 0013 ‚Äî `policy_assignments`

Per-tenant table mapping `shift_policies` rows to a resolution
scope:

| Column | Notes |
| --- | --- |
| `id` (PK) | autoinc serial. |
| `tenant_id` | FK to **unqualified** `tenants(id)` (schema-agnostic). |
| `policy_id` | FK `shift_policies.id`, ON DELETE CASCADE. |
| `scope_type` | CHECK in `('tenant','department','employee')`. |
| `scope_id` | NULL for `tenant`; required for `department` / `employee`. CHECK enforces coherence. |
| `active_from` / `active_until` | date range; NULL `active_until` is open-ended. |

Index: `(tenant_id, scope_type, scope_id)` so the resolver's
per-tier lookups are cheap. Grants mirror the per-tenant pattern.
Provisioning's `_APP_CRUD_TABLES` extended.

### `resolve_policies_for_employees(...)` ‚Äî the only DB surface

`maugood/attendance/repository.py`:

* `resolve_policies_for_employees(conn, scope, *, the_date,
  employee_ids) -> dict[int, ShiftPolicy]` walks the cascade:
  1. Employee-scoped match (highest priority).
  2. Department-scoped match (employee's `department_id`).
  3. Tenant-scoped match.
  4. Legacy fallback ‚Äî any active `shift_policies` row in the
     tenant. Lets the pilot keep working without explicit
     assignments.
* Issues at most four queries regardless of input size:
  one per scope tier + one to resolve employee ‚Üí department.
* Empty `employee_ids` ‚Üí empty dict (no work).

### Scheduler integration

`attendance/scheduler.py::recompute_today` now:

1. Loads `active_employee_ids`.
2. Calls `resolve_policies_for_employees` once per scope.
3. For each employee, fetches events + calls `compute(...)`
   with their resolved policy. An employee with no resolvable
   policy is skipped (the missing-policy log line at the
   tenant level still fires when nothing resolves anywhere).

The pilot path (one tenant, one Fixed policy, no
`policy_assignments` rows) lands cleanly on the legacy fallback ‚Äî
existing behaviour unchanged for any tenant that hasn't opted into
the new system.

### CRUD API ‚Äî `maugood/policies/`

* `GET /api/policies` (Admin + HR) ‚Äî list.
* `POST /api/policies` ‚Äî create (Pydantic validates Fixed vs
  Flex config shape). Audit `shift_policy.created`.
* `PATCH /api/policies/{id}` ‚Äî partial update; `active_until`
  can be explicitly nulled. Audit `shift_policy.updated`.
* `DELETE /api/policies/{id}` ‚Äî soft-delete by setting
  `active_until = today - 1`. Hard delete refused ‚Äî RESTRICT FK
  on `attendance_records.policy_id` makes that a non-starter
  anyway. Audit `shift_policy.soft_deleted`.
* `GET /api/policy-assignments` ‚Äî list.
* `POST /api/policy-assignments` ‚Äî validates the
  policy + scope reference exist in the tenant; CHECK
  constraints reject bad payloads at the DB. Audit
  `policy_assignment.created`.
* `DELETE /api/policy-assignments/{id}` ‚Äî drop. Audit
  `policy_assignment.deleted`.

Pydantic `PolicyConfig` validates `HH:MM` strings on every time
field and reasonable ranges on `grace_minutes` (0‚Äì180) and
`required_hours` (1‚Äì24). The `PolicyCreateRequest` model_validator
refuses Fixed missing `start`/`end` and Flex missing any of the
four window times ‚Äî the API can't accept a half-formed config.

### Frontend Policies page

`frontend/src/policies/`:

* `types.ts` + `hooks.ts` ‚Äî TanStack Query.
* `PoliciesPage.tsx` ‚Äî Admin + HR. Lists every policy with type
  badge, window summary, required-hours, active range, current
  assignments. Create form switches fields per type. Per-row
  assignment cell shows a chip per existing assignment (with √ó
  to remove) plus three quick-assign buttons:
  - **Tenant default** ‚Äî single click; only shown when no
    tenant-wide row exists yet.
  - **Dept‚Ä¶** / **Employee‚Ä¶** ‚Äî reveal a small numeric input
    that sends `scope_type=<type>, scope_id=<n>`.

  Soft-delete confirms at the modal level and surfaces server
  errors inline.

The `policies` nav id (already present in the design's NAV) now
routes to the new page; the placeholder is retired.

## Notes for whoever resumes next

- **`active_role` and the policy/assignment endpoints**: Admin
  and HR are both allowed, mirroring the BRD. The seeded matrix's
  `dual.role@omran.test` user (HR + Manager) covers the
  cross-role case ‚Äî see the M2 test-accounts doc.
- **Soft-delete vs hard delete**: `attendance_records.policy_id`
  has a RESTRICT FK to `shift_policies`. We don't try to hard-
  delete; the soft-delete (`active_until = yesterday`) preserves
  history while making the policy invisible to the resolver from
  today onward.
- **Ramadan / Custom**: the type enum already accepts both, but
  the engine routes them to Fixed flags as a placeholder. Their
  own phases will add per-type helpers without touching the
  public `compute(...)` signature.
- **Resolver perf**: at most four queries, all small index
  scans. If the tenant grows past a few thousand active
  employees the per-employee `dept_by_employee` lookup may want
  caching, but that's a P22+ concern.
- **Frontend assignment UI**: the Dept‚Ä¶ / Employee‚Ä¶ inputs take
  raw numeric ids today. A v1.0+ refinement would surface a
  searchable picker using the existing `/api/employees` and a
  forthcoming `/api/departments` endpoint.
