## Goal

Stand up the Live Capture page that the design archive shows
under nav id `live`. Two parts:

1. **Sidebar version chip** — replace the hardcoded `v0.1`
   string with a config-driven value sourced from
   `package.json` so the brand mark matches reality.
2. **Live Capture page** — a viewer on top of the existing P8
   capture pipeline. MJPEG stream + WebSocket detection events
   + on-demand stats. No new capture workers; the existing P8
   per-camera worker fans its frames out via an in-memory
   ``FrameBuffer``.

## Architecture

The capture worker is the single producer. Three new pieces
plumb the producer to viewers:

* ``maugood/capture/frame_buffer.py`` — process-global, thread-
  safe, one slot per camera. Writes overwrite (never queue).
  Memory ceiling: 20 cameras × ~80 KB JPEG ≈ 1.6 MB. No
  persistence.
* ``maugood/capture/event_bus.py`` — in-process asyncio pub/sub
  keyed by ``(tenant_id, camera_id)``. The capture worker
  publishes; the WebSocket handler subscribes. Per-subscriber
  ``asyncio.Queue`` (maxsize=64) drops oldest on overflow so
  a slow client can't stall the producer.
* ``maugood/capture/annotate.py`` — `cv2.rectangle` + `cv2.putText`
  primitives that draw bounding boxes + labels (employee name +
  confidence, or "Unknown") onto the BGR frame in place. Boxes
  are baked in *before* JPEG encoding so the frontend `<img>`
  doesn't need a canvas overlay — the design's `id-known` /
  `id-unknown` colours map to BGR `(128, 222, 74)` /
  `(21, 204, 250)` respectively.

The capture worker tap is in ``capture/reader.py``. After every
read+detect+match cycle:

1. Pre-match every detection against the matcher cache (so the
   annotation step has the result to draw labels).
2. Annotate the frame, JPEG-encode at quality 70, push to
   ``frame_buffer.set(camera_id, jpeg)``.
3. For each NEW track, call ``emit_detection_event(...,
   pre_matched=...)`` — no second matcher call. After the DB
   row commits, ``emit`` publishes a ``DetectionEvent`` to the
   bus with the resolved employee name + code (read once from
   ``capture/directory.py``, the per-tenant employee directory
   cache).

Empty-frame path: still pushes the bare frame to the buffer so
viewers see live video when the room is empty.

## Endpoints (all Admin-only, all tenant-scoped)

* ``GET /api/cameras/{id}/live.mjpg`` — multipart MJPEG at 10 fps.
  Boxes baked in. Caps at 10 concurrent viewers per camera per
  tenant; beyond that → 503. Stale buffer (no fresh frame for
  10 s after a 2 s cold-start grace) → stream ends so the
  browser shows the offline state.
* ``GET /api/cameras/{id}/events.ws`` — WebSocket. Cookie-auth.
  Sends `{type: "detection", ...}` per new track,
  `{type: "heartbeat", ...}` every 5 s, `{type: "stats", ...}`
  every 30 s. Same 10-subscriber cap.
* ``GET /api/cameras/{id}/live-stats`` — JSON. Polled every 10 s
  by the frontend. Pulls a 10-minute window from
  ``detection_events`` plus the latest
  ``camera_health_snapshots`` row for fps + reachability.
* ``GET /api/cameras/{id}/events.csv?hours=N`` — Streams the
  camera's recent events as CSV. Same column set as P11
  Camera Logs export, kept thin.

Audit pattern: `live_capture.{mjpg,events}.{subscribed,unsubscribed}`
on stream open + close (close written from the generator's
`finally`, so a network drop counts as an unsubscribe). NEVER
audit per frame — at 10 fps × N viewers the audit log would
double in volume in a day.

## Tenant isolation red line

Every endpoint resolves `cameras` with
`WHERE id = :id AND tenant_id = :scope_tenant_id` BEFORE yielding
a single byte. Cross-tenant guesses 404. The ``frame_buffer`` is
process-global but every read is tenant-checked first; there's
no API surface that hands a viewer a frame from a camera the
caller doesn't own. The two-tenant isolation suite
(`test_two_tenant_isolation.py`, the P0 blocker) still passes.

## Reused from the Round 2 prototype

The Round 2 prototype isn't vendored into this repo. The MJPEG
generator pattern (multipart boundary, idle-tick bail, Cache-Control
+ X-Accel-Buffering disable headers) and the rectangle-and-label
drawing approach were adapted to fit Maugood's existing capture
pipeline + tenant scoping. Nothing was lifted verbatim — the
pieces that survived were the *shape* of the producer/consumer
split and the JPEG-quality 70 budget.

## Frontend

* `frontend/src/features/live-capture/LiveCapturePage.tsx` —
  ported from `design-reference/pages.jsx::LiveCapture`. Header
  strip, 2:1 grid (viewer + camera list), full-width event-stream
  table. Plain `<img>` for the viewer; nonce-bumped `src` forces
  reconnects.
* `useLiveStats(cameraId)` — TanStack Query, 10 s poll, paused
  on no selection or pause.
* `useEventStream(cameraId)` — WebSocket lifecycle + rolling
  buffer of last 50 events. Exponential backoff reconnect (1 s →
  30 s cap; resets on every successful open).
* Pause/Resume cycles the `<img>` src (which closes the MJPEG
  on the browser side); WebSocket stays open during pause so
  the events table keeps updating.
* Reconnect bumps the nonce + reopens the WebSocket.
* Empty / offline / reconnecting overlays use design tokens —
  no Tailwind, no inline brand colours.

## Performance budgets

* JPEG quality 70, 1280×720 max → ~80 KB per frame on a typical
  scene. Tuned for LAN; external exposure would warrant lower
  quality or a switch to H.264 via WebRTC.
* Capture worker CPU bumps ~5–10% per camera due to box
  drawing + JPEG encode. Within budget for the BRD's 20-camera
  ceiling on the target hardware.
* MJPEG endpoint serves at 10 fps regardless of capture rate.
  20 cameras × 10 fps × 80 KB ≈ 16 MB/s aggregate with one
  viewer per camera — fine on LAN, would be tight on WAN.
