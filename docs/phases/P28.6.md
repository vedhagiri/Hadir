## Goal

Stand up the Attendance Calendar page that the sidebar's
`calendar` entry has been routing to a placeholder for since P4.
Two views:

1. **Company** — tenant-wide month aggregate (Admin/HR see the
   whole tenant; Manager sees their visible employee union).
2. **Per-person** — one employee, one month, with a day-detail
   drawer carrying the engine's status, in/out times, totals,
   policy applied, day timeline ribbon, and evidence crops.

Plus a cross-format export (XLSX) and a "+ Submit exception"
shortcut into the existing P14 request flow.

## Architecture

**No new tables.** P10 already produces one row per
`(employee, date)` in `attendance_records` with the flags
`late`/`absent`/`early_out`/`short_hours`/`overtime_minutes`.
P11 added `approved_leaves` (covers leave dates), `holidays`
(covers holiday dates), and `tenant_settings.weekend_days` /
`timezone`. The calendar is a read-side aggregation over those.

Three new pieces:

* `backend/hadir/attendance_calendar/queries.py` — the pure-ish
  aggregation module. `parse_month` (YYYY-MM bounds),
  `iter_days`, `is_weekend`, plus the four payload builders:
  - `company_view(...)` runs one `GROUP BY date` against
    `attendance_records` joined to `employees` for role scoping,
    then folds in holidays + weekend membership for one
    `CompanyDay` per calendar day.
  - `person_view(...)` builds one `PersonDay` per day for one
    employee, using the same status-priority cascade as the
    drawer (future > holiday > weekend > leave > absent > late
    > present > no_record).
  - `collapse_timeline(...)` is a pure helper: given a sorted
    list of detection times, collapses adjacent times within
    `TIMELINE_GAP_MINUTES` (10) into intervals.
  - `pick_evidence(...)` picks up to 5 detection events spread
    across 5 buckets (arrival / morning / midday / afternoon /
    departure), highest-confidence per bucket. Skips events
    whose underlying file has been swept by orphan cleanup
    (`has_crop=False`) — those would 404 from the crop endpoint.
  - `fetch_day_detail(...)` is the drawer payload: employee
    header, attendance row + policy + leave-type joins,
    `holidays_on` lookup, detection events join with cameras,
    timeline + evidence built from the events.

  Status enum is computed server-side (`STATUS_PRESENT` etc.)
  so the frontend stays dumb — it renders whatever the server
  hands it.

* `backend/hadir/attendance_calendar/router.py` — four
  endpoints under `/api/attendance/calendar`:

  | Method + path | Role gate | Notes |
  | --- | --- | --- |
  | `GET /company?month=YYYY-MM` | Admin/HR/Manager | Manager scoped to visible employee union; Employee 403. Bad month → 400. |
  | `GET /person/{employee_id}?month=YYYY-MM` | Admin/HR any; Manager visible-only; Employee self-only | Out-of-scope → **404, never 403** (red line — 403 leaks scope) |
  | `GET /day/{employee_id}/{date}` | same as person | Audit-logs `attendance_calendar.day_viewed` per fetch |
  | `GET /export?month=&employee_id=&date=` | same as person; company aggregate Admin/HR/Manager | XLSX. Three modes: single-day / per-person month / company aggregate. Audit-logs `attendance_calendar.exported`. |

  Tenant isolation is enforced in `_check_can_view_employee`:
  we run a `SELECT id FROM employees WHERE tenant_id=:scope
  AND id=:requested` first, so a cross-tenant guess from an
  mts_demo Admin against an inaisys employee_id returns 404
  before the role check even runs.

* `backend/hadir/main.py` — adds the import + `include_router`
  call. No middleware changes.

**Frontend** lives in `frontend/src/features/calendar/`:

* `types.ts` — TypeScript wire types mirroring the Pydantic
  response models.
* `hooks.ts` — three TanStack Query hooks
  (`useCompanyCalendar`, `usePersonCalendar`, `useDayDetail`)
  with 60s stale time on the month views and 30s on the day
  detail.
* `CompanyView.tsx` — month grid (7-col, Sunday-first, design
  archive's `cal-month-grid`). Each day cell is a button with
  `cal-day status-{status}` class so the existing CSS
  (`styles.css` lines 569-611) paints the tinted backgrounds.
  Click → switches the page to per-person view scoped to that
  month so the operator picks their employee.
* `PersonView.tsx` — same month grid, one row per day, status
  pill + in/out times + holiday/leave name. Click → opens
  `DayDetailDrawer`. Future cells are non-clickable (the
  endpoint would just return `status: future` with empty
  evidence anyway).
* `DayDetailDrawer.tsx` — the heavy drawer. Status pill,
  summary tiles (In/Out/Total/Overtime), 24-hour SVG day
  timeline ribbon (one bar per `TimelineInterval`), policy
  applied panel with description + scope, evidence grid (5
  crops max, lazy-loaded via the existing
  `/api/detection-events/{id}/crop` endpoint), Export +
  Submit exception buttons.
* `CalendarPage.tsx` — top-level page. Tab strip toggles
  Company / Per-person; Employee role auto-locks to
  themselves and hides the Company tab. Month picker drives
  both views.

The "+ Submit exception" button reuses `NewRequestDrawer`
verbatim; we added two optional props (`initialType`,
`initialStartDate`) so the drawer pre-fills with
`type="exception"` and the date the operator was looking at.
No duplication of the request submission logic.

## Status priority

Same priority used in both `person_view` and
`fetch_day_detail`. **Order is deterministic and load-bearing**:

1. **future** (cell is in tomorrow or later)
2. **holiday** (date hits `holidays`, `active=true`)
3. **weekend** (`date.strftime("%A")` is in
   `tenant_settings.weekend_days`)
4. **leave** (attendance row has `leave_type_id`, OR an
   `approved_leaves` row covers the date and there's no
   attendance row yet)
5. **absent** (attendance row's `absent` flag)
6. **late** (attendance row's `late` flag)
7. **present** (attendance row exists with neither absent nor
   late)
8. **no_record** (workday with no attendance row and no leave)

A holiday on a weekend reads as **holiday** — the operator
wants to see the named day, not the generic weekend.
