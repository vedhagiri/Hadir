# P11 — Leaves + holidays module

**Phase:** v1.0 P11 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P11 — Leaves + holidays module"`

## Goal

Three new operational concepts move into Maugood proper:

* **Leave types** — Annual, Sick, Emergency, Unpaid (seeded);
  operators can add custom types.
* **Holidays** — tenant-wide non-working days. The calendar lives
  in the DB; bulk-add or .xlsx import for the year.
* **Approved leaves** — the ledger the engine reads at recompute
  time. Submission + approval workflow ships in P14/P15; this
  phase ships only the storage + manual create endpoint.

Plus a new **tenant_settings** table that finally moves the
**timezone tenant-scoped** — every attendance comparison runs in
the tenant's timezone, not the server's. That's the P11 red line.

## What was done

### Migration 0014

`alembic/versions/0014_leaves_holidays_settings.py`. Four new
per-tenant tables + one column on `attendance_records`:

| Table | Notes |
| --- | --- |
| `leave_types` | id, tenant_id, code, name, is_paid, active. UNIQUE (tenant_id, code). |
| `holidays` | id, tenant_id, date, name, active. UNIQUE (tenant_id, date). |
| `approved_leaves` | id, tenant_id, employee_id, leave_type_id, start_date, end_date, notes, approved_by_user_id (SET NULL), approved_at. CHECK start_date ≤ end_date. |
| `tenant_settings` | tenant_id PK, weekend_days JSONB default `["Friday","Saturday"]`, timezone default `Asia/Muscat`, updated_at. |
| `attendance_records.leave_type_id` | nullable FK to `leave_types(id)` ON DELETE SET NULL. |

Schema-agnostic — every FK targets the unqualified `tenants(id)`
so the migration runs cleanly under every tenant schema. Idempotent
seed at the bottom (cross-join `tenants` filtered by
`current_schema()`) inserts the four default leave types and one
default `tenant_settings` row for the migrating tenant.

### `db.py` + provisioning

Four new `Table` definitions added in `maugood/db.py` plus the
`leave_type_id` column on `attendance_records`. Provisioning's
`_APP_CRUD_TABLES` extends to include the four new tables; the
provisioning seed step inserts the default leave types and
tenant_settings row inline so a freshly-provisioned tenant looks
identical to an existing one.

### Engine — leaves + holidays + weekends (still pure)

`maugood/attendance/engine.py`:

* New `LeaveRecord` / `HolidayRecord` dataclasses.
* `AttendanceRecord` gains `leave_type_id: Optional[int]` and
  `leave_type_name: Optional[str]` (default None).
* `compute()` accepts `leaves=()`, `holidays=()`, and
  `weekend_days=()`. Behaviour:
  - Approved leave covering the date → `absent=False`,
    `leave_type_name` surfaces.
  - Holiday OR weekend with events → entire `total_minutes` →
    `overtime_minutes`; `late` / `early_out` / `short_hours` are
    forced to `False`. Per-type flag math is skipped entirely on
    a non-working day.
  - Holiday OR weekend without events → `absent=False` (no
    penalty for not showing up on a non-working day).
  - Holiday-on-weekend collapses to a single overtime treatment
    — no double counting.

Engine **stays pure** (P11 red line): no DB, no clock, no
network. The repository fetches the inputs and hands them in.

### Repository + scheduler — tenant-scoped timezone

`maugood/attendance/repository.py`:

* `TenantTimeSettings` value object + `load_tenant_settings(conn,
  scope)` returns the tenant's `weekend_days` + `timezone` (with
  a defence-in-depth fallback to the global setting if no row).
* `local_tz_for(settings)` — ZoneInfo from the tenant's timezone.
  Existing `local_tz()` stays as a server-scoped legacy fallback.
* `holidays_on(conn, scope, *, the_date)` and
  `leaves_for_employee_on(conn, scope, *, employee_id, the_date)`
  inflate `HolidayRecord` / `LeaveRecord` lists ready for the
  engine.
* `upsert_attendance` now persists `leave_type_id`.

`maugood/attendance/scheduler.py::_recompute_today_inner` reads
tenant settings up front, picks the date in the tenant's
timezone, and per employee passes leaves + holidays +
`weekend_days` into `compute()`.

### CRUD API — `maugood/leave_calendar/`

* `GET / POST / PATCH /api/leave-types` (Admin + HR).
* `GET / POST / DELETE /api/holidays` + `POST /api/holidays/import`
  (.xlsx with `date` + `name` columns; idempotent).
* `GET / POST / DELETE /api/approved-leaves`.
* `GET / PATCH /api/tenant-settings` — `weekend_days` validated
  against the seven English weekday names; `timezone` validated
  via `ZoneInfo`.

Audit hooks on every mutation:
`leave_type.{created,updated}`, `holiday.{created,deleted,bulk_imported}`,
`approved_leave.{created,deleted}`, `tenant_settings.updated`.

### Frontend Leave & Calendar page

`frontend/src/leave-calendar/`:

* Routed at `/leave-policy` (the existing `leave-policy` nav id).
  Pilot placeholder retired.
* Three tabs:
  - **Leave types** — list + create form + per-row Paid/Active
    chips (toggle PATCH).
  - **Holidays** — year picker, manual add, list with weekday
    column, .xlsx import button.
  - **Approved leaves** — list + create form selecting an
    employee id + leave_type from the active list.
* Tenant-settings panel at the top of the page exposes
  `weekend_days` (toggle pills) and `timezone` (free-form IANA
  input on blur).

## Notes for whoever resumes next

- **Submission + approval workflow** lands in P14–P15. Until
  then, the only way an `approved_leaves` row appears is via
  Admin/HR creating it manually through `POST
  /api/approved-leaves`. The frontend's Approved Leaves tab
  exposes that surface today.
- **Calendar grid view** is left as a list — the pilot date
  picker on the Holidays tab is sufficient for most operators
  to add a year's worth of holidays. A proper month-grid view
  with click-to-add can land later if HR asks.
- **Hijri labelling** for holidays parallels the P10 note —
  Maugood stores Gregorian dates throughout. A Hijri-aware label
  on the holidays list is a small follow-up.
- **Per-tenant tz drift** — if attendance ever drifts by a day
  near midnight, timezone is the first suspect. The contract is
  enforced through `load_tenant_settings(conn, scope)` followed
  by `local_tz_for(settings)`; the legacy `local_tz()` (server-
  scoped) remains only as a defence-in-depth fallback for
  partially-bootstrapped DBs and should never be the real path
  in production.
