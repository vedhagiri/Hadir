## Goal

Stand up the operational metrics + alerting stack: backend
emits Prometheus metrics, Prometheus scrapes them on a 15 s
cadence, Grafana renders the seven panels Omran IT will watch
during a normal day, and Alertmanager fans the four BRD-named
alerts to either a webhook or an Admin email.

The whole observability surface lives **inside** the docker
network — `/metrics` is never on the public internet, and PII
is never on a metric label.

## What was done

### `backend/hadir/metrics.py`

Single source of truth for every custom metric. Module-level
`Counter` + `Gauge` objects + tiny `observe_*` helpers that
do the right label coercion and silently drop on
already-registered exceptions. The seven prompted metrics:

| Metric                                       | Type    | Labels                       |
| -------------------------------------------- | ------- | ---------------------------- |
| `hadir_capture_frames_total`                 | counter | tenant, camera               |
| `hadir_detection_events_total`               | counter | tenant, identified           |
| `hadir_camera_reachable`                     | gauge   | tenant, camera               |
| `hadir_attendance_records_computed_total`    | counter | tenant                       |
| `hadir_scheduler_jobs_failed_total`          | counter | tenant, job                  |
| `hadir_email_send_total`                     | counter | tenant, provider, status     |
| `hadir_active_sessions`                      | gauge   | tenant                       |

`install_scheduler_failure_listener(scheduler, job_name=...,
tenant_id=...)` is the helper every scheduler calls inside
`start()`; it registers an APScheduler `EVENT_JOB_ERROR`
listener so any unhandled exception bumps the failure
counter. Wired into the four schedulers we already had:
attendance, retention, notification worker, report runner.

### `/metrics` endpoint

`prometheus-fastapi-instrumentator==7.0.0` is wired into
`hadir.main.create_app`. The instrumentator auto-wraps every
handler with a request-duration histogram + a status-code
counter and exposes them at `/metrics`. The instrumentator
mount lives **outside** the nginx-proxied `/api/` block — so
on production deployments, `/metrics` is reachable only on
the private docker network where Prometheus runs.

### Hot-path instrumentation

* **Capture worker** (`backend/hadir/capture/reader.py`) —
  bumps `hadir_capture_frames_total` per frame read. Best-
  effort try/except around the metric call so a stray
  Prometheus client error never sinks the capture loop.
* **Detection emitter** (`backend/hadir/capture/events.py`)
  — bumps `hadir_detection_events_total` per event with
  `identified="true"` when the matcher pinned an
  `employee_id`, `"false"` otherwise.
* **Health-snapshot writer** (same file) — sets
  `hadir_camera_reachable` to 0/1 in lock-step with the
  per-minute snapshot insert.
* **Attendance scheduler** — bumps
  `hadir_attendance_records_computed_total` by the
  per-tenant upsert count.
* **Notification worker** (`backend/hadir/notifications/worker.py`)
  — emits `hadir_email_send_total` for every
  `sent`/`failed`/`skipped_pref`/`skipped_no_email` outcome,
  labelled by the `provider` (`smtp`/`microsoft_graph`/
  `recorder` in dev / `disabled`).
* **Active sessions** — refreshed on the same 30 s tick the
  worker uses for email drains. SELECT runs inside the
  per-tenant `tenant_context` so the schema picks up
  cleanly.

### Compose

* `docker-compose.prod.yml` ships **prometheus**,
  **alertmanager**, and **grafana** services on the
  `hadir-internal` network. Prometheus + Alertmanager are not
  port-exposed on the host; Grafana is exposed on
  `${HADIR_GRAFANA_PORT:-3000}` with an admin password
  required at deploy time.
* `docker-compose.observability.yml` is a dev-friendly
  overlay that brings the same three services up alongside
  the dev stack with :9090 + :9093 + :3000 all reachable so
  operators can author dashboards locally.

### Provisioning

* `deploy/prometheus/prometheus.yml` — 15 s scrape interval,
  scrapes `backend:8000`, loads `alerts.yml`, talks to
  `alertmanager:9093`.
* `deploy/prometheus/alerts.yml` — four rules:
  * `HadirCameraUnreachable` (warning, 5 min)
  * `HadirCaptureRateLow` (warning, 10 min, gated on
    reachable=1)
  * `HadirSchedulerJobFailing` (critical, ≥3 failures in 15
    min)
  * `HadirEmailFailureRateHigh` (warning, > 10% over 1 hour
    with NaN-safe denominator)
* `deploy/alertmanager/alertmanager.yml` — webhook +
  email receivers driven by env (
  `HADIR_ALERTMANAGER_WEBHOOK_URL`,
  `HADIR_ALERTMANAGER_SMTP_*`,
  `HADIR_ALERTMANAGER_ADMIN_EMAIL`). Either path can be
  left unset.
* `deploy/grafana/provisioning/datasources/prometheus.yml`
  + `deploy/grafana/provisioning/dashboards/hadir.yml`
  drive auto-provisioning.
* `deploy/grafana/dashboards/hadir.json` — seven panels:
  capture rate per camera, identification rate per tenant,
  attendance records computed per hour, camera reachability
  matrix, HTTP p50/p95/p99 latency, email delivery success
  rate, active sessions. `allowUiUpdates: false` in the
  provider so operators editing in the UI export back to
  source.

### Dev-only `_test/tick_metrics`

A new endpoint mounted only when `HADIR_ENV=dev` ticks every
custom counter once so the dashboard demo doesn't depend on
a live RTSP camera. Bumps:

* 24 frames on `tenant=1 camera=99001`
* 2 identified + 1 unidentified detection events
* 7 attendance records computed
* 2 sent + 1 failed email send (smtp)

Useful when authoring dashboards locally; doesn't ship in
production builds (the dev-only `_test_endpoints` package is
gated by `HADIR_ENV` per pilot-era P13 rule).

## Backward compatibility

* All 435 prior tests pass — no regressions.
* `/metrics` is a NEW endpoint; nothing existing depended on
  its absence.
* Two new pip deps (`prometheus-client==0.21.0`,
  `prometheus-fastapi-instrumentator==7.0.0`) — both pure-
  Python, no system deps.
* The dev compose is unchanged. The new
  `docker-compose.observability.yml` is opt-in (operators
  who don't bring it up don't pay any cost).
* The four scheduler `start()` methods now register an
  APScheduler `EVENT_JOB_ERROR` listener — additive, doesn't
  change scheduling cadence.
* `_test/tick_metrics` is dev-only (gated on
  `HADIR_ENV=dev`). Production builds don't even mount the
  router.

## Notes for whoever resumes next

* **Internal-only is the load-bearing red line.** Don't
  proxy `/metrics` from nginx — Prometheus already reaches
  the backend over the private docker network. Future work
  that wants metrics off-host should ship a Prometheus
  remote-write to a managed TSDB rather than exposing the
  scrape endpoint.
* **Label cardinality.** Counters are fine to label by
  `tenant` + `camera_id`. Don't add `employee_id` to a
  counter that ticks once per detection — that would balloon
  the cardinality and accidentally surface PII via the label
  set's lookup table. The `_safe_label` helper caps any
  label value at 128 chars as a defence-in-depth.
* **Histogram buckets.** The instrumentator uses Prometheus'
  default request-duration buckets (5 ms → 10 s). PDF
  generation lands in the 10 s bucket; if you ever add a
  panel for "what % of /api/reports/*.pdf calls land in the
  > 10 s bucket?", customise via the Instrumentator
  constructor's `buckets=` kwarg.
* **Alertmanager templates.** The default email template is
  Alertmanager's stock one. Operators who want branded
  alerts mount a custom template into
  `/etc/alertmanager/templates/` and add a
  `templates: [...]` entry to the YAML. Out of scope for
  this phase.
* **Prometheus retention.** TSDB retention is 30 days
  (`--storage.tsdb.retention.time=30d`). Operators who want
  longer trend lines either bump the volume size + the flag
  OR ship to a central long-term store via
  `remote_write`. The pilot's `prometheus_data` named
  volume has no size cap on Docker Desktop; on a real
  Linux host the volume's filesystem is the limit.
* **Active sessions gauge accuracy.** Refreshed on the
  notification worker's 30 s tick — there's up to a 30 s
  lag between a session creation and Grafana seeing it. If
  a future phase wants tighter resolution, attach the
  refresh to the auth router's create/touch/delete
  callbacks rather than tightening the tick interval (which
  would also speed up email drains, an unrelated change).
* **Open critical item carries forward**: Omran HR native-
  speaker review of the Arabic translations from P21 is
  still pending before v1.0 launch.
