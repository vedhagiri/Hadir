# P4 — Per-tenant branding

**Phase:** v1.0 P4 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P4 — Per-tenant branding"`

## Goal

Each tenant picks a primary colour from a curated palette, a font
from a curated list, and an optional logo. Changes apply at the
next page load (and live in the editor, via a small ContextVar-free
preview hook). Hard BRD red lines: no free-form hex entry, no custom
CSS upload, no font uploads.

## What was done

### Migration 0010 — `tenant_branding`

`alembic/versions/0010_tenant_branding.py`. One row per tenant in
each tenant schema:

- `tenant_id INTEGER PRIMARY KEY`, FK to the **unqualified**
  `tenants(id)` (search_path resolution puts this on the global
  registry — schema-agnostic, no hardcoded `public` literal in the
  migration source).
- `primary_color_key TEXT NOT NULL DEFAULT 'teal'` with
  `CHECK (primary_color_key IN ('teal','navy','slate','forest',
  'plum','clay','rose','amber'))`.
- `logo_path TEXT NULL` — set when an admin uploads a logo.
- `font_key TEXT NOT NULL DEFAULT 'inter'` with
  `CHECK (font_key IN ('inter','lato','plus-jakarta-sans'))`.
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT now()`.

`hadir_admin` owner; `hadir_app` full CRUD. Idempotent default seed
at the end of the migration:
```sql
INSERT INTO tenant_branding (tenant_id)
SELECT id FROM tenants
WHERE schema_name = current_schema()
ON CONFLICT (tenant_id) DO NOTHING;
```
That seeds Omran (id=1) when running against `main` and the
appropriate row for any tenant schema iterated by the orchestrator.
Provisioning's create_all path materialises the table for new
tenants and the CLI now also `INSERT`s the default branding row.

### `hadir/branding/` package

- `constants.py` — `BRAND_PALETTE` (8 OKLCH triples; each tested in
  light + dark mode), `FONT_OPTIONS` (3 families), `ALLOWED_*` sets.
  Mirrors the Postgres CHECK constraints; change both together.
- `schemas.py` — `BrandingResponse`, `BrandingPatchRequest` (with
  `validated_color()` / `validated_font()` raising `ValueError` on
  off-list keys).
- `repository.py` — `get_branding` (lazy-creates the row on first
  read), `update_branding` (partial update + bumps `updated_at`).
- `logo.py` — magic-byte validation: PNG by `89 50 4E 47 …` prefix,
  SVG by `<?xml`/`<svg` opener + `</svg>` closer; rejects everything
  else with `LogoValidationError`. 200 KB hard size limit. Writes
  to `/data/branding/{tenant_id}/logo.{ext}` (replacing any prior
  logo), reads, deletes.
- `css.py` — derives the `:root { --accent: …; … }` + `body { font-family: …; }`
  CSS body for one `BrandingRow`. Per-process cache keyed on
  `(tenant_id, primary_color_key, font_key)`; `invalidate_tenant`
  clears all entries for a tenant (called after every write).
- `router.py` — two FastAPI routers in one module:
  - `/api/branding` — tenant-side. `GET /` (any user), `PATCH /`
    (Admin), `GET /options` (palette + fonts), `GET /.css` (CSS
    body), `POST /logo` + `DELETE /logo` + `GET /logo`.
  - `/api/super-admin/tenants/{tenant_id}/branding/*` — operator
    surface. Same logic, scoped via `tenant_context(schema_name)`
    after looking up the schema name from `public.tenants`.

### Audit

- Tenant-side `PATCH` writes `branding.updated` to the per-tenant
  `audit_log` (and gets dual-logged via the P3 `_super_admin_context`
  ContextVar if an operator is impersonating).
- Operator-side `PATCH` calls `write_audit_dual` directly so a row
  appears in BOTH `<schema>.audit_log` (tenant Admins see MTS's
  touch) AND `public.super_admin_audit` (operator history).
- Logo upload / delete emit `branding.logo_uploaded` /
  `branding.logo_deleted` with the same dual-write rules. The plain
  bytes never hit the audit row — only `ext` + `size_bytes`.

### Frontend (`frontend/src/branding/`)

- `types.ts` — wire types mirroring the Pydantic responses.
- `hooks.ts` — TanStack Query hooks for both surfaces: `useMyBranding`,
  `useBrandingOptions`, `usePatchMyBranding`, `useUploadMyLogo`,
  `useDeleteMyLogo` for tenant Admin; the same set scoped per
  `tenantId` for the Super-Admin tab.
- `BrandingProvider.tsx` — mounts a single `<style id="hadir-branding-overrides">`
  in `document.head` and rewrites its `textContent` whenever
  `useMyBranding` returns. Pure side-effect — returns null. The
  Layout drops it in above the sidebar.
- `BrandingForm.tsx` — shared editor used by both Admin + operator
  pages. Eight-swatch picker, three font cards, live preview block
  (header + paragraph + Primary/Secondary buttons in the chosen
  palette + font), and an upload zone. Validates 200 KB client-side
  before sending so the server-side rejection never fires for the
  obvious case.
- `BrandingPage.tsx` — `/settings/branding` (Admin nav already had
  a `settings` entry; routed there via `<Navigate to="/settings/branding"/>`).
  Sets `applyToDocument` so the Admin sees their own shell update
  while picking — no reload required.
- `SuperAdminBrandingTab.tsx` — embedded in the existing
  `TenantDetailPage` "Branding" card. Operates on the target
  tenant's id; `applyToDocument` is left off so the operator's red
  console shell stays the operator's red console shell while editing
  another tenant's brand.

### `index.html` font preload

Added `<link rel="stylesheet" …>` to Google Fonts for Inter (400/
500/600/700), Lato (400/700), and Plus Jakarta Sans (400/500/600/
700). Preconnect hints for `fonts.googleapis.com` +
`fonts.gstatic.com`. No runtime font upload — BRD red line.

## Notes for whoever resumes next

- **P5** asks for a second-tenant non-prod smoke. The
  `tenant_demo` tenant the smoke above provisioned was
  deprovisioned to keep the dev DB clean — bring it back with the
  same provisioning command, then flip its branding to navy + lato
  (or anything else) and load `/login` to confirm the colour /
  font show up live.
- **PDF reports** (P17) and **Email** (P18) are the two surfaces
  outside the SPA shell that should also respect the branding row.
  P17 reads `tenant_branding.primary_color_key` for letterhead
  accent + `font_key` for body text + `logo_path` for the header
  image. P18 does the same for HTML email templates. Neither is
  wired in P4 — they're explicit P17 / P18 work.
- The lazy default row in `repository.get_branding` will fire for
  any tenant where the migration's seed `INSERT … FROM tenants
  WHERE schema_name = current_schema()` somehow missed (e.g. a
  tenant provisioned by an earlier release whose schema went
  stamp-only). It's a belt-and-braces fallback; the provisioning
  CLI now seeds the row directly so new tenants don't depend on it.
- **Red lines** (BRD FR-BRD-002, mirrored in PROJECT_CONTEXT §"Tenant
  branding"): no free-form hex, no custom CSS upload, no custom font
  upload. If a future prompt asks to add any of these, the answer is
  **no** without a BRD revision. Curated slots is the design choice —
  not a temporary limitation.
