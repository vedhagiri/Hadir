# P21 — Arabic + RTL

**Phase:** v1.0 P21 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P21 — Arabic + RTL"`

## Goal

End-to-end Arabic locale + right-to-left layout. Every operator-
facing surface — login, sidebar, topbar, approvals inbox, my-
requests, notifications bell, notifications history,
notification preferences, settings tabs — renders in either
English or Arabic, with `<html dir="rtl">` flipping the layout
when Arabic is active. Notification subjects + bodies are also
translated so the email a recipient receives matches their UI
language. Per-user preference persists on `users.preferred_language`.

## What was done

### Migration 0022 — `users.preferred_language`

`alembic/versions/0022_users_preferred_language.py`. Adds a
nullable `TEXT` column with a CHECK constraint locking the value
to `NULL`, `'en'`, or `'ar'`. Schema-agnostic — runs cleanly under
every tenant schema via the orchestrator. `NULL` means
"follow `Accept-Language`" — only filled when the user
explicitly picks a language.

### `db.py`

`users` gains the `preferred_language` column + the matching
`CheckConstraint`.

### Backend i18n module — `hadir/i18n/`

* `locale.py` — pure resolver:
  * `t(key, lang, **fmt)` — dotted-key lookup with English
    fallback and `str.format` interpolation; missing keys return
    the key itself so a typo in a producer surfaces visibly.
  * `parse_accept_language(header)` — q-weighted parse with
    regional sub-tag folding (`en-US` → `en`).
  * `resolve_language(*, user_preference, accept_language)` —
    user pref > Accept-Language > default. Matches the prompt's
    detection chain.
* `en.yaml`, `ar.yaml` — bundled string stores. Top-level
  namespaces: `notifications.{6 categories}.{subject,body}`,
  `email.*`, `errors`, `categories`, `stages`, `statuses`. YAML
  chosen over JSON for nested keys + Unicode-without-escaping
  (operators editing translations don't fight backslashes).
* PyYAML 6.0.2 added to `pyproject.toml`.

### Auth surfaces (backend)

* `CurrentUser` (`hadir/auth/dependencies.py`) gains
  `preferred_language: str | None`. `_load_current_user_bundle`
  reads it; `current_user` propagates it.
* `MeResponse` (`hadir/auth/router.py`) gains
  `preferred_language` so `/api/auth/me` ships the value to the
  frontend on every page load.
* `PATCH /api/auth/preferred-language` — accepts `{en, ar, null}`,
  audits as `auth.preferred_language.updated`, refuses synthetic
  Super-Admin (no real `users` row), DB CHECK rejects anything
  else as a defence-in-depth backstop.

### Producers (backend)

Every wrapper in `hadir/notifications/producer.py` now resolves
the recipient's preferred language before calling `t(...)`:

* `notify_approval_assigned` — per-recipient lang loop; stage
  label translated via `stages.{lower}` key.
* `notify_approval_decided` — `status_label` via `statuses.{key}`.
* `notify_admin_override` — switches between `body` /
  `body_no_prior` based on the previous decider; status maps to
  `admin_approved` / `admin_rejected`.
* `notify_overtime_flagged`, `notify_camera_unreachable`,
  `notify_report_ready` — same per-recipient pattern.

The load-bearing rule: **two recipients of the same event get
two different translations**. Manager A in Arabic and Manager B
in English both see the same factual event but their bell + email
copy are independent.

### Frontend i18n setup — `frontend/src/i18n/`

* `index.ts` — i18next config with `LanguageDetector` +
  `initReactI18next`. Detection order:
  `localStorage > navigator > htmlTag`. `applyLanguageAttributes`
  sets `<html lang>` + `<html dir="rtl">` on every change (and
  once on boot). `setLanguage(lang)` calls
  `i18n.changeLanguage` then PATCHes `/api/auth/preferred-
  language`. `applyServerPreferred(lang)` applies a
  server-resolved value without firing another network call.
* `locales/en.json`, `locales/ar.json` — namespaced keys covering
  common, login, topbar, nav.sections, nav.items (every nav id),
  approvals (incl. tabs, columns, stages), myRequests (filters,
  columns), notifications (bell, page, categories, preferences),
  settings.tabs, language.
* Wired into `main.tsx` so it boots before any component.

### React surfaces

Every visible string on the cross-cutting surfaces routes
through `useTranslation`:

* `auth/LoginPage.tsx` — title, subtitle, labels, placeholders,
  submit/submitting, OIDC button + break-glass link.
* `shell/Topbar.tsx` — logout button, role-switcher aria-label.
* `shell/LanguageSwitcher.tsx` — new EN / العربية toggle.
* `shell/Sidebar.tsx` — section headers (mapped from English to
  `nav.sections.{key}`) + every nav item label
  (`nav.items.{id}`).
* `notifications/NotificationBell.tsx` — bell label, unread
  aria, dropdown title, mark-all-read, empty, see-all, category
  labels for rows.
* `notifications/NotificationsPage.tsx` — title, subtitle (with
  `<Trans>` for the inline link), table headers, empty/loading,
  per-row mark-read button + category labels.
* `notifications/PreferencesPage.tsx` — title, subtitle, table
  headers, per-row category labels, accessibility labels.
* `requests/ApprovalsPage.tsx` — title, subtitle, tab labels,
  empty, table headers, override CTA, primary chip, stage label
  via `approvals.stages.{key}`.
* `requests/MyRequestsPage.tsx` — title, subtitle, new-request
  CTA, filter chip labels + values, table headers, empty/loading.
* `settings/SettingsTabs.tsx` — every tab label via
  `settings.tabs.{key}`.

`AuthProvider.useMe` calls `applyServerPreferred(me.preferred_language)`
on every successful resolve so a fresh login on another browser
immediately reflects the saved choice.

### CSS logical properties + RTL flips

* `frontend/src/styles/styles.css`:
  * `.sidebar` `border-right` → `border-inline-end`.
  * `.table th` `text-align: left` → `start`.
* `frontend/src/styles/styles-enhancements.css`:
  * `.drawer` flips to `inset-inline-end` + `border-inline-start`
    so the slide-in drawer enters from the correct side under RTL.
  * Calendar header alignment + cell border to logical equivalents.
  * `.cal-legend .dot` `margin-right` → `margin-inline-end`.
  * Mobile sidebar `left/transform` rules made dir-aware.
* `frontend/src/styles/styles-enhancements2.css` — `.mgr-item`
  `text-align: left` → `start`.
* `frontend/src/styles/styles-enhancements3.css` — report tile,
  required-asterisk, docs nav button + endpoint indent +
  list indent all converted to logical properties.
* `frontend/src/styles/styles-enhancements3.css` end-of-file
  block adds `[dir="rtl"]` selectors that flip directional icons
  (chevron-right, chevron-left, arrow-right, arrow-left, plus
  the breadcrumb chevron) via `transform: scaleX(-1)`.

`shell/Icon.tsx` now stamps an `icon-<kebab-name>` class on the
`<svg>` so the CSS RTL rule above can target individual glyphs
without per-component branching.

The verbatim-design rule from pilot CLAUDE.md is intentionally
relaxed for this phase only — the prompt explicitly authorizes
the logical-property sweep, and it's the only viable path to
real RTL support (a sibling stylesheet would have left the base
file ahead under LTR and broken under RTL).

### Tests

* `backend/tests/test_p21_i18n.py` (+15) — `t()` round-trip,
  format-kwargs, missing-key fallback, unknown-lang fallback,
  `parse_accept_language` (q-weighted, regional folding,
  malformed q, empty header), resolver chain (user pref vs
  Accept-Language vs default vs invalid pref), full-bundle
  spot-check for every category × language, PATCH endpoint
  (set, clear via null, reject invalid).
* `backend/tests/test_p21_i18n_lint.py` (+10 parametrised) —
  best-effort heuristic scan of every translated TSX file.
  Flags `>Some Text<` JSX text and `placeholder|title|aria-
  label="Some Text"` attributes that look like multi-word
  English. Skips gracefully when the frontend tree isn't
  mounted; otherwise blocks regressions.

`docker-compose.yml` mounts `./frontend:/frontend:ro` on the
backend service so the lint test can scan TSX from inside the
backend pytest run.

## Notes for whoever resumes next

* **Native-speaker review is the open critical item.** Engage
  Omran HR for a pass through `ar.yaml` + `ar.json`. Likely
  changes are tone, idiomatic phrasing on the BRD-required copy
  ("manager rejection is final unless overridden"), and the
  notification subject lines.
* **Translation keys not yet covered.** A handful of feature
  surfaces (employees, cameras, attendance, dashboards, system,
  audit log, reports) still carry English literals. The lint
  test allow-lists them implicitly by listing only the
  translated files. Future work is mechanical: extract → key →
  use `useTranslation`. The lint test's allow-list is the
  natural backlog.
* **Number / date locale formatting.** This phase translates
  text only — `toLocaleString()` etc. still pick the browser's
  default locale. If Omran asks for Arabic numerals on dates
  and amounts, that's a separate sweep using
  `Intl.NumberFormat(lang)` + `Intl.DateTimeFormat(lang)`.
  Backend: `Decimal.format` + `babel`-style date formatting.
* **RTL bidi gotchas.** Mixed Arabic + Latin text (e.g. an
  English employee_code inside an Arabic sentence) renders
  correctly in modern browsers under `dir="rtl"`, but if a
  reviewer flags weirdness, wrap the Latin token in
  `<bdi>...</bdi>` or `<span dir="ltr">...</span>` rather than
  changing the surrounding direction.
* **Stale i18next bundles.** The locale resources are imported
  statically — Vite bundles them. Editing `ar.json` requires a
  page reload to pick up changes; HMR doesn't replace the
  resource bundle by default. If translation iteration becomes
  painful, swap to `i18next-http-backend` so JSON files load on
  demand.
* **CSS sweep is partial.** I converted every directional rule
  I found via grep across the four design CSS files, but the
  inline-style values (`right: 0`, `marginLeft: 8`) inside
  individual TSX files were converted opportunistically only on
  files I touched. If a stray physical-direction rule shows up
  under RTL, run `grep -n 'margin-left\|margin-right\|padding-
  left\|padding-right\|text-align: \(left\|right\)\|border-
  \(left\|right\)' frontend/src/` and fold those in.
* **Synthetic Super-Admin.** The PATCH endpoint refuses to
  persist for `user.id == 0` (the impersonation synthetic). In
  practice this means the operator's own SaaS-side preference
  doesn't get clobbered by a `setLanguage` call inside the
  impersonation banner — they keep whatever language they
  picked in the SaaS console.
