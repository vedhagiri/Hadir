# P7 — Multi-role user switcher

**Phase:** v1.0 P7 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P7 — Multi-role user switcher"`

## Goal

A user who holds more than one role (`HR` + `Manager`,
`Admin` + `HR`, etc.) can switch between them inside their
existing session. Navigation, page content, and backend
authorisation all re-evaluate against the **active** role, not
against the union of every role the user could ever hold. The
pilot's `primaryRole(roles)` "highest role only" shortcut is
retired in favour of an explicit selection persisted on the
session row.

Hard red line (prompt + BRD FR-AUTH-007): backend authorisation
must re-read the active role per request. Never trust the
frontend to enforce role scope. The implementation makes that
property structural — the active role lives on
`user_sessions.data` and `current_user` reads it on every
authenticated request.

## What was done

### Session storage carries `active_role`

`maugood/auth/sessions.py`:

* `SessionRow` gains a `data: dict[str, Any]` field;
  `load_session` populates it from the `data` JSONB column.
* `create_session` accepts an optional `active_role` and writes
  it onto the row at insert time.
* New `update_active_role(conn, session_id, *, active_role)`
  helper — read-modify-write the JSONB so missing or malformed
  `data` is tolerated. Same defensive pattern as the
  Super-Admin impersonation update from P3.

No migration needed: the `data` column has held arbitrary
session claims since the pilot. Legacy sessions without an
`active_role` key fall back to the user's primary role on the
next read.

### `current_user` narrows roles to `(active_role,)`

`maugood/auth/dependencies.py`:

* `CurrentUser` gains `available_roles: tuple[str, ...]` (the
  full set the user holds) and `active_role: str` (the one role
  they're currently acting as). `roles` carries only
  `(active_role,)` — preserves the contract every existing
  guard checks against, so `require_role`, `require_any_role`,
  and `require_department` re-evaluate per request without
  changes.
* `primary_role(roles)` is the new public helper for highest-
  ranked role selection (used at login + as the no-stored-claim
  fallback). The frontend used to compute this client-side
  (`primaryRole(me.roles)`); that path is now gone.
* The synthetic Super-Admin produced during impersonation
  surfaces with `active_role="Admin"` and the full set in
  `available_roles` — the topbar can still render the dropdown
  visually while the switch endpoint refuses (there's no real
  session row to update).

### `POST /api/auth/switch-role`

`maugood/auth/router.py`:

* New endpoint. Body: `{"role": "<target>"}`. Verifies the role
  is in `current_user.available_roles`; if not, **403**. If
  it's the same role the user is already in, returns the
  current `/me` payload without writing an audit row.
* Otherwise persists the new active role via
  `update_active_role`, writes an `auth.role.switched` audit
  row carrying `before={active_role: …}` /
  `after={active_role: …}`, and returns the refreshed `/me`
  payload (so the frontend can update its cache before
  reloading).
* Refuses for the synthetic Super-Admin (`user.id == 0`) with
  400 — operators QA non-Admin role views by signing in as a
  real test user.

`MeResponse` extended with `active_role` + `available_roles`;
all login surfaces (local + OIDC) seed `active_role` to the
user's primary role at session create.

### Frontend

* `frontend/src/types.ts` — `MeResponse` adds `active_role` +
  `available_roles`. The `roles` field is now the active role
  only (matches the backend contract).
* `frontend/src/auth/AuthProvider.tsx` — new `useSwitchRole()`
  TanStack Query mutation hook.
* `frontend/src/shell/Layout.tsx` — drops `primaryRole(me.roles)`
  in favour of `me.active_role`. Sidebar + Topbar render
  against that single value.
* `frontend/src/shell/Topbar.tsx` — replaces the static role
  badge with a `RoleChip`. When `available_roles.length === 1`,
  it renders the same flat badge the pilot had. When the user
  holds more than one, it becomes a small dropdown listing each
  role; selecting one calls `useSwitchRole` then
  `window.location.reload()`. The reload is intentional and
  matches the prompt — every page that reads `me.active_role`
  re-renders cleanly without piecemeal cache invalidation
  across the dozens of TanStack queries scattered through the
  feature folders.

## Backward compatibility

- All 142 prior tests pass.
- Single-tenant pilot login flow unchanged for users with a
  single role.
- Legacy sessions (no `active_role` claim) keep working —
  `current_user` falls back to the user's primary role on the
  next read.
- `MeResponse.roles` keeps the same shape (a list of role
  codes); existing client code that iterates it continues to
  work, just now operating on a single-element list.

## New tests

`tests/test_role_switcher.py` (+7):

- `test_login_seeds_active_role_to_highest` — login response
  carries `active_role=Admin` for an Admin+HR user.
- `test_me_carries_available_and_active` — `/api/auth/me`
  reflects both fields.
- `test_switch_role_flips_active_and_audits` — successful
  switch + audit row with correct `before` / `after`.
- `test_switch_role_refuses_unheld_role` — 403.
- `test_switch_role_noop_returns_current_state_without_audit` —
  switching to the role you're already in is a 200 with no
  audit churn.
- `test_role_guard_re_evaluates_per_request_after_switch` — an
  Admin+HR user's call to `/api/auth/oidc/config` (Admin only)
  succeeds while active=Admin, **403** the moment they switch
  to HR. Proves the per-request authorisation re-read.
- `test_switch_role_refuses_for_synthetic_super_admin` —
  Super-Admin in impersonation gets 400 from the switch
  endpoint.

## Notes for whoever resumes next

- The seeded M2 matrix (`docs/testing/v1.0-m2-test-accounts.md`)
  uses `.test` domains, which `EmailStr` rejects at the API
  boundary. The test-suite fixtures use `*.maugood`, and the live
  smoke above uses `*.maugoodsmoke.io` — both work. If we want
  the documented matrix to be live-callable from curl, either
  loosen `LoginRequest.email` to a plain `str` (regression on
  validation strictness) or rebrand the matrix domains. Track
  with the EmailStr quirk note already in `P6.md`.
- The reload-on-switch behaviour is documented as the chosen
  approach. If a future phase needs softer transitions
  (in-place fade, no reload), the right answer is to migrate
  the dozens of feature-folder TanStack queries off keys that
  bake the role in. Today nothing does, so the reload is
  effectively free.
- The synthetic Super-Admin's `available_roles` list shows all
  four roles in the topbar even though the switch endpoint
  refuses. Acceptable visual-only inconsistency — the operator
  is in red impersonation context anyway, so the "switch
  refused" hint is a reasonable break-glass surface. If we
  wanted to tighten the UX we'd hide the dropdown when
  `is_super_admin_impersonation=true`.
