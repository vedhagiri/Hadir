# P16 — Admin override

**Phase:** v1.0 P16 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P16 — Admin override"`

## Goal

Make the Admin override path surface clearly in the UI and feed
both the audit log and the notifications queue. The state machine
itself was already in place since P13; the work in this phase is
mostly UX + the audit/notification trail required by BRD
FR-REQ-006.

## What was done

### Migration 0018

`alembic/versions/0018_notifications_queue.py`. One per-tenant
table:

| Table | Notes |
| --- | --- |
| `notifications_queue` | id, tenant_id, recipient_user_id (nullable; SET NULL on user delete), kind, request_id (nullable; CASCADE on request delete), payload JSONB, created_at, sent_at NULL. Index on `(tenant_id, sent_at)` so P20's drain query stays fast. |

Schema-agnostic — every FK targets unqualified `tenants(id)` /
`requests(id)` / `users(id)`. Same pattern as 0010-0017.
Provisioning's `_APP_CRUD_TABLES` extends to include the table.

### Comment rule tightened

`AdminOverrideBody.comment` was previously `min_length=1`. P16
bumps it to `min_length=10` plus a `model_validator` that strips
whitespace before re-checking — a 9-character comment, or a
whitespace-padded short comment, returns 422 with
"String should have at least 10 characters". The frontend mirrors
the rule for UX; the server is the source of truth (the P16 red
line: "override without a comment is a 400 at the API level").

### Audit row enrichment

`request.admin.{approve,reject}` now records:

* `before.status` — the status the row was in immediately before
  the override.
* `before.previous_stage` — `"hr"`, `"manager"`, or `null`. Picked
  by the new `_identify_previous_decider` helper — HR wins if there
  was an HR decision, else Manager, else None (Admin acting on a
  fresh `submitted` row).
* `before.previous_decider_user_id` — the user_id of the most
  recent decider, or `null`.
* `after.status` — the new admin_approved / admin_rejected state.
* `after.comment` — the comment **verbatim** (the P16 red line).

### Notifications queue

`_queue_override_notifications` writes one row per audience:

* Manager (when there's a `manager_user_id` on the request).
* HR (when there's an `hr_user_id`).
* Employee — resolved by lower-cased email match against `users`;
  if the match misses the row's `recipient_user_id` is left NULL
  and the payload carries `recipient_email` so P20's delivery can
  still email them.

Each payload carries `request_id`, `decision`, `comment`,
`actor_user_id`, `actor_email`, `previous_stage`,
`previous_decider_user_id`, `new_status`, and `kind`. A user who
happened to be both the Manager and HR decider gets exactly one
row, not two — `_queue_override_notifications` dedupes on
`recipient_user_id`.

### Frontend

* **`frontend/src/requests/OverrideModal.tsx`** — new modal. Red
  alert banner reads "Overriding the {Manager/HR} decision. This
  will be audit-logged and visible to all parties (the original
  {stage} decider, the employee, and any administrator reviewing
  the request later)." (The stage name is derived from the same
  precedence the backend uses — HR > Manager > "the pending
  Manager" / "the pending HR review" / "the prior".) Decision
  pills (approve / reject), required comment textarea with the
  client-side `min 10 chars` rule mirrored from the server. The
  submit button stays disabled until 10 trimmed characters land in
  the textarea; the action button uses a danger tone.
* **Approvals row action** — every Admin row in the Approvals
  table now has an "Override" button to the right of the chevron.
  Click stops propagation so it doesn't also open the detail
  drawer. Pending and Decided tabs surface the same affordance —
  the "All" tab admins reach today via the Pending and Decided
  lists. (A separate "All" list-view follow-up could be wired later
  if HR asks; the prompt's data is already in `useRequests` for
  Admin via the role-scoped GET.)
* **Detail drawer marker** — the timeline now labels the override
  stage **"⚠ Overridden by admin"** with the same comment +
  timestamp it always rendered, satisfying the P16 spec.

### Routing / nav

No new routes. The modal is overlay-only and rendered inside
`ApprovalsPage`.

## Notes for whoever resumes next

- **P20 (Notifications)** drains `notifications_queue` (oldest
  unsent first). The `kind` field is the discriminator — P16 only
  emits `override.{manager,hr,employee}_notified` today; P20's
  template-renderer keys off it. The `recipient_email` payload
  fallback covers the "no users row matches" case so P20 doesn't
  have to re-derive it.
- **HR auto-routing** still keys off the manager-approved status
  set; an Admin override on a manager-rejected row doesn't
  re-engage HR. That mirrors the BRD's "Admin override is final"
  language — the override outcome is terminal.
- **Override-on-submitted edge case.** Admin can override a fresh
  request before either Manager or HR has decided. The audit row
  records `previous_stage=null`; the modal copy gracefully falls
  back to "Overriding the pending Manager decision". P14's
  Approvals "All" view exposes this row; the override button is
  rendered for any Admin-visible request regardless of state.
- **`.test` TLD gotcha.** Same as P11–P15 — Pydantic `EmailStr`
  rejects reserved TLDs, so the smoke uses `.hadir`.
- **Idempotent re-override?** Re-POSTing an override on an already-
  overridden row is allowed (Admin override always returns
  admin_approved / admin_rejected per the state machine) and will
  queue a fresh batch of notifications. P20 should treat the
  per-`(request_id, kind, recipient)` triple as the dedupe key if
  this turns out to be noisy in practice.
