# P25 — Log rotation + retention cleanup + PDPL delete

**Phase:** v1.0 P25 (M3 hardening — third phase)
**Date (UTC):** 2026-04-26
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P25 — Log rotation +
retention cleanup"`

## Goal

Three pieces of operational hygiene:

1. **Log rotation** so `backend/logs/{app,audit}.log` don't
   eat the disk in production.
2. **Retention cleanup** so `camera_health_snapshots`,
   `notifications`, `report_runs`, and `user_sessions`
   trim themselves to BRD-mandated windows.
3. **PDPL right-to-erasure** so an Admin can satisfy a
   PDPL/GDPR delete request without dropping the audit
   trail or the attendance history.

The protected tables — `audit_log`, `attendance_records`,
`detection_events`, `employees`, `employee_photos`,
`requests`, `approved_leaves` — are never purged by the
retention sweep. That's the load-bearing red line and the
test suite enforces it.

## What was done

### `backend/maugood/logging_config.py`

* `GzipRotatingFileHandler` extends the stdlib
  `TimedRotatingFileHandler`. Rotates at midnight UTC; on
  rotation, every just-rotated sibling that isn't already
  `.gz` gets compressed in place and the plain copy
  unlinked. ~80% size win on a typical Maugood log.
* `configure_logging(*, log_dir, backup_count, enable_files)`
  is the idempotent entrypoint. Sets up the root logger
  (stdout + `app.log`) and the `maugood.audit` logger
  (separate `audit.log`, `propagate=False` so it doesn't
  echo to root). Tests opt out of the file handlers via
  `MAUGOOD_LOG_DISABLE_FILES=1`.
* `audit_logger()` is the convenience accessor producers
  use — `audit_logger().info(...)` lands in
  `audit.log` only, never in the root file or stdout.
* `main.create_app` calls `configure_logging()` once at
  boot. Backend container's WORKDIR is `/app`, so the
  default `MAUGOOD_LOG_DIR=/app/logs` resolves to a writable
  directory that operators can mount as a volume.

Backup count default is 30 (BRD §"Logs"). Override via
`MAUGOOD_LOG_BACKUP_COUNT`.

### Migration 0024

`alembic/versions/0024_employees_status_deleted.py`. Expands
the `employees.status` CHECK to allow `deleted` (added the
PDPL terminal state). Schema-agnostic — runs cleanly under
every tenant schema. Downgrade refuses to flip back to a
2-value enum if any rows already use the new state.

### `backend/maugood/retention/`

Two modules:

* `sweep.py` — pure(-ish) DELETE worker. Iterates every row
  of `public.tenants`, applies four DELETE statements per
  schema:
  * `camera_health_snapshots` where `captured_at < now() - 30d`
  * `notifications` where `created_at < now() - 90d`
  * `report_runs` where `finished_at < now() - 90d`
    (drops the on-disk file first, then the row)
  * `user_sessions` where `expires_at < now() - 7d`
  Each tenant's sweep writes one `audit_log` row with the
  delete counts (action `retention.swept`) and one line to
  `audit.log`.
  The sweep **never** queries the protected tables; that's
  enforced by an explicit assertion in the test suite that
  snapshots their row counts and asserts no shrinkage.
* `scheduler.py` — APScheduler `BackgroundScheduler` cron at
  `03:00` in `MAUGOOD_LOCAL_TIMEZONE` (default `Asia/Muscat`).
  Wired into `main.py`'s lifespan alongside the existing
  schedulers; neutralised in `tests/conftest.py` the same way.

Cutoffs are env-overridable (`MAUGOOD_RETENTION_*_DAYS`); 0 or
non-numeric values fall back to the BRD defaults.

### `backend/maugood/employees/pdpl.py`

`pdpl_delete_employee(conn, scope, *, employee_id, actor_user_id,
confirmation_phrase)`:

1. Loads the existing employee row (404s if missing, 409s if
   already `deleted`).
2. Drops every `employee_photos` row + Fernet file.
3. Drops every `custom_field_values` row.
4. Sets `full_name='[deleted]'`,
   `email='deleted-{id}@maugood.local'`, `status='deleted'`.
5. Calls `matcher_cache.invalidate_employee` so a captured
   face never re-matches post-delete.
6. Writes an `audit_log` row with action `pdpl_delete` —
   the audit row preserves the *before* PII so an auditor
   can verify the request was honoured. The audit row is
   append-only at the DB grant level so it cannot be
   expunged later.
7. Echoes a one-line breadcrumb to `backend/logs/audit.log`
   (no PII echo in the file copy).

`PDPL_CONFIRMATION_PHRASE = "I CONFIRM PDPL DELETION"` is the
exact (case + whitespace sensitive) phrase the router checks.
Exposed for the UI to render verbatim.

### Endpoint

`POST /api/employees/{id}/gdpr-delete` — Admin-only,
confirmation phrase in body. 400 on wrong phrase, 404 on
missing, 409 on already-deleted, 200 on success with
delete-counts response.

The endpoint also unblocks reads on a deleted employee: the
schema literal split (`Status` writable vs `StatusOut`
readable) means PATCH still can't push rows into `deleted`,
but GET cleanly serialises a deleted row. Without the split
the GET handler 500'd post-PDPL.

### Tests

`backend/tests/test_p25_retention_pdpl.py` (+10):

* `GzipRotatingFileHandler` rotates and gzips correctly.
* `configure_logging` writes to `audit.log` only (not
  duplicated to `app.log`).
* Retention sweep deletes old `camera_health_snapshots` but
  preserves recent ones.
* Retention sweep deletes long-expired `user_sessions` but
  preserves recently-expired ones.
* Retention sweep **never** shrinks the seven protected
  tables.
* PDPL delete redacts name + email, drops photos + cfv,
  removes the on-disk photo file.
* PDPL delete refuses without the exact phrase (400).
* PDPL delete 404s on a missing employee.
* PDPL delete 409s on an already-deleted one.
* PDPL delete 403s when the caller is `Employee` role.

  readable message) rather than corrupt the constraint.

  inside the existing backend process.

## Notes for whoever resumes next

* **Audit log is the load-bearing red line.** The retention
  sweep never queries `audit_log`; the test suite enforces
  no shrinkage on every protected table. Any future phase
  that wants to "thin" audit history needs explicit BRD
  sign-off + a separate, audited workflow — not a tweak to
  this sweep.
* **PDPL audit row preserves the previous PII.** The
  before-state full_name + email land in the audit row's
  `before`/`after` payload so an auditor can verify the
  request was honoured. That's a deliberate trade-off — the
  alternative ("redact everywhere including the audit
  trail") makes the deletion unverifiable.
* **Schema literal split.** `Status` (write) vs `StatusOut`
  (read) means PATCH still can't push rows into `deleted`.
  If a future phase adds another terminal state (e.g.
  `archived` for a tax-history view), follow the same
  split.
* **Encrypted photo files on disk.** The Fernet ciphertext
  is left in the operator's volume after PDPL delete (we
  unlink the active file path, but a backup taken before
  the delete still has the ciphertext). The DR rehearsal
  doc (P24) should call this out — operators with strict
  PDPL interpretations must rotate `MAUGOOD_FERNET_KEY` to
  invalidate the backup copy. v1.x key-escrow work
  documents the per-employee rotation flow.
* **Daily 03:00 in Asia/Muscat.** The retention scheduler
  uses `MAUGOOD_LOCAL_TIMEZONE`. Operators serving multiple
  tenants in different regions should pin the sweep to the
  off-peak hour for their main tenant — running the sweep
  during a different tenant's working hours is harmless
  (DELETEs are quick) but ugly in the logs.
* **`audit.log` rotation and PDPL.** The PDPL flow writes
  one line to `audit.log`. After 30 days that line moves to
  `audit.log.<date>.gz` — operators who want a longer tail
  set `MAUGOOD_LOG_BACKUP_COUNT=365` (or higher). The DB-
  side `audit_log` row is the durable record regardless.

  speaker review of the Arabic translations from P21 is
  still pending before v1.0 launch.
