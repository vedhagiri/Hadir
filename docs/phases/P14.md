# P14 — Request submission UI

**Phase:** v1.0 P14 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P14 — Request submission UI"`

## Goal

Wire up the Employee-facing surface on top of the P13 state machine.
Three pieces ship together: the My Requests page (filterable list +
new-request entry point), the New Request drawer (type radio +
reason-category dropdown sourced from a per-tenant table + optional
attachment dropzone with size + magic-byte enforcement), and the
detail drawer (status timeline + attachments with download links +
cancel-while-submitted). Plus an Admin settings page for the reason
categories themselves.

## What was done

### Migration 0017

`alembic/versions/0017_request_reason_categories.py`. One new
per-tenant table:

| Table | Notes |
| --- | --- |
| `request_reason_categories` | id, tenant_id, request_type ['exception','leave'], code, name, display_order, active, timestamps. UNIQUE (tenant_id, request_type, code). CHECK on `request_type`. Seeded with the BRD §FR-REQ-008 list — 5 exception (Doctor / Family / Traffic / Official / Other) + 4 leave (Annual / Sick / Emergency / Unpaid). |

Schema-agnostic seed via `INSERT ... FROM tenants WHERE schema_name =
current_schema() ON CONFLICT DO NOTHING`. Provisioning seeds the
same defaults inline.

### Config knobs

`backend/hadir/config.py` gains two fields:

* `request_attachment_max_mb` (default 5) — the cap the prompt asked
  for. Surfaced via `GET /api/requests/attachment-config` so the
  client can pre-validate. Server enforces independently — the P14
  red line.
* `request_attachment_root` (default `/data/attachments`) — where
  Fernet-encrypted files land on disk: `{root}/{tenant_id}/requests/{uuid}.{ext}`.

### Backend reason-category CRUD

`hadir/requests/reason_categories.py` (DB access) +
`reason_categories_router` in `router.py`. Mounted at
`/api/request-reason-categories`. **Reads** are open to every
authenticated role (every form needs the dropdown); **writes** are
Admin-only. Audit hooks on every mutation:
`request_reason_category.{created,updated,deleted}`.

### Backend attachments

`hadir/requests/attachments.py` owns the validation + on-disk
plumbing:

* `validate_and_store(...)` — sniffs magic bytes against an explicit
  signature table (JPEG / PNG / GIF / WEBP / PDF / ZIP), pairs ZIP
  matches with `.docx` content-type or filename to refuse a bare
  ZIP, enforces the per-tenant max-MB cap, then Fernet-encrypts the
  bytes and writes them under `{root}/{tenant_id}/requests/{uuid}.{ext}`.
* `read_decrypted(file_path)` — symmetric inverse used by the
  download endpoint. Same `HADIR_FERNET_KEY` the photos pipeline
  uses.
* Raises `AttachmentError` with operator-safe messages; the router
  maps "max is" messages to **413** and everything else to **400**.

Endpoints (all on the existing `/api/requests` router so they sit
naturally under `/{request_id}/...`):

| Method + Path | Notes |
| --- | --- |
| `GET /api/requests/attachment-config` | Returns `max_mb` + `accepted_mime_types`. Declared **before** the `/{request_id}` route so FastAPI's matcher picks the static path. |
| `POST /api/requests/{id}/attachments` | Owner-only modify (Employee for own + still-submitted; Admin always). 413 on size, 400 on bad magic, 403 on stage gate. |
| `GET /api/requests/{id}/attachments` | Same role visibility as the parent request — Employee own, Manager assigned, HR what-reached-them, Admin all. |
| `GET /api/requests/{id}/attachments/{aid}/download` | Decrypts on the fly + writes a `request.attachment.downloaded` audit row. |
| `DELETE /api/requests/{id}/attachments/{aid}` | Owner-only modify, status=submitted only. |

Audit hooks on every mutation:
`request.attachment.{uploaded,downloaded,deleted}`.

### Frontend `frontend/src/requests/`

* `types.ts` — wire types for requests, reason categories, and
  attachments.
* `hooks.ts` — TanStack Query hooks for every endpoint above plus
  the existing P13 surface (request list/get/cancel, attachment
  upload/list/delete, reason-category CRUD).
* `StatusPill.tsx` — shared status pill that maps each of the 8
  states to a tone + human label.
* `MyRequestsPage.tsx` — Employee `/my-requests` route. Filter chips
  for type (all / exception / leave) + status (open / approved /
  rejected / cancelled). Empty state. Row click opens the detail
  drawer.
* `NewRequestDrawer.tsx` — type radio toggles between exception
  (single date) and leave (range + leave-type dropdown sourced from
  `useLeaveTypes()` from P11). Reason category sourced from
  `useReasonCategories(type)` and filtered to `active=true`.
  Attachment dropzone uses native HTML5 drag-and-drop, validates the
  client-side size against the same `max_mb` returned by the config
  endpoint, then POSTs the parent request and uploads in sequence
  (the upload failing doesn't roll back the request — the operator
  can re-attach from the detail drawer later).
* `RequestDetailDrawer.tsx` — status timeline (Submitted → Manager
  → HR plus an Admin override stage when present), attachment list
  with `Content-Disposition`-honouring blob downloads, "Cancel
  request" button when allowed, "Add another file" affordance while
  the row is still submitted.
* `ReasonCategoriesPage.tsx` — Settings → Request reasons. Two
  segmented tables (Exception / Leave) with inline create / activate
  / delete. Wired into the shared `SettingsTabs`.

Routes registered in `App.tsx`:

* `/my-requests` → `MyRequestsPage`
* `/settings/reason-categories` → `ReasonCategoriesPage`

The `my-requests` id is already in `nav.ts` for Manager + Employee
(seeded since P4); routing now resolves that nav link to the real
page instead of the placeholder.

## Notes for whoever resumes next

- **Approvals inbox (P15)** is the natural follow-up: it consumes
  the same `GET /api/requests` (role-scoped to Manager / HR), opens
  the existing `RequestDetailDrawer` with `allowOwnerActions=false`,
  and adds a decision footer that hits
  `POST /api/requests/{id}/manager-decide` or `/hr-decide`. No
  backend work needed.
- **Admin override UI (P16)** drops a comment box + the
  `POST /api/requests/{id}/admin-override` call onto the same
  drawer. The Pydantic `model_validator` already rejects an empty
  comment with 422, so the UI should keep the submit button
  disabled until at least one non-whitespace character is typed.
- **Magic-byte table.** The accepted set deliberately excludes
  legacy formats (BMP, TIFF, RTF, .doc) — adding one is a tiny
  diff: append a `(signature, mime, ext)` tuple to `_SIGS` in
  `attachments.py` and add the mime to `ALLOWED_TYPES`.
- **Server-enforced size.** The client-side check exists for UX —
  the server is authoritative. Setting
  `HADIR_REQUEST_ATTACHMENT_MAX_MB=10` in env raises both the
  surfaced cap and the actual limit; the docker-compose .env should
  pin a sane default before production.
- **Encrypted attachments survive a key rotation only if the new
  key is added to a Fernet `MultiFernet`.** Today we use a single
  Fernet — a key rotation invalidates every existing attachment.
  Same trade-off the photos pipeline made; revisit when v1.0 wires
  rotation.
- **`.test` TLD gotcha.** The smoke uses `.hadir` for the throwaway
  Employee account because the seeded `*.omran.test` accounts can't
  log in (Pydantic `EmailStr` rejects reserved TLDs). Same
  workaround as P11/P12/P13.
