# P18 — Scheduled reports + email

**Phase:** v1.0 P18 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P18 — Scheduled reports + email"`

## Goal

A recurring-report subsystem on top of P17's PDF + the existing
Excel builder. Operators define schedules in the UI; an APScheduler
job picks each one up at its cron time, runs the report, and emails
the file (attached when small, signed-URL link when large). Email
credentials per tenant — SMTP or Microsoft Graph — secrets
Fernet-encrypted at rest. Signed URLs are HMAC-token gated, IP
rate-limited, and audit-logged on every access.

## What was done

### Migration 0019

`alembic/versions/0019_email_and_schedules.py`. Three per-tenant
tables:

| Table | Notes |
| --- | --- |
| `email_config` (tenant_id PK) | provider ('smtp'\|'microsoft_graph'), SMTP host/port/username/password_encrypted/use_tls, Graph tenant_id/client_id/client_secret_encrypted, from_address, from_name, enabled, updated_at. CHECK on provider. Empty row seeded for the migrating tenant. |
| `report_schedules` | id, tenant_id, name, report_type ('attendance'), format ('xlsx'\|'pdf'), filter_config JSONB, recipients JSONB, schedule_cron, active, last_run_at, last_run_status, next_run_at, created_by_user_id, timestamps. Index on (tenant_id, active, next_run_at). |
| `report_runs` | id, tenant_id, schedule_id (CASCADE), started_at, finished_at, status ('running'\|'succeeded'\|'failed'), file_path, file_size_bytes, recipients_delivered_to JSONB, error_message, delivery_mode ('attached'\|'link'). |

Schema-agnostic — every FK targets unqualified `tenants(id)`.
Provisioning seeds the empty `email_config` row inline so a
freshly-provisioned tenant just needs the operator to fill in
credentials.

### Config knobs

`backend/hadir/config.py`:

* `email_attachment_max_mb` (default 10) — files at-or-below this
  go in the email body; larger gets a signed-URL link. `0` is a
  valid choice for "always link".
* `report_output_root` (default `/data/reports`) — where the runner
  writes generated files. Layout: `{root}/{tenant_id}/runs/{uuid}.{ext}`.
* `report_signed_url_secret` — HMAC key. Rotating invalidates every
  outstanding token.
* `report_signed_url_ttl_days` (default 7) — token TTL.
* `report_signed_url_rate_limit_per_minute` (default 30) — per-IP
  bucket on the validation endpoint.
* `report_runner_poll_seconds` (default 60) — APScheduler interval.

### Python deps

`pyproject.toml` adds `croniter==2.0.5`. SMTP is stdlib; Graph uses
the existing `httpx` (no `msal` SDK — token exchange + sendMail are
two HTTP calls).

### Email module — `hadir/emailing/`

* `secrets.py` — `encrypt_secret` / `decrypt_secret` over
  `HADIR_AUTH_FERNET_KEY`. Same key the OIDC `client_secret` from
  P6 uses; the photo/RTSP key stays focused on biometric +
  camera secrets.
* `providers.py` — `EmailMessage` value object, `EmailSender`
  protocol, `SmtpSender`, `GraphSender`, plus a pluggable
  `set_sender_factory` so tests can swap in a recording sender. A
  cross-process file recorder (`HADIR_EMAIL_RECORDER_PATH`) gives
  the live smoke a way to capture a delivery without standing up a
  real SMTP server.
* `templates/report.html` — inline-styled email body honouring
  tenant branding (accent hex + name in the header). Two variants:
  attached copy ("attached as foo.pdf, 12.3 KB") and link copy
  ("Download PDF" button + TTL note).
* `render.py` — Jinja env + `render_report_email_html(context)`.

### Signed URL — `hadir/scheduled_reports/signed_url.py`

Token format `{run_id}.{exp}.{sig}` (base64url HMAC-SHA256). Pure;
the validation endpoint translates `TokenError` into HTTP 403 with
the operator-safe detail.

### Runner — `hadir/scheduled_reports/runner.py`

* `run_schedule_now(scope, schedule_id)` — the engine. Inserts a
  `report_runs` row with status=`running`, builds the report
  (XLSX or PDF), picks attach-vs-link based on the size cap,
  renders the email body via Jinja, dispatches via the configured
  provider, then updates the run row + bumps the schedule's
  `last_run_*` and `next_run_at` (computed by croniter).
* Errors are caught + recorded on the run row; the schedule's
  `next_run_at` always advances even on failure so the runner
  doesn't get stuck in a dispatch loop.
* `_tick()` — the APScheduler-driven scan. `tenant_context("public")`
  for the cross-tenant lookup against the global `tenants` table,
  then per-tenant `tenant_context(slug)` blocks for the
  `report_schedules` queries. Each due schedule runs in turn.
* `ReportRunner` supervisor (`start` / `stop`) wired into the
  FastAPI lifespan. `report_runner.start = lambda: None` for tests
  (autouse fixture in conftest).

### Router — `hadir/scheduled_reports/router.py`

| Method + Path | Role | Notes |
| --- | --- | --- |
| `GET /api/email-config` | Admin | Returns provider + non-secret fields + `has_*_password` boolean flags. |
| `PATCH /api/email-config` | Admin | Empty `smtp_password` / `graph_client_secret` leave the existing ciphertext alone; non-empty rotates. Audit row records `fields_changed` + boolean rotation flags — never the plain secret. |
| `POST /api/email-config/test` | Admin | Sends a one-off test email to the supplied address. |
| `GET /api/report-schedules` | Admin / HR | List per tenant. |
| `POST /api/report-schedules` | Admin | Validates cron via croniter (422 on bad expr). Computes `next_run_at`. |
| `PATCH /api/report-schedules/{id}` | Admin | Cron change recomputes `next_run_at`. |
| `DELETE /api/report-schedules/{id}` | Admin | Past runs survive in the audit log; CASCADE drops `report_runs` rows. |
| `POST /api/report-schedules/{id}/run-now` | Admin | Same engine the background loop uses. |
| `GET /api/report-runs?schedule_id=...` | Admin / HR | Recent runs, newest first. |
| `GET /api/reports/runs/{run_id}/download?token=...` | Anonymous | HMAC validation + per-IP rate limit + audit. Resolves the run's tenant by scanning `public.tenants` (the token is the gate, not the session). |

## Notes for whoever resumes next

- **Token caching for Graph**. `GraphSender` does a fresh token
  exchange on every `send`. Pilot scale (a few schedules per
  tenant per day) is fine — but a per-process `(tenant_id,
  client_id) → (token, exp)` cache lands here when traffic grows.
  Skip the cache during a key rotation so rotations take effect on
  the next send.
- **One-shot signed URLs.** The pilot allows reuse + per-IP rate
  limiting; if HR ever asks for one-shot semantics, add a
  `consumed_at` column on `report_runs` and check it inside
  `download_run`. The audit row already records every access.
- **Email recorder file mode.** `HADIR_EMAIL_RECORDER_PATH` is a
  dev-only convenience — never set it in staging or production.
  The router's `get_sender` reads the env on every send so an
  operator can flip recorder mode without restarting the backend
  for a debugging session.
- **Cron evaluation timezone.** `compute_next_run` uses UTC so the
  same expression in `tenant_omran` and `tenant_demo` fires at the
  same wall-clock instant. The PDF builder still picks "today" in
  the tenant's own timezone (P11), so the report's data range is
  tenant-local even when the cron tick is global.
- **Runner scan window.** A schedule whose `next_run_at` lands
  inside a 60-second tick interval gets fired on the **next** poll
  — average half-minute lag is acceptable for a reporting cadence
  measured in days. Tighter polling (5s) would buy lower lag at
  the cost of useless DB scans.
- **`.test` TLD gotcha.** Same as P11–P17 — Pydantic `EmailStr`
  rejects reserved TLDs, so the recipient list in the smoke uses
  `*.pilot.hadir`.
