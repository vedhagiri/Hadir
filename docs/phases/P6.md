**Phase:** v1.0 P6 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md ¬ß"P6 ‚Äî Entra ID OIDC"`

## Goal

Each tenant configures their own Microsoft Entra ID app, flips
"enabled" on, and from then on the tenant's login page leads with
"Sign in with Microsoft". Local password auth stays available as a
break-glass fallback. The session that comes out of an OIDC
round-trip is identical in shape to the local-auth session ‚Äî same
cookies, same role guards, same audit trail.

Hard red lines (BRD FR-AUTH-006, mirrored in
``backend/CLAUDE.md`` and at the top of ``hadir/auth/oidc.py``):

* **Never auto-provision users from claims.** A successful Entra
  authentication that doesn't match an existing tenant ``users``
  row by lower-cased email returns 403 with the exact prescribed
  message: _"Your Microsoft account is not registered in Hadir.
  Contact your administrator."_
* **Never derive roles from Entra groups.** Roles live in
  ``user_roles`` and are managed only inside Hadir.
* **Never log the client secret or the access token.** Audit rows
  carry ``has_secret: bool`` + ``email_attempted`` only. The
  in-process logger never sees either.

## What was done

### Migration 0011 ‚Äî `tenant_oidc_config`

Per-tenant table:

| Column | Notes |
| --- | --- |
| `tenant_id` (PK) | FK to **unqualified** `tenants(id)` so the migration is schema-agnostic. |
| `entra_tenant_id` | GUID or verified domain. Free-form text ‚Äî Entra accepts both. |
| `client_id` | The Entra app registration's application id. |
| `client_secret_encrypted` | Fernet ciphertext, **separate key** from photos/RTSP ‚Äî see `HADIR_AUTH_FERNET_KEY`. Nullable until the operator types one. |
| `enabled` | Defaults `false`. Flips the LoginPage's primary CTA. |
| `updated_at` | Bumped on every PUT. |

Idempotent default seed (`INSERT ‚Ä¶ FROM tenants WHERE schema_name =
current_schema() ON CONFLICT DO NOTHING`) ‚Äî provisioning's CLI
inserts the row directly for new tenants, the migration handles
existing ones during the orchestrator's loop.

### `hadir/auth/oidc.py`

The whole OIDC flow lives in one module so the audit surface is
easy to read end-to-end. Key pieces:

* **Discovery + JWKS cache** (`_fetch_discovery`, `_fetch_jwks`):
  per-process, 5-minute TTL, keyed on the Microsoft tenant id
  (discovery doc) or JWKS URI (keys). Fronted by a
  `set_test_oidc_provider` hook so tests swap the network calls
  for an in-process fake.
* **Signed state cookie** (`_sign_state` / `_verify_state`): the
  ContextVar-free, library-free implementation ‚Äî `base64(json) .
  base64(hmac_sha256)`. The HMAC key derives from the auth Fernet
  key (`sha256(HADIR_AUTH_FERNET_KEY)`). Cookie TTL is
  `HADIR_OIDC_STATE_TTL_SECONDS` (default 600). Replay across the
  TTL or sig-tamper both fail closed.
* **Token exchange** (`exchange_code`): plain `httpx.Client` POST
  to the token endpoint. Errors swallow the response body before
  raising ‚Äî Entra's error blobs sometimes echo the client_id and
  we don't want anything operator-facing carrying it.
* **ID token validation** (`validate_id_token`): authlib's
  `jwt.decode` with `claims_options` for `iss` / `aud` / `nonce`,
  `validate(leeway=‚Ä¶)` for `exp` / `nbf`. Raises `RuntimeError` on
  any failure; the caller writes the audit row + 400.

### Endpoints

| Method + Path | Purpose | Auth |
| --- | --- | --- |
| `GET /api/auth/oidc/status?tenant=<slug>` | Anonymous probe ‚Äî does this tenant have OIDC enabled? Used by LoginPage. | none |
| `GET /api/auth/oidc/login?tenant=<slug>` | Kicks off the redirect to Entra. Sets the signed state cookie on `/api/auth/oidc` path. | none |
| `GET /api/auth/oidc/callback` | Entra redirects back here. Verifies state, exchanges code, validates ID token, matches email, creates session, sets `hadir_session` + `hadir_tenant` cookies, 302‚Üí`/`. | none |
| `GET /api/auth/oidc/config` | Returns the tenant's OIDC config (secret masked ‚Üí `has_secret: bool`). | Admin |
| `PUT /api/auth/oidc/config` | Partial update. Pings discovery before persisting iff the operator is turning OIDC on or changing the entra_tenant_id. Refuses with 400 if discovery fails. | Admin |

Audit actions (all written to the per-tenant `audit_log`):

* `auth.oidc.login.success` ‚Äî successful sign-in. `after` carries
  the matched user id, ip, session id, tenant_schema, email.
* `auth.oidc.login.failure` ‚Äî every refusal path. `after.reason`
  is one of `state_mismatch`, `token_exchange_failed:<exc-class>`,
  `id_token_invalid:<exc-class>`, `no_id_token`, `no_email_claim`,
  `no_user_match`, `entra_error:<code>`, `config_disabled`.
* `auth.oidc.config_updated` ‚Äî Admin saved a new config.
  `after.secret_rotated: bool` flags whether the secret was
  replaced (the bytes themselves never appear).

### Frontend

* `frontend/src/auth-oidc/{types.ts,hooks.ts,AuthenticationPage.tsx}`
  ‚Äî types mirroring the Pydantic responses, TanStack Query hooks
  for the status probe + Admin config CRUD, and the
  `/settings/authentication` page.
* `frontend/src/auth/LoginPage.tsx` ‚Äî rewritten:
  - Reads the tenant slug from `?tenant=` or asks for it via a
    small picker on entry.
  - Calls `useOidcStatus(slug)`. When `enabled === true`, primary
    CTA is **"Sign in with Microsoft"** (anchor ‚Üí `/api/auth/oidc/login?tenant=‚Ä¶`).
    "Use local account (break-glass)" link reveals the existing
    email + password form, which now sends `tenant_slug` in its
    payload.
  - When OIDC is disabled, primary surface is the local form.
* `frontend/src/App.tsx` ‚Äî adds `/settings/authentication` route.
* `frontend/src/api/client.ts` ‚Äî adds `PUT` to the allowed methods.

### `HADIR_AUTH_FERNET_KEY` separation

New env var, dev placeholder generated alongside the existing
`HADIR_FERNET_KEY` and tracked in `.env.example`. The split is the
whole point of the new key: a leak of the photo/RTSP key doesn't
let an attacker decrypt OIDC client secrets and vice versa.

### Provisioning + grants

`scripts/provision_tenant.py` now seeds a default
`tenant_oidc_config` row (disabled, empty fields) alongside roles,
departments, branding, and the Admin user.
`_APP_CRUD_TABLES` extended so `hadir_app` keeps full CRUD on the
new table inside every fresh tenant schema.

## Notes for whoever resumes next

- **Live test against a real Entra app**: register a new application
  in Microsoft Entra ‚Üí Identity ‚Üí Applications ‚Üí App registrations.
  Add `http://localhost:8000/api/auth/oidc/callback` as a Web
  redirect URI (the dev value of `HADIR_OIDC_REDIRECT_BASE_URL`).
  Production must be HTTPS (FR-AUTH-002). Generate a client secret,
  paste it into the Authentication settings page, toggle
  `enabled=true`, and save ‚Äî the discovery validation step in the
  PUT handler will refuse if the entra_tenant_id is wrong.
- **Email TLD validator quirk**: `LoginRequest.email: EmailStr` (the
  pydantic v2 default) rejects `.test` / `.example` reserved TLDs.
  The seed-test-accounts matrix uses `.test` (per the doc) so live
  curl smoke against those accounts hits 422. Tests bypass this by
  using `@*.hadir` in fixtures. If you need a UI demo, seed an
  Admin with a non-reserved domain or relax the validator.
- **Tenant resolution from Host header**: the prompt mentions
  subdomain routing (`omran.hadir.example.com ‚Üí tenant_omran`).
  P6 supports this via the cookie/picker path; subdomain inference
  in middleware is left for the production-ingress phase (P22ish)
  when the reverse-proxy contract is settled. The cookie is the
  same one P5 introduced.
- **Multi-tenant operator config**: P6 ships only the tenant Admin
  config endpoint (`PUT /api/auth/oidc/config`). A Super-Admin
  operator-side `/api/super-admin/tenants/{id}/oidc/config` would
  be a small addition if MTS needs to set up OIDC for a tenant
  before handing them their first login. Deferred until the need
  shows up.
