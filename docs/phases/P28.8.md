# P28.8 — Worker monitoring + Camera metadata + Pipeline health

**Phase:** v1.0 P28.8 (between P28.7 employee management and M4 launch)
**Date (UTC):** 2026-04-27
**Engineer:** Suresh (MTS) via Claude Code

## Goal

Three additive surfaces:

1. **Pipeline health dashboard.** A new `/operations/workers` page
   for tenant Admin shows every camera worker's 4-stage pipeline
   (RTSP / Detection / Matching / Attendance) at a glance, polled
   every 5 seconds. Conditional-red logic prevents the obvious
   trap — Matching can only be red when Detection is firing,
   Attendance only when Matching is green/amber.
2. **Camera metadata.** Auto-detected RTSP properties (resolution,
   fps, codec) on first successful read; manual fields (brand,
   model, mount_location) edited by Admin. Both surface on the
   Workers page footer + the Cameras list column.
3. **Super-Admin System page.** Host CPU/memory/disk + capture
   pipeline stats + detector lock contention + tenants summary +
   scheduled jobs.

## Architecture decisions (locked)

* Polling every 5 seconds, plain HTTP. No WebSocket.
* Camera metadata mixed: auto-detect from RTSP open, manual entry
  for brand/model/location.
* Pipeline = four stages, each independently scored.
* Two pages, two scopes: tenant Admin sees `/operations/workers`;
  Super-Admin sees `/super-admin/system`.
* Restart per-camera and restart-all allowed for tenant Admin.
  Restart-all requires type-to-confirm (`RESTART ALL`).
* No metrics history persistence. Current state only — time-series
  is v1.x.

## Schema (migration `0031_p28_8_camera_metadata`)

Eight nullable columns added to `cameras`:

* `detected_resolution_w` / `detected_resolution_h` (INTEGER)
* `detected_fps` (NUMERIC(5,2))
* `detected_codec` (TEXT — "H264", "H265", "MJPG")
* `detected_at` (TIMESTAMPTZ)
* `brand`, `model`, `mount_location` (TEXT)

Backfill: existing rows get NULL. Auto-detected fields populate
naturally on next worker (re)start.

## Backend surface

* `detection/detectors.py` — replaced `_detect_lock = threading.Lock()`
  with a `TimedLock` that records held-time for the
  `contention_pct_60s()` helper. Caller code (`with _detect_lock:`)
  unchanged.
* `capture/reader.py` — extended:
  - On first successful `cap.read()`, the worker reads
    `CAP_PROP_FRAME_WIDTH/HEIGHT/FPS/FOURCC` and UPSERTs the
    `cameras` row. One-shot per worker start; restart re-reads.
  - Tracks `_last_frame_at`, `_last_analyzer_cycle_at`,
    `_last_match_at`, recent-errors deque, rolling 60s counters.
  - New `record_successful_match()` and `record_face_saved()`
    hooks for the matcher integration + emit path.
  - `get_full_stats()` returns the new 4-stage pipeline payload
    (legacy `get_stats()` kept for the WS heartbeat).
  - `_compute_stage_states()` implements the conditional-red
    logic described in the prompt (matching unknown when
    detection red, attendance unknown when matching red, etc.).
  - Worker `status` enum normalised to
    `starting | running | reconnecting | stopped | failed`.
* `capture/manager.py` — added `restart_camera`,
  `restart_all_for_tenant`, `workers_for_tenant`,
  `get_full_worker_stats`, `get_worker`,
  `get_subscriber_counts`. The reconcile loop already keeps
  the live worker set in sync with `cameras.worker_enabled`,
  so a restart only needs to stop+start for the explicit
  camera; sequential by design (keeps detector lock contention
  sane on a busy box).
* `operations/router.py` (new) — five endpoints:
  - `GET /api/operations/workers` (Admin)
  - `POST /api/operations/workers/restart-all` (Admin) —
    audit-logged
  - `POST /api/operations/workers/{id}/restart` (Admin) —
    audit-logged; cross-tenant 404
  - `GET /api/operations/workers/{id}/errors` (Admin)
  - `PATCH /api/cameras/{id}/metadata` (Admin) — manual fields
    only; `detected_*` rejected by schema
* `super_admin/system.py` (new) — two endpoints:
  - `GET /api/super-admin/system/metrics` — psutil host metrics
    + DB pool + capture stats + scheduled jobs
  - `GET /api/super-admin/system/tenants-summary` — per-tenant
    high-level health
* `cameras/schemas.py` + `cameras/repository.py` — extended
  `CameraOut` + the SELECT tuple to surface the new metadata
  fields on existing `/api/cameras` responses.
* `requirements.txt` (`pyproject.toml`) — added `psutil==6.1.0`.

## Tests

* `tests/test_operations_workers.py` (new, 9 tests):
  - Stage states are `unknown` for first 60 s of uptime
  - RTSP transitions green → amber → red as time-since-last-frame
    grows
  - Conditional red: Matching `unknown` when Detection red;
    Attendance `unknown` when Matching red/unknown
  - Matching is `red` ONLY when detection is firing but no match
    landed in the last hour
  - Workers list shape carries summary + per-worker stages
  - PATCH metadata round-trips brand/model/mount_location
  - PATCH metadata silently drops `detected_*` (not in schema)
  - Cross-tenant: unknown camera → 404 on restart + errors
  - Employee role: 403 on every operations endpoint

* `tests/test_super_admin_system.py` (new, 7 tests):
  - System metrics shape (host, data_partition, database, capture,
    scheduled_jobs)
  - psutil values within sane bounds
  - Detector lock contention is a float, 0-100 range
  - Tenants summary returns one row per active tenant
  - Tenant Admin / Employee / anonymous → 401/403

**542 backend tests passing** (526 prior + 16 new). P5 isolation
