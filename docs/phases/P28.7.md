## Goal

Five additions to the employees surface that the BRD does not explicitly
require but that Omran needs before cutover:

1. Per-employee Add / Edit drawer with extended fields (designation,
   phone, reports-to, joining + relieving dates).
2. Approval-workflow-gated Delete (HR approves Admin's delete requests;
   Admin can override another Admin's request with a mandatory comment).
3. Active / Inactive toggle that flows down to identification (inactive
   employees do NOT match for attendance, but are flagged as a security
   signal).
4. Former-employee detection — inactive employees still get face-matched,
   recorded as a security signal on `detection_events` with no
   attendance side effects.
5. Joining / relieving lifecycle automation — daily cron at 00:01
   tenant-local auto-flips `status='active' → 'inactive'` once
   `relieving_date` has passed; future `joining_date` excludes from
   attendance.

## Three locked decisions (per the prompt — do not relitigate)

1. **Admin's delete request goes to HR.** HR's own delete requests
   auto-approve on submit. Admin may override another Admin's pending
   request with a mandatory 10-char comment (NOT their own — the
   override is for cases where a second Admin needs to act).
2. **Inactive employee detection** writes
   `detection_events.former_employee_match=true` with `employee_id`
   NULL. They don't count for attendance.
3. **Joining/relieving lifecycle.** Future `joining_date` excludes from
   attendance; past `relieving_date` auto-flips to inactive via the
   00:01 tenant-local cron. Manual flip via PATCH still works.

## Schema (migration `0030_p28_7_employee_lifecycle`)

* `employees` gains seven nullable columns: `designation`, `phone`,
  `reports_to_user_id` (FK users.id ON DELETE SET NULL),
  `joining_date`, `relieving_date`, `deactivated_at`,
  `deactivation_reason`.
* `detection_events` gains two columns:
  - `former_employee_match BOOLEAN NOT NULL DEFAULT false` — set when
    matcher hits an inactive employee.
  - `former_match_employee_id` (FK employees.id ON DELETE SET NULL) —
    snapshot link for the security report. Primary `employee_id`
    stays NULL on former-employee matches so attendance queries
    (filtering `employee_id IS NOT NULL`) automatically exclude them.
    Three-column model documented in `backend/CLAUDE.md`.
* New per-tenant `delete_requests` table with the lifecycle status
  enum + a partial unique index `uq_delete_requests_pending_per_employee`
  (Postgres rejects two pending rows for the same employee even on a
  buggy direct INSERT).

The `delete_requests` table is **separate** from the existing P13
`requests` table. Different state machine, different actor rules. Both
surface in the same Approvals UI but are queried separately.

## Backend surface

* `hadir/employees/router.py` — extended PATCH/POST schemas, status
  flip rules (deactivation_reason min 5 chars on flip-to-inactive,
  cleared on flip-to-active), audit before/after for every field.
  Routes opened from Admin-only to Admin OR HR (PDPL hard-delete
  stays Admin-only).
* `hadir/employees/delete_requests.py` (new) — submit / decide /
  override / get-pending / list endpoints. Hard-delete logic:
  - Read every face-crop file path from `detection_events` for this
    employee, unlink each (defence-in-depth: refuse to delete
    anything outside `/data/faces/`).
  - Recursively remove `/data/faces/{tenant_id}/{employee_code}/`
    (P6 reference photos).
  - DELETE the employees row → cascades drop `employee_photos`,
    `custom_field_values`, `manager_assignments`,
    `attendance_records`, `requests`, `approved_leaves`, the
    `delete_requests` row itself; `detection_events.employee_id`
    + `former_match_employee_id` SET NULL (history survives
    anonymously).
  - Single `audit_log` row `employee.hard_deleted` with the
    snapshot identity + reason + actor — the verifiable record
    after the row is gone.
  - `matcher_cache.invalidate_employee(id)` AFTER commit.
* `hadir/identification/matcher.py` — extended cache to load lifecycle
  metadata (`status`, `joining_date`, `relieving_date`,
  `employee_code`, `full_name`) alongside embeddings. New
  `Match.classification` field: `active | inactive | future`.
  Tenant-local timezone is honored when comparing dates.
* `hadir/capture/events.py` — branches on classification:
  - `active` → set `employee_id`, attendance flows.
  - `inactive` → set `former_match_employee_id` + flag, INFO log,
    `employee_id` stays NULL.
  - `future` → leave both NULL, treat as Unknown.
* `hadir/employees/lifecycle_cron.py` (new) — APScheduler job, ONE
  cron trigger per tenant (each tenant has its own timezone — load
  from `tenant_settings`). Daily at 00:01 local. Idempotent.
  `scripts/run_lifecycle_cron.py` is the manual trigger for the
  validation walkthrough.
* `hadir/attendance/repository.py` — `active_employee_ids(on_date=…)`
  excludes employees whose `joining_date > on_date` or
  `relieving_date < on_date`. Wired through
  `attendance/scheduler.py`'s recompute pass.
* `hadir/reporting/former_employees.py` (new) —
  `GET /api/reports/former-employees-seen?from=&to=&format=json|xlsx`.
  Joins `detection_events` to the employees snapshot via
  `former_match_employee_id`. HR + Admin only.
* `hadir/users_lookup.py` (new) — tiny `/api/users` endpoint for
  the Edit drawer's "Reports to" picker. Filters by tenant + active
  + optional role.

## Frontend surface

* `features/employees/EmployeeDrawer.tsx` — full Add/Edit drawer.
  Sections: Identity / Assignment / Lifecycle / Reference photos /
  Status. Pending-delete yellow banner shows for any employee with
  an open `delete_request`; HR sees inline Approve/Reject; Admin
  (other than the requester) sees an "Override and approve" CTA
  with mandatory 10-char comment textarea.
* `features/employees/DeleteConfirmModal.tsx` (new) — role-aware
  copy. HR confirm → immediate delete. Admin confirm → pending
  request. Mandatory 10-char reason.
* `features/employees/EmployeesPage.tsx` — Add Employee button,
  Active/Inactive/All segmented filter chip (defaults to Active),
  Pending-deletion red badge on rows with an open request,
  greyed-out styling for inactive rows.
* `features/employees/ImportModal.tsx` — help text mentions the new
  optional columns (`designation`, `phone`, `reports_to_email`,
  `joining_date`, `relieving_date`).
* `requests/DeleteRequestsTab.tsx` (new) — Approvals page tab for
  HR + Admin. Inline approve / reject for HR; Admin sees a
  read-only "Review" CTA that opens the affected employee's drawer.
* `requests/ApprovalsPage.tsx` — wired the new tab; it shows a
  count badge.
* `features/camera-logs/CameraLogsPage.tsx` — red "Former employee"
  pill in the employee column; new "Former employees only" filter
  chip; tooltip carries the snapshot name.
* `features/reports/FormerEmployeesSeenReport.tsx` (new) — date
  range picker + table view + XLSX export. Mounted at
  `/former-employees`; sidebar nav item under Reports for Admin + HR.
* en + ar i18n bundles for every new label (Arabic Claude-generated;
  the standing P21 native-speaker review carryover applies).

## Audit log additions

| Action | When |
| --- | --- |
| `delete_request.submitted` | Admin files a pending request |
| `delete_request.hr_self_approved` | HR self-files (auto-approve path) |
| `delete_request.approved` | HR approves a pending request |
| `delete_request.rejected` | HR rejects a pending request |
| `delete_request.admin_override_approve` | Another Admin overrides + approves |
| `delete_request.admin_override_reject` | Another Admin overrides + rejects |
| `employee.hard_deleted` | Surviving record after the row is gone |
| `employee.auto_deactivated` | Lifecycle cron flipped status |
| `report.former_employees_seen` | Anyone pulled the security report |
