# P1 — Multi-tenant routing switch

**Phase:** v1.0 P1 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P1 — Multi-tenant routing switch"`

## Goal

Same codebase runs in `single` mode (pilot behaviour, schema `main`)
and `multi` mode (schema-per-tenant). In multi mode, tenant resolves
per-request from the session claim. No query may execute without a
resolved tenant — fail-closed in production, crash loud in dev.

## What was done

- `maugood/db.py`: dropped `schema=` from `MetaData()` and from every
  FK target string (`f"{SCHEMA}.tenants.id"` → `"tenants.id"`, etc.).
  Tables are now unqualified; the active schema is selected per
  connection via `SET search_path`.
- `maugood/db.py`: added `_tenant_schema_var: ContextVar[str | None]`,
  `set_tenant_schema` (validated against the same regex used by the
  DB CHECK), `tenant_context` (context manager for non-request entry
  points), and `_attach_search_path_listener(engine)` that wires a
  SQLAlchemy `checkout` event into both `make_engine()` and
  `make_admin_engine()`. Single-mode default to `main`; multi-mode
  raises `RuntimeError("no tenant schema in scope")` when no
  contextvar is set.
- Migration `0007_tenants_schema_name`: added `tenants.schema_name
  TEXT NOT NULL DEFAULT 'main'`, with a `CHECK` matching the same
  regex (`^[A-Za-z_][A-Za-z0-9_]{0,62}$`) and a `UNIQUE` constraint.
  Existing pilot tenant (id=1, Omran) gets `schema_name='main'` via
  the server default; no backfill needed.
- `maugood/tenants/scope.py`: extended `TenantScope` with
  `tenant_schema: str = "main"`. Added `resolve_tenant_schema(conn,
  tenant_id)` and a `resolve_tenant_schema_via_engine` helper for
  rare callers without a connection.
- `maugood/tenants/middleware.py` (new): `TenantScopeMiddleware` is an
  ASGI Starlette middleware that reads the session cookie, looks up
  `user_sessions.data` under `main`, and either:
    1. Honours `data.impersonated_tenant_id` if present (Super-Admin
       hook, UI in P3) — re-resolves the schema from the impersonated
       tenant.
    2. Trusts the `data.tenant_schema` claim written at login.
    3. Falls back to single-mode default or fail-closed in multi.
- `maugood/auth/sessions.py`: `create_session` now accepts
  `tenant_schema` and writes `{tenant_id, tenant_schema}` into
  `user_sessions.data`.
- `maugood/auth/router.py` login: resolves
  `user.tenant_id → tenants.schema_name` once and passes it through.
- `maugood/main.py`: registers `TenantScopeMiddleware`. The lifespan
  enrolment-backfill thread wraps its DB calls in
  `tenant_context(scope.tenant_schema)`.
- `maugood/capture/reader.py`: `_run` now wraps its inner loop in
  `tenant_context(scope.tenant_schema)` once per worker thread.
- `maugood/attendance/scheduler.py`: `recompute_today(scope)` wraps in
  `tenant_context(scope.tenant_schema)` before the inner work.

## Backward compatibility

All 92 pre-existing pytest tests still pass in single mode (the
default for `MAUGOOD_ENV=dev`). The `checkout` event defaults to
`main` when no contextvar is set in single mode, so existing fixtures
(`admin_engine.begin()`, `clean_employees`, etc.) keep working
without modification.

## New tests

`tests/test_multi_tenant_isolation.py` (+7) — the v1.0 isolation
canary:
- Provisions two real schemas with one `widgets` table each, disjoint
  rows.
- Asserts queries under one schema only return that schema's rows.
- Asserts an insert under tenant_a is invisible to tenant_b.
- Asserts the multi-mode + no-context path raises with the
  fail-closed error message.
- Asserts multi-mode + a valid `tenant_context` proceeds normally.

## Notes for whoever resumes next

- v1.0 P2 needs to:
  - Move `tenants` from `main` to `public`.
  - Add per-schema Alembic migration runner.
  - Add `scripts/provision_tenant.py` CLI to `CREATE SCHEMA`, run all
    migrations on it, seed default roles + departments + a Fixed
    policy, and create the first Admin user.
- Once `tenants` lives in `public`, the middleware's
  `tenant_context("main")` lookup of session rows must change to
  resolve sessions from `public.user_sessions` (or wherever P2 puts
  them — the session table is also a candidate for the public
  schema since session→tenant resolution must happen before tenant
  context is set).
- Existing migrations (`0001_initial` … `0006_attendance` and
  `0007_tenants_schema_name`) all bake `schema='main'` into their
  `op.create_table(..., schema=…)` calls. Going forward (per the
  v1.0 phase plan) new migrations must be schema-agnostic — P2 also
  enforces this via a pytest that scans migration files for
  forbidden patterns.
