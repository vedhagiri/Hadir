# P28.5b — Worker / display split + per-camera capture knobs

> Status: implementation complete, full backend test suite green
> (470 passed + 1 skipped), frontend typecheck clean, reconcile loop
> verified ticking on the dev stack. **Awaiting Suresh's physical
> validation sign-off** at the bottom of this file before commit.

P28.5a delivered the multi-tenant CaptureManager + reader/analyzer
worker split + boot-time auto-start. P28.5b builds on that to give
operators independent control over the two operational levers
(``worker_enabled`` for the capture pipeline + ``display_enabled``
for the Live Capture viewer) and surfaces the per-camera face-save
knobs that pre-existed as constants in the prototype.

## Architecture decisions

### 1. Split ``cameras.enabled`` into two independent flags

In production the two operational concerns are independent:

* "Should this camera be recording?" (CPU + DB load — a feed under
  privacy review may need to keep its events but stop streaming
  bytes; conversely an Admin troubleshooting an RTSP issue may want
  to stop recording without losing the camera's row).
* "Should the Live Capture viewer surface this camera?" (operator
  visibility — privacy constraints during a sensitive event, a
  quick "hide this feed" without affecting recordings).

Migration ``0027_cameras_capture_split`` renames ``cameras.enabled``
→ ``cameras.worker_enabled`` (data-preserving 1:1 backfill — every
existing row's ``enabled=true`` survives as ``worker_enabled=true``)
and adds ``cameras.display_enabled BOOL NOT NULL DEFAULT true`` so
existing rows keep showing in Live Capture without operator action.

### 2. Per-camera ``capture_config`` JSONB

The prototype's ``capture.py`` defined four module-level constants
that are operationally interesting and per-camera-specific in
production:

* ``MAX_FACES_PER_EVENT`` (10) — top-N face crops to keep per track.
* ``MAX_EVENT_DURATION_SEC`` (60) — force-retire a track after N s.
* ``min_face_quality_to_save`` (0.35, Suresh-tuned) — quality
  threshold below which a detection is skipped.
* ``save_full_frames`` (false) — debug aid; also save the full
  annotated frame alongside the face crop.

Migration 0027 adds ``cameras.capture_config JSONB NOT NULL DEFAULT``
with these four knobs at their tested defaults. The schema stays
"open by design" — adding a new knob in a future phase doesn't
require a new migration; the API + audit log carry the full JSONB
payload.

### 3. ``CaptureManager`` reconcile loop (every 2 seconds)

P28.5a wired ``on_camera_*`` hooks for synchronous CRUD reactions —
the request thread fires the hook immediately. P28.5b adds a
periodic 2-second reconcile tick on top so:

* Out-of-band camera-row mutations (psql, ad-hoc scripts, future
  replication follower writes) propagate without a CRUD call.
* Worker crashes (the worker thread exited unexpectedly) get
  noticed and restart on the next pass.
* ``capture_config`` changes hot-reload via
  ``CaptureWorker.update_config`` without a worker restart, so a
  knob tweak doesn't drop frames mid-walk-past.

The tick is driven by ``apscheduler.BackgroundScheduler`` (same
pattern as ``hadir.auth.ratelimit`` and the notification worker),
``coalesce=True + max_instances=1`` so a slow tick can't backlog.

### 4. Live-capture endpoints honor ``display_enabled``

``hadir/live_capture/router.py``'s MJPEG handler returns
**503 with detail ``camera_display_disabled``** when
``display_enabled=false``; the WebSocket handler closes with code
**1008** (policy violation). The capture worker keeps running in
the background — events still flow into ``detection_events`` /
Camera Logs / attendance — but the live feed is hidden.

The frontend treats both signals as "show empty state with
explanatory tooltip" rather than "loading".

### 5. Quality threshold without pose info

Our ``Detection`` dataclass currently doesn't carry InsightFace
landmark/pose data. The prototype's ``quality_score`` formula
weighs face area (60%) + pose symmetry (25%) + det score (15%);
without pose we use a re-balanced formula:

    0.75 × area_norm + 0.25 × det_score

where ``area_norm = min(face_w * face_h / 200², 1.0)`` so a 200px
face saturates the area term. Suresh's tested ``0.35`` threshold
behaves slightly differently than on the original (which was
pose-aware). Re-tuning on real walk-past video may be needed once
landmarks land in ``Detection`` (tracked under "future work" — not
P28.5b scope).

The ``min_face_quality_to_save`` filter applies BEFORE the JPEG
encode + DB insert — cheap to evaluate, and a low-quality
detection doesn't pay the disk + DB round-trip cost.

### 6. ``max_faces_per_event`` is stored, surfaced, and audited but
not yet enforced > 1

The current single-row-per-track architecture caps the effective
``max_faces_per_event`` at 1 — every track yields exactly one
``detection_events`` row with one ``face_crop_path``. The knob is
stored on the camera row, exposed in the API and Cameras UI,
audited on every change, and propagated to the worker via
``update_config`` — but the multi-face per event accumulation
(top-N quality-keeping with replacement) lands in a follow-up
phase. P28.5b's validation walkthrough exercises max_faces=1 and
notes that max_faces=10 (default) will surface only one crop until
that follow-up.

### 7. Audit before/after on every flip

Camera CRUD writes the full operational state in the audit row's
native ``before`` / ``after`` JSONB columns — name, location,
``rtsp_host``, ``worker_enabled``, ``display_enabled``,
``capture_config``. Reconcile-driven config updates audit as
``capture.worker.config_updated`` with the worker's pre- and
post-update config. The audit log query returns a clean diff.

## Backend changes

* ``hadir/cameras/repository.py`` — ``CameraRow`` exposes the new
  fields; ``_normalise_capture_config`` merges row JSONB onto
  defaults with type coercion.
* ``hadir/cameras/schemas.py`` — Pydantic ``CaptureConfig`` model
  with bounded fields (max_faces 1–50, duration 5–600s, quality
  0.0–1.0).
* ``hadir/cameras/router.py`` — CRUD accepts/returns the new
  fields; audit ``before`` / ``after`` carries the full state.
* ``hadir/capture/manager.py`` — reconcile scheduler at 2 s ticks,
  ``start_camera`` accepts ``capture_config``, ``stop_camera``,
  ``reconcile_all()`` returns a report dict for tests, audit
  ``capture.worker.{started_at_boot,start_failed,config_updated}``
  with proper before/after.
* ``hadir/capture/reader.py`` (CaptureWorker) — accepts
  ``capture_config``, exposes ``get_capture_config`` /
  ``update_config``; the latter propagates
  ``max_event_duration_sec`` to the tracker without restart.
* ``hadir/capture/tracker.py`` — adds
  ``max_duration_sec`` + ``update_max_duration``; force-retires
  tracks past the threshold.
* ``hadir/capture/events.py`` — ``quality_score`` standalone
  function; ``emit_detection_event`` accepts ``capture_config`` +
  ``annotated_frame_bgr``; applies ``min_face_quality_to_save``
  pre-encode and writes ``…_full.jpg`` siblings on
  ``save_full_frames=true``.
* ``hadir/live_capture/router.py`` — MJPEG returns 503
  ``camera_display_disabled`` and WebSocket closes 1008 when
  ``display_enabled=false``.
* ``hadir/system/router.py`` — system-health "cameras enabled"
  count tracks ``worker_enabled`` (the operationally-relevant
  flag); ``CameraHealthOut`` exposes both new flags.
* ``hadir/_test_endpoints/router.py`` — dev-only seed_detection
  helper updated for the new column names.

## Future work (deferred)

* Multi-face per event with the prototype's top-N quality keeping.
  Requires a new ``detection_event_faces`` child table (or an array
  column on ``detection_events``); the ``max_faces_per_event``
  knob already exists ready to drive it.
* Pose-aware ``quality_score``: add InsightFace ``kps`` extraction
  to ``Detection`` so the prototype's pose-symmetry component
  (25% weight) returns. Re-tune the 0.35 threshold afterwards.
* Bigger ``capture_config`` schema: per-camera detector
  ``det_size`` (160/224/320/480/640 from the prototype's
  ``DEFAULT_DET_SIZE`` tuning), per-camera motion-skip thresholds
  for cameras pointed at busy backgrounds (fans, monitors).
