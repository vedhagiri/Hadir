# P28.5c — System-wide detection & tracker configuration

> Status: implementation complete, full backend suite green, frontend
> typecheck clean. **Awaiting Suresh's physical-validation sign-off**
> at the bottom of this file before commit.

P28.5b delivered per-camera ``capture_config`` (face-save knobs).
P28.5c adds the system-wide pieces — the detector backend (mode +
``det_size`` + thresholds) and the tracker (IoU + idle timeout +
max event duration) — with both ports of the prototype's two
detector backends (``insightface`` and ``yolo+face``).

## Architecture decisions

### 1. Tenant-level config split from per-camera

Three knob bags in three places by intent:

| Layer | Knob bag | Lives on | Edited where |
| --- | --- | --- | --- |
| Per-camera | ``capture_config`` (P28.5b) | ``cameras.capture_config`` | Cameras → Capture settings |
| Per-tenant | ``detection_config`` (P28.5c) | ``tenant_settings.detection_config`` | System Settings → Detection |
| Per-tenant | ``tracker_config`` (P28.5c) | ``tenant_settings.tracker_config`` | System Settings → Tracker |

Detector backend / input size apply to every camera in the tenant
because the InsightFace model itself is process-wide (shared across
workers, see ``hadir/detection/detectors.py``). Tracker thresholds
and max-event-duration could be per-camera in principle, but the
prototype tested them as tenant-level defaults — we follow that
pattern.

### 2. Capture configuration precedence

``max_event_duration_sec`` is intentionally a knob in BOTH the
per-camera ``capture_config`` (P28.5b) AND the tenant-level
``tracker_config`` (P28.5c). The **per-camera value wins** when set:
a single high-traffic camera can keep tracks longer than the rest.

In code this happens at worker construction in
``hadir/capture/reader.py``:

```python
self._tracker = IoUTracker(
    iou_threshold=tracker_iou,           # from tenant tracker_config
    idle_timeout_s=tracker_timeout,      # from tenant tracker_config
    max_duration_sec=float(              # PER-CAMERA wins
        self._capture_config["max_event_duration_sec"]
    ),
)
```

The tracker's ``update_max_duration`` is wired only from the
per-camera path; the tenant-level ``tracker_config["max_duration_sec"]``
never propagates to the tracker if a per-camera value is set. The
UI's "Max event duration" hint says so explicitly: *"Per-camera
setting overrides this default."*

Documented in ``backend/CLAUDE.md`` § "Capture configuration
precedence" so future capture-pipeline work doesn't accidentally
swap the precedence.

### 3. Port the prototype's detectors verbatim

``hadir/detection/detectors.py`` is a near-direct port of
``prototype-reference/backend/detectors.py``:

* Module-level ``_detect_lock`` serialises detection across cameras
  (CPU-bound; serial is faster than parallel because parallel
  thrashes L1/L2 caches on a single CPU).
* Both modes return the same dict shape — the analyzer adapts at
  the boundary in ``hadir/capture/analyzer.py``.
* ``_load_face_app(det_size)`` re-prepares InsightFace when the size
  changes — the load-bearing hot-reload mechanic.
* ``quality_score`` keeps the prototype's tested ``0.6 / 0.25 /
  0.15`` weights (face area / pose symmetry / det score).

The prototype's ``DetectorMode`` literal type is preserved, the
``DetectorConfig`` dataclass is the same shape, and a new
``DetectorConfig.from_dict`` classmethod accepts the JSONB blob
shape for the migration's server_default.

### 4. Tracker stays v1.0; only the config-dict adapter is new

v1.0's ``IoUTracker`` already covered the prototype's behaviour
(P28.5b added ``max_duration_sec``). What was missing was a way to
build it from a JSONB dict and to hot-update it from the reconcile
loop. P28.5c adds:

* ``IoUTracker.from_tracker_config(dict)`` — classmethod that maps
  the JSONB key ``timeout_sec`` to v1.0's ``idle_timeout_s`` kwarg.
* ``IoUTracker.update_tracker_config(dict)`` — hot-update; applies
  to NEW tracks only (existing tracks keep their original
  semantics until they retire naturally — red line).

Naming divergence: the JSONB key is ``timeout_sec`` (matches the
prototype) but v1.0's tracker keeps the historical kwarg
``idle_timeout_s``. The boundary conversion lives in
``from_tracker_config``.

### 5. Hot-reload propagation

CaptureWorker reads tenant-level config at startup (passed via
``start_camera``) AND on the 2-second reconcile tick. Three
update paths:

* ``CaptureWorker.update_config(capture_config)`` — per-camera
  knobs (P28.5b).
* ``CaptureWorker.update_tracker_config(tracker_config)`` — tenant
  tracker knobs (P28.5c). New tracks only.
* ``CaptureWorker.update_detection_config(detection_config)`` —
  tenant detection knobs (P28.5c). Forwards to
  ``analyzer.update_config`` which triggers an InsightFace re-prep
  when ``det_size`` changes.

Each of these audits separately when the manager's reconcile loop
sees drift:

* ``capture.worker.config_updated``
* ``capture.worker.tracker_config_updated``
* ``capture.worker.detection_config_updated``

All carry before/after JSONB.

### 6. 400 (not 422) on validation failure

FastAPI's default for body-validation errors is **422 Unprocessable
Entity**. The P28.5c spec asked for **400 Bad Request** with the
offending field name. Implemented by accepting the request body as
``dict`` and validating manually inside the handler via
``DetectionConfigIn.model_validate``; ValidationError → re-raise as
``HTTPException(status_code=400, detail={field, message, errors})``.

The error body shape is intentionally simple so the frontend can
render it directly:

```json
{
  "detail": {
    "field": "mode",
    "message": "Value error, mode must be one of: insightface, yolo+face",
    "errors": [{"field": "mode", "message": "..."}]
  }
}
```

The full ``errors`` list is for clients that want to surface every
validation failure; the headline ``field`` + ``message`` is for the
toast.

## Backend changes

* ``hadir/detection/__init__.py`` + ``detectors.py`` — port from
  ``prototype-reference/backend/detectors.py`` with v1.0 conventions
  (logging, threading lock, ``DetectorConfig.from_dict``,
  ``set_yolo_model_dir`` for production model staging).
* ``hadir/capture/analyzer.py`` — ``InsightFaceAnalyzer`` now
  delegates to ``hadir.detection.detect`` with a per-instance
  ``DetectorConfig`` snapshot; ``update_config`` is the hot-reload
  pivot. ``embed_crop`` always uses InsightFace mode regardless of
  the configured live-capture mode (enrollment photos are
  pre-framed single-person crops; YOLO body-box would be wasted).
* ``hadir/capture/tracker.py`` — adds ``from_tracker_config`` +
  ``update_tracker_config``. Constructor default kept at
  ``idle_timeout_s=3.0`` for test compat; production always builds
  via ``from_tracker_config`` from the JSONB (default 2.0).
* ``hadir/capture/reader.py`` (CaptureWorker) — accepts
  ``tracker_config`` + ``detection_config`` kwargs; new
  ``get_tracker_config`` / ``update_tracker_config`` /
  ``get_detection_config`` / ``update_detection_config``.
* ``hadir/capture/manager.py`` —
  ``_load_tenant_capture_settings(tenant_id, schema)`` reads both
  blobs from ``tenant_settings``; the boot loop and reconcile pass
  both through. Reconcile now diffs against ``get_tracker_config``
  and ``get_detection_config`` and audits each separately.
* ``hadir/system/router.py`` — four new endpoints:
  ``GET/PUT /api/system/detection-config`` and
  ``GET/PUT /api/system/tracker-config``. Admin-only. Manual
  validation with 400 on invalid + named field. Audit row carries
  before + after JSONB.

## Performance documentation (added to backend/CLAUDE.md)

Rough CPU costs at ``det_size=320`` on a typical office laptop
(8-core x86, no GPU):

* ``insightface`` mode: **~80 ms per analyzer cycle** at 1080p.
* ``yolo+face`` mode: **~150 ms per analyzer cycle** at 1080p.

With the analyzer max-fps at 6 and motion-skip on (P28.5a), a
quiet camera burns near-zero CPU. An active office camera with
foot traffic burns ~10–20% of one core. Five cameras with mixed
activity → ~30–50% of one core, well within budget on the pilot
hardware target.

YOLO model deployment: production deploys should pre-stage
``yolov8n.pt`` (~6 MB) at ``/data/models/yolov8n.pt`` so the
ultralytics first-use download doesn't fire when an operator
flips the mode at runtime. ``hadir.detection.set_yolo_model_dir``
overrides the lookup path. Documented in
``docs/deploy-production.md``.

The prompt's checklist:

1. Restart backend.
2. Log in as ``admin@inaisys.local``. Navigate to System Settings →
   Detection. All fields render with default values.
3. Mode swap: walk past camera with default ``insightface`` →
   capture works. Switch to ``yolo+face``, save, wait 10 s, walk
   past → backend log shows YOLO body-box detection then face
   detection inside it; event captured.

   ``fps_analyzer`` drops noticeably.

   10000 pixels). Stand far → no event. Walk close → event.
6. Tracker ``iou_threshold`` test: set to 0.7, walk side-to-side
   → more events than usual (tracker can't maintain track across
   rapid motion at the strict threshold).
7. Negative test via curl: ``PUT /api/system/detection-config``
   with ``mode: yolov12`` → **400** with detail naming ``mode``.
8. Per-tenant: mts_demo settings != inaisys settings.
9. Audit log: every change appears with before/after JSONB.

## Future work (deferred)

* Pose-aware ``quality_score`` already lives in the ported
  ``hadir/detection/detectors.py`` (the kps-driven ``pose_score``
  is in the dict shape). The capture-side ``quality_score`` in
  ``hadir/capture/events.py`` (P28.5b) still uses the simplified
  area+det_score blend because v1.0's ``Detection`` dataclass
  doesn't carry kps. Plumbing kps through the dataclass is the
  next pose-aware-quality task.
* Multi-face per event (the prototype's top-N keep logic) — flagged
  in P28.5b future-work, ``max_faces_per_event > 1`` still caps at
  1 today.
* Per-camera detection_config override — currently tenant-level
  only. Edge case for "this camera in a low-light corridor needs a
  smaller min_det_score" use cases. Defer until an operator asks.
