# P28.5a — Live Capture refactor (reader/analyzer split + multi-tenant
worker keying) + brand v1.0 verification

> Status: implementation complete, **awaiting Suresh's physical
> validation sign-off** at the bottom of this file. Do not commit
> until the sign-off block is filled in.

P28.5 (committed-but-uncommitted in the working tree before P28.5a)
shipped the Live Capture viewer using a single-thread reader at 4 fps
and a process-global ``frame_buffer.py`` keyed by ``camera_id``. P28.5a
is the architectural refactor onto the prototype's proven design:
two threads per camera (reader at native fps + analyzer at ≤6 fps with
motion-skip), per-worker ``latest_jpeg`` slot, and the manager's
``_workers`` dict re-keyed by ``(tenant_id, camera_id)`` for
defence-in-depth tenant isolation.

## Architecture decisions

### 1. Adopt the prototype's reader/analyzer split

**Why**: pre-P28.5a the capture worker was one thread doing
read→detect→annotate→emit at 4 fps total. Detection takes 100-300 ms
per frame on CPU, so the preview ticked at the analyzer's pace —
laggy. The prototype's split lets the reader run at the camera's
native rate (15-30 fps) for smooth preview, while the analyzer runs
only as fast as detection allows, with motion-skip bypassing the
expensive call entirely on a still scene.

**What changed**: ``maugood/capture/reader.py`` is now a true two-thread
worker:

* **Reader thread** runs the RTSP read loop at native fps. On every
  successful read it (a) hands the frame to the analyzer via a
  shared ``_latest_frame`` reference under a lock, (b) annotates a
  COPY of the frame with the *most recent cached boxes from the
  analyzer*, and (c) JPEG-encodes that into ``self._latest_jpeg``.
  Boxes-baked-in convention is preserved — viewers stream straight
  to ``<img>`` without canvas overlay.
* **Analyzer thread** loops at ≤6 fps (``analyzer_max_fps`` config).
  Each iteration: snapshot the latest frame + seq, motion-check
  (downscaled grayscale absdiff), run ``detect`` only when motion is
  present OR the force-detect timeout elapsed, run the matcher,
  update the IoU tracker, emit one ``detection_events`` row per new
  track, and republish the cached annotation boxes for the reader.

**Motion-skip parameters** (ported verbatim from the prototype):
``MOTION_GRAY_WIDTH=160``, ``MOTION_PIXEL_THRESHOLD=25``,
``MOTION_MIN_PIXELS=80``, ``FORCE_DETECT_EVERY_S=3.0``. Quiet office
hallways → analyzer drops to ~zero CPU; live activity → full
detection runs.

### 2. Per-worker ``latest_jpeg`` slot replaces ``frame_buffer.py``

**Why**: the P28.5 ``FrameBuffer`` was process-global and keyed by
``camera_id`` only. Tenant scoping was enforced at the router layer
via ``WHERE tenant_id = …``. Moving the slot onto the worker makes
the producer/consumer locality obvious — the worker that wrote the
JPEG is also the one that gets stopped — and removes the chance of a
stale frame outliving its worker.

**What changed**:
* ``maugood/capture/frame_buffer.py`` deleted.
* ``CaptureWorker`` exposes ``get_latest_jpeg() → (bytes, ts) | None``,
  ``is_preview_fresh(max_age_s) → bool``, and ``get_stats() → dict``.
* ``CaptureWorker.stop()`` clears the slot, so a stale frame can't
  leak past worker shutdown.

### 3. ``CaptureManager._workers`` keyed by ``(tenant_id, camera_id)``

**Why**: the prompt's load-bearing red line. With per-tenant camera_id
sequences (each schema has its own ``cameras_id_seq``), two tenants
can both have a camera id 7 — which would collide in a flat
``dict[int, …]``. Keying by the tuple makes the collision impossible,
and the new ``CaptureManager.get_preview(tenant_id, camera_id)``
becomes a defence-in-depth check: even if a router bug let a
cross-tenant camera_id through, the manager refuses to serve bytes.

**What changed**:
* ``manager._workers: dict[(int, int), CaptureWorker]``.
* ``on_camera_*`` hooks accept an optional ``tenant_id`` (the cameras
  router now passes ``scope.tenant_id``); single-mode falls back to
  ``settings.default_tenant_id`` for legacy callers.
* New ``manager.get_preview(tenant_id, camera_id)`` and
  ``is_preview_fresh(tenant_id, camera_id)`` and
  ``get_worker_stats(tenant_id, camera_id)`` consumed by
  ``maugood/live_capture/router.py``.
* ``active_camera_ids(tenant_id=…)`` accepts an optional scope so
  test_capture.py's existing assertions keep passing.

### 4. Live-capture router consumes ``manager.get_preview``

The MJPEG generator and live-stats / WS heartbeat handlers all go
through the manager rather than reading the singleton frame buffer.
The 25 fps cap on the MJPEG generator (``MJPEG_FRAME_INTERVAL_S``)
matches the reader's native rate; idle bail at ~10 s
(``MJPEG_IDLE_TICK_LIMIT=250``) closes the stream when the worker
goes silent.

The WS heartbeat now carries ``status`` /``fps_reader`` /
``fps_analyzer`` / ``motion_skipped`` from the worker's live stats —
``capture_manager.get_worker_stats(scope.tenant_id, camera_id)``.
The frontend ignores these for display today (the page uses the 10 s
``/live-stats`` poll for the visible counters); they're emitted now
so the WebSocket can surface fast-changing per-second stats in a
future polish.

### 5. Frontend rename ``features/live-capture/`` → ``pages/LiveCapture/``

Pure organisational. Three files moved (``LiveCapturePage.tsx`` →
``LiveCapture.tsx``, ``hooks.ts``, ``types.ts``); App.tsx import
updated; ``../cameras/hooks`` adjusted to
``../../features/cameras/hooks``. No behavioural change. Types
extended with the new endpoint fields (``fps_reader``,
``fps_analyzer``, ``motion_skipped``).

### 6. Sidebar version chip — already correct, no change

P28.5 had already wired ``frontend/package.json`` (``version:
"1.0.0"``) → Vite ``define.__APP_VERSION__`` →
``frontend/src/config.ts`` ``APP_VERSION_SHORT`` ("v1.0") →
``shell/Sidebar.tsx`` ``brand-tag`` element. P28.5a verified the
chain renders ``v1.0`` and confirms the source of truth lives in a
single place: ``package.json``.

## Notes for future sessions

* **The motion-skip is not free of edge cases**. A camera pointed at
  a slowly-rotating fan or flickering monitor can produce subtle
  motion that the threshold misses. ``FORCE_DETECT_EVERY_S=3.0`` is
  the safety net — every 3 seconds detection runs regardless. If a
  pilot deployment surfaces "missed detections on subtle motion",
  drop ``MOTION_PIXEL_THRESHOLD`` from 25 → 15 first, then consider
  shortening the force interval. Don't disable motion-skip — that
  burns CPU on every empty frame for no payoff.

* **``analyzer_consume_every_seq`` is test-only**. Production must
  stay at ``False`` (skip-to-latest) so a slow analyzer can't backlog
  frames forever. The test knob exists for deterministic feeding of
  scripted captures; if you find yourself wanting it in production
  code, you've probably built something that should be a separate
  worker mode.

* **The tracker is unchanged**. ``maugood/capture/tracker.py``
  (``IoUTracker``) is the same pure-logic component the pilot
  P8 shipped; P28.5a only changes who calls it (the analyzer
  thread instead of the read loop) and how often (capped at 6 fps
  instead of 4).
