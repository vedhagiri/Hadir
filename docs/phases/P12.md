# P12 — Custom fields editor

**Phase:** v1.0 P12 (M2)
**Date (UTC):** 2026-04-25
**Engineer:** Suresh (MTS) via Claude Code
**Plan reference:** `v1.0-phase-plan.md §"P12 — Custom fields editor"`

## Goal

Admin defines extra columns for the employee record — text, number,
date, or single-select. The columns appear on the employee detail
drawer (editable inline) and round-trip through the Excel export +
import.

Two new tables. No new field type. No conditional logic. No
validation rules beyond `required`. Custom-field values live in their
own table — never as free-form JSON on the `employees` row (the P12
red line).

## What was done

### Migration 0015

`alembic/versions/0015_custom_fields.py`. Two new per-tenant tables:

| Table | Notes |
| --- | --- |
| `custom_fields` | id, tenant_id, name, code, type (text/number/date/select), options JSONB nullable, required, display_order, timestamps. UNIQUE (tenant_id, code). CHECK on type + CHECK that select fields carry a non-NULL options array. |
| `custom_field_values` | id, tenant_id, employee_id, field_id, value text, updated_at. UNIQUE (tenant_id, employee_id, field_id). ON DELETE CASCADE from both employees and custom_fields — deleting a field drops every employee's value for it. |

Schema-agnostic — every FK targets unqualified `tenants(id)` so the
migration runs cleanly under every tenant schema. Same pattern as
0010-0014.

### `db.py` + provisioning

Two new `Table` definitions added in `maugood/db.py`. Provisioning's
`_APP_CRUD_TABLES` extends to include both tables so a freshly-
provisioned tenant has the same CRUD grants on the per-tenant copies.
The seed step is intentionally empty — custom fields default to "no
fields defined" (operator opts in via the editor).

### Backend `maugood/custom_fields/` package

* `schemas.py` — Pydantic request/response models. Code constrained
  to `^[a-z][a-z0-9_]*$` so the spreadsheet header it has to match is
  always a clean lower-snake slug (e.g. `badge_number`). Select
  fields validate that `options` is a non-empty, deduplicated list.
* `repository.py` — DB access plus the **typed coercion** layer:
  `coerce_for_store` (rejects garbage per type — bad number → 422 in
  the API path; Excel coerce errors become row warnings) and
  `coerce_for_read` (deserialises stored text back to int/float/date
  string for API responses).
* `router.py` — endpoints below. Field-def CRUD is **Admin-only**
  (it changes the schema operators see for every employee); per-
  employee value reads/writes are **Admin + HR**.

| Method + Path | Notes |
| --- | --- |
| `GET /api/custom-fields` | List, ordered by `display_order`. |
| `POST /api/custom-fields` | Create. New field appended at `display_order = max+1`. |
| `PATCH /api/custom-fields/{id}` | Rename, toggle `required`, edit `options` (select only), set `display_order`. |
| `DELETE /api/custom-fields/{id}` | Hard delete. Cascades to values via FK. |
| `POST /api/custom-fields/reorder` | Apply `[{id, display_order}, …]` in one go. |
| `GET /api/employees/{id}/custom-fields` | One row per defined field, with the typed value (or `null`). |
| `PATCH /api/employees/{id}/custom-fields` | Upsert/clear `[{field_id, value}, …]`. `value=null` deletes the row. |

Audit hooks on every mutation:
`custom_field.{created,updated,deleted,reordered}`,
`employee.custom_fields.updated`.

### Employee Excel export + import (P5 extension)

`maugood/employees/excel.py`:

* `build_export(rows, *, custom_field_codes, values_by_employee)` —
  appends one column per defined field (ordered by `display_order`)
  using its `code` as the header. Empty cell when the employee has
  no value.
* `parse_import` now recognises **any** non-required header as a
  candidate custom-field column. The router walks each row's
  `custom_values` map: known code → coerce + upsert; unknown code or
  coerce failure → **warning**, not error. The standard columns
  still import. `ImportResult` gains a `warnings: list[ImportWarning]`
  field.

Pydantic schemas extend `ImportResult` with `warnings`. Existing
client callers that don't expect the field continue to work
(default `[]`).

### Frontend custom fields editor

`frontend/src/custom-fields/`:

* `CustomFieldsPage.tsx` — Admin-only editor at
  `/settings/custom-fields`. Inline create form, drag-handle reorder
  (native HTML5, no new deps — same approach as P8 manager
  assignments), per-row inline edit, and a delete-confirmation modal
  that explicitly warns the value cascade.
* `EmployeeCustomFieldsSection.tsx` — sub-section rendered inside
  the existing P6 employee drawer, between the standard fact-grid
  and the photo gallery. Renders a typed input per field
  (`<select>` for select with the proper options list, `<input
  type="date">` for date, `<input type="number">` for number, plain
  text otherwise) and a "Save custom fields" button when the draft
  diverges from the stored values.
* `hooks.ts` + `types.ts` — TanStack Query hooks for the four
  surfaces above.

Settings sub-nav (`frontend/src/settings/SettingsTabs.tsx`) — small
shared tab strip at the top of Branding, Authentication, and Custom
Fields so the operator can hop between siblings without going back
to the sidebar.

## Notes for whoever resumes next

- **HR write access.** HR can read + update per-employee custom-field
  values today. Field-definition CRUD stays Admin-only. If HR needs
  to create new fields, add `HR` to the `ADMIN`-guarded routes — but
  weigh it against the schema-change blast radius (every employee
  inherits the new column instantly).
- **Required fields are advisory.** The `required` flag is surfaced
  in the editor and on the drawer chip, but not enforced server-side
  on employee create. Wiring an enforcement pass is a follow-up:
  either reject `POST /api/employees` until every required field is
  present, or surface a "missing required values" badge on the
  employee row. The pilot's preference is the badge — operators
  shouldn't be blocked from creating a row.
- **Free-form JSON column on `employees` is the wrong shape.** The
  P12 red line says no, and the value table is the single source of
  truth. If a future phase wants to query employees by a custom
  field value, add an index on `custom_field_values (tenant_id,
  field_id, value)` — don't denormalise into the employee row.
- **Excel reserved-TLD gotcha.** The seeded `admin@omran.test` users
  cannot log in via the API because Pydantic `EmailStr` rejects
  `.test`. The smoke script uses the pilot's `admin@pilot.maugood`
  (re-seeded with `MAUGOOD_SEED_PASSWORD`) instead. Same workaround
  the P11 smoke used.
