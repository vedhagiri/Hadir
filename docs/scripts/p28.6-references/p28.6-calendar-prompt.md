## What this prompt builds

Three views of the same data, role-scoped:

| Role | Default view | Can switch to |
|---|---|---|
| **Admin** | Company-wide percentages | Per person |
| **HR** | Company-wide percentages | Per person |
| **Manager** | Per person, defaulted to one of their team | Team-only company view (just their dept) |
| **Employee** | Their own month, no view switcher | (locked to self) |

Plus a day-detail drawer that opens when any day cell is clicked.

> ### Architecture decisions (read carefully)
>
> 1. **The data already exists.** `attendance_records` rows are produced by the engine (P10). One row per (employee, date). No new tables needed.
>
> 2. **Aggregations are computed on read, not stored.** The Company view's "% present per day" is a SQL `GROUP BY date` over `attendance_records` plus a count of active employees that day. Don't introduce a `daily_aggregate` table — query latency at 100 employees × 30 days is well under 100ms.
>
> 3. **Role scoping uses the existing `current_user` + tenant + department machinery.** No new RBAC code. Manager queries filter by department + manager_assignments; Employee queries filter by self-id. This is the same pattern the existing Daily Attendance page uses.
>
> 4. **The day detail drawer is a port of the existing Daily Attendance detail drawer**, with the addition of the EVIDENCE section showing face crop thumbnails. Reuse, don't reimplement.
>
> 5. **Cell colour comes from a single status field.** The engine already produces `late`, `early_out`, `absent`, `short_hours`, `overtime_minutes` flags on each `attendance_records` row. Plus `approved_leaves` for that date overrides absent. Plus `holidays` + tenant `weekend_days` setting. Compute cell status server-side in the query, not in JS — keeps the frontend dumb and fast. Status enum: `present | late | absent | leave | weekend | holiday | future | no_record`.
>

> ### Backend endpoints
>
> All in `backend/maugood/attendance_calendar/router.py`. Tenant-scoped. Role-scoped per the matrix in this file's preamble.
>
> 1. **`GET /api/attendance/calendar/company?month=YYYY-MM`** — Admin / HR. For each day in the month, returns:
>    ```json
>    {
>      "date": "2026-04-13",
>      "present_count": 93,
>      "late_count": 10,
>      "absent_count": 3,
>      "leave_count": 0,
>      "active_employees": 106,
>      "is_weekend": false,
>      "is_holiday": false,
>      "holiday_name": null,
>      "percent_present": 97
>    }
>    ```
>
>    Manager calls this and gets the SAME shape but scoped to their department(s) — counts are dept-only, `active_employees` is dept-only headcount.
>
>    Implementation: one SQL query joining `attendance_records`, `employees`, `holidays`, `tenant_settings.weekend_days`, with `GROUP BY date`. Apply role scope as a WHERE clause on `employees.department_id IN (manager's depts)`.
>
> 2. **`GET /api/attendance/calendar/person/{employee_id}?month=YYYY-MM`** — Returns 30/31 daily records for one employee:
>    ```json
>    {
>      "date": "2026-04-13",
>      "status": "present",
>      "in_time": "07:28:00",
>      "out_time": "16:19:00",
>      "total_minutes": 531,
>      "overtime_minutes": 48,
>      "policy_name": "Flex 07:30–16:30",
>      "is_weekend": false,
>      "is_holiday": false,
>      "holiday_name": null,
>      "leave_name": null
>    }
>    ```
>
>    Role check: Admin/HR can request any `employee_id`. Manager can only request employees in their departments OR directly assigned. Employee can only request their own record (404 otherwise — 403 leaks role information).
>
> 3. **`GET /api/attendance/calendar/day/{employee_id}/{date}`** — full day detail for the drawer:
>    ```json
>    {
>      "employee": {"id": ..., "code": "OM0045", "full_name": "Fatima Al-Kindi", "title": "Operations Analyst", "avatar_url": ...},
>      "date": "2026-04-05",
>      "status": "present",
>      "in_time": "07:28:42",
>      "out_time": "15:34:12",
>      "total_minutes": 485,
>      "overtime_minutes": 6,
>      "policy": {"name": "Flex 07:30–16:30", "scope": "dept-level", "description": "Must complete 8 hours. Window opens 07:30, closes 16:30."},
>      "timeline": [
>        {"start": "07:28", "end": "12:30"},
>        {"start": "13:15", "end": "15:34"}
>      ],
>      "evidence": [
>        {"detection_event_id": 152, "captured_at": "07:28:42", "camera_code": "CAM-01", "confidence": 0.97, "crop_url": "/api/detection-events/152/crop"},
>        ...
>      ]
>    }
>    ```
>
>    The `timeline` field is a list of contiguous "in-the-office" intervals derived from detection events. Compute server-side: take all detection events for that employee on that date, sort by time, collapse runs separated by less than ~10 minutes into single intervals. The drawer renders this as the horizontal bar chart visible in screenshot 03.
>
>    The `evidence` array is up to 5 face crops sorted by confidence × spread-across-the-day (so we don't return 5 crops all from 07:28). Picking algorithm: take the highest-confidence crop in each of these time windows: arrival (in_time ± 30min), morning (in_time+30min to noon), midday (11:00–14:00), afternoon (14:00 to out_time-30min), departure (out_time ± 30min). Empty windows skipped.
>
> 4. **`GET /api/attendance/calendar/export?employee_id=...&month=YYYY-MM`** — Excel export of the per-person view. For company view, use the existing scheduled report Excel from P18.
>

## What this prompt does NOT do

- **Does not change attendance computation logic.** The engine still produces the same flags. This is presentation only.
- **Does not introduce real-time updates.** Page fetches on load and on month change. Live updating during a demo isn't worth the complexity.
- **Does not add a year view or week view.** Only month view. If anyone asks for those, file as v1.x.
- **Does not redesign the existing Daily Attendance page.** That stays as-is. Calendar is a different lens on the same data.

## Why this is worth doing before P29

When you walk Omran through Maugood:
- "Here's our attendance system" → currently they see Daily Attendance (a table)
- vs "Here's our attendance system" → they see a month grid where every day is a colored cell, click any day for face evidence

Same data. Dramatically different perception. The Calendar page is the screenshot Omran will remember and quote when describing Maugood to other potential customers in Oman.

It's also the screen that proves Maugood works as an integrated whole — capture pipeline → identification → attendance engine → policy application → leave handling → calendar visualization, all in one click. Live Capture proves the technology. Calendar proves the product.
