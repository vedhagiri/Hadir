# P28.7 — Employee Management: Add / Edit / Delete + Lifecycle

**Status:** Pre-Omran validation. Adds significant business-logic depth beyond what the BRD originally scoped.

**BRD note (read first):** BRD FR-EMP-001 through FR-EMP-009 covered employee management at a basic level — Excel import/export, photo ingestion, soft delete by Admin. **This prompt adds five capabilities the BRD does not explicitly require:**
1. Per-employee Add/Edit drawer with extended fields (designation, phone, reports-to, joining/relieving dates)
2. Approval-workflow-gated Delete (HR approves Admin's delete requests; Admin can override with mandatory comment)
3. Active/Inactive toggle that flows down to identification (inactive employees do not match for attendance)
4. Former-employee detection (inactive employees still get face-matched, but tagged as security signal — not attendance)
5. Joining/relieving lifecycle automation (cron auto-flips on relieving_date)

These are real product additions, not bug fixes. Treat as additive scope like P28.5 and P28.6 were.

**Why we're building it now:** The Employees page exists with Add/Import buttons but the Add drawer is incomplete and Delete doesn't exist. Without Edit/Delete and lifecycle handling, Omran's HR has no way to manage employee records over time. Plus the inactive-employee-still-being-tracked case is a real PDPL concern — Maugood would record attendance for someone who left three weeks ago if Admin forgot to delete them. The cron auto-flip closes that hole.

## Decisions (locked in by Suresh — do not relitigate)

These three decisions are the foundation. Don't second-guess them in the implementation:

1. **Delete approval:** Admin's delete requests go to HR for approval (via existing approval workflow from P13-P16). Admin may override with mandatory comment, audit-logged. HR's own delete requests go through the workflow but auto-approve themselves on submit (HR is the final approver, no further escalation needed).

2. **Inactive employee detection:** When an inactive employee walks past camera, the matcher still identifies them — but the match is recorded as `former_employee_match=true` on `detection_events` instead of populating `employee_id`. This:
   - Does NOT count toward attendance (employee is inactive)
   - Does NOT show in Daily Attendance, Calendar, or Reports as "present"
   - DOES appear in Camera Logs with a special "Former employee" badge
   - DOES enable a "Former employees seen on premises" report for HR/security

3. **Joining/relieving lifecycle:**
   - `joining_date` future → employee record exists but is excluded from attendance computation (no "absent" rows generated for days before they joined)
   - `relieving_date` past → daily cron at 00:01 tenant-time auto-flips `status` from `active` to `inactive`. Manual flip also works any time.
   - Both fields are nullable. Most existing employees won't have them set.

## Schema migration

Per-tenant Alembic migration:

### Extend `employees` table

Add columns:
- `designation TEXT` — job title (e.g. "HR Director", "Site Engineer"). Display field, not used for logic.
- `phone TEXT` — display field. Format `+968 XXXX XXXX` for Oman; no validation beyond non-empty.
- `reports_to_user_id INTEGER` REFERENCES `users(id)` ON DELETE SET NULL — manager for HR purposes (separate from `manager_assignments` which drives approval scope).
- `joining_date DATE`
- `relieving_date DATE`
- `deactivated_at TIMESTAMPTZ` — set when status flipped to inactive
- `deactivation_reason TEXT` — free text, e.g. "Resigned 2026-04-15", "Termination — see HR file"

`status` already exists from pilot P5 (`active` / `inactive`). Don't rename. Don't add new statuses.

### Extend `detection_events` table

Add column:
- `former_employee_match BOOLEAN NOT NULL DEFAULT false`

Set when matcher finds an inactive employee. The `employee_id` column stays NULL in this case (so attendance queries that filter on `employee_id IS NOT NULL` automatically exclude former employees, no other code changes needed).

### New `delete_requests` table (per-tenant)

```
id              INTEGER PK
tenant_id       INTEGER NOT NULL
employee_id     INTEGER NOT NULL FK employees(id) ON DELETE CASCADE
requested_by    INTEGER NOT NULL FK users(id)
reason          TEXT NOT NULL                    -- mandatory, min 10 chars
status          TEXT NOT NULL                    -- 'pending' | 'approved' | 'rejected' | 'admin_override' | 'cancelled'
hr_decided_by   INTEGER FK users(id)
hr_decided_at   TIMESTAMPTZ
hr_comment      TEXT
admin_override_by INTEGER FK users(id)
admin_override_at TIMESTAMPTZ
admin_override_comment TEXT                       -- mandatory if admin_override
created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
```

Note: this is a separate table from the existing `requests` table (P13). The existing `requests` table is for attendance/leave exception requests; mixing employee-deletion requests in there muddies the schema. Keep them separate but show them in the same Approvals UI.

### Backfill rules

- `joining_date` for existing employees: leave NULL (we don't know when they joined; manual data entry by HR over time)
- `designation`: leave NULL
- `phone`: leave NULL
- `reports_to_user_id`: leave NULL
- All existing detection_events: `former_employee_match=false` (the column default)

## What this prompt does NOT do

## Why this is worth doing before P29

Three reasons:

1. **Without lifecycle automation, Maugood leaks data.** If HR enters a relieving date in their main system but forgets Maugood, you'd be tracking attendance for a non-employee for weeks. The cron closes that hole.

2. **Without former-employee detection, you lose a real security signal.** A terminated employee returning to premises is a category of incident; Maugood is in the right position to catch it.

3. **Without HR-approved delete, Admin has too much unilateral power.** PDPL audits ask "who can delete data, what's the process?" — having HR-in-the-loop with Admin override + mandatory comment is a defensible answer. "Admin clicks delete, gone forever" is not.

This is the prompt that makes Maugood feel like a real HR system instead of a clever face-recognition demo.
