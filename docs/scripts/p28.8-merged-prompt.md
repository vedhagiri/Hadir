# P28.8 — Worker monitoring + Camera metadata + Pipeline health

**Status:** Pre-Omran validation. Operational visibility addition. Not in BRD.

**Why we're building it:** Hadir's capture pipeline runs background workers per camera with multiple stages (RTSP read → detection → matching → attendance). Today, an Admin has zero visibility into whether all stages are healthy — they have to SSH in and tail logs. Failures cascade silently: RTSP works but matching is broken, so events get recorded as "Unknown" and HR notices wrong attendance a week later. This prompt closes that gap with a polling-based dashboard plus per-camera technical metadata.

This is the operational closing-of-loops phase. After this, Hadir is a defensible operations product — one that exposes its own health rather than hiding behind logs.

---

## What this prompt builds

Three connected things, one phase:

### 1. Camera technical metadata (auto + manual)

When a worker opens its RTSP stream, it auto-detects: codec, resolution, native fps, color format. These persist on the camera record. Admin can manually enter: brand, model, location notes, mount height. All of this displayed on the Cameras page edit drawer and on the Workers page rows.

### 2. Worker monitoring page (Admin, tenant-scoped)

Per-camera worker view with **per-stage pipeline indicators** (RTSP / Detection / Matching / Attendance), each with its own state and timing. Plus runtime stats (fps, errors, faces saved) and restart actions.

### 3. System metrics page (Super-Admin, host-wide)

Host CPU/memory/disk, capture aggregates, detector lock contention, scheduled jobs status. Cross-tenant summary without per-tenant detail.

---

## Decisions locked in

These were chosen by Suresh; do not relitigate:

- **Update mechanism:** plain HTTP polling every 5 seconds. No WebSocket, no SSE.
- **Camera metadata:** auto-detect from RTSP stream what's available (codec/resolution/fps); manual entry for brand/model/location notes. No ONVIF discovery.
- **Pipeline health detail:** breakdown of four stages (RTSP / Detection / Matching / Attendance), each with its own indicator. No collapsed traffic-light view.
- **Scope split:**
  - Admin sees `/operations/workers` for their tenant only. Restart actions allowed.
  - Super-Admin sees `/super-admin/system` for host-wide metrics. No per-tenant worker detail.
  - HR / Manager / Employee: no access. 403 on direct URL.
- **Restart actions:** per-camera and "restart all" both available. "Restart all" requires type-to-confirm because of the ~10-second capture pause across all cameras.

---

## The four pipeline stages — definitions

Each stage has a state (`green | amber | red | unknown`) and a "last activity" timestamp. The thresholds are deliberately tiered because each stage has different normal cadence.

| Stage | Green | Amber | Red |
|---|---|---|---|
| **RTSP** | Last frame < 5s ago | Last frame 5–30s ago (transient hiccup) | Last frame > 30s ago, or RTSP closed |
| **Detection** | Last analyzer cycle < 30s ago | Last cycle 30–120s ago (slow but alive) | Last cycle > 120s ago, or analyzer thread dead |
| **Matching** | Last successful match < 10min ago | 10min–1hr ago (low traffic — possibly normal) | > 1hr ago AND detection has been firing (proves matcher is broken, not just empty hallway) |
| **Attendance** | Attendance row created from this camera within last 1hr | 1–4 hours (could be lunch / weekend) | > 4 hours during business hours AND matching has been firing |

The **conditional reds** (Matching red requires Detection firing; Attendance red requires Matching firing) are important. Without those, every quiet-hallway camera would show false-red alarms overnight. The conditions encode "the upstream stage is working but the downstream isn't, which means *this stage* is the problem."

`unknown` (grey) shown when: worker just started (under 60s old) and we don't have enough data to judge yet.

> ### Part 2 — Backend: camera metadata auto-detection
>
> 1. **In `CameraWorker._open_rtsp()`** (or wherever the RTSP open lives — should match the prototype's pattern), after a successful `cap.read()` returns its first frame, read these via OpenCV:
>    ```python
>    width = int(cap.get(cv2.CAP_PROP_FRAME_WIDTH))
>    height = int(cap.get(cv2.CAP_PROP_FRAME_HEIGHT))
>    fps = float(cap.get(cv2.CAP_PROP_FPS))
>    fourcc_int = int(cap.get(cv2.CAP_PROP_FOURCC))
>    fourcc = "".join(chr((fourcc_int >> (8 * i)) & 0xFF) for i in range(4)).strip().upper()
>    # fourcc is like "H264", "HEVC" (= H265), "MJPG"
>    ```
>
>    Then UPSERT to the cameras row:
>    ```sql
>    UPDATE cameras
>    SET detected_resolution_w = :w,
>        detected_resolution_h = :h,
>        detected_fps = :fps,
>        detected_codec = :codec,
>        detected_at = now()
>    WHERE id = :camera_id;
>    ```
>
>    Wrap in try/except — if any property is unavailable (some cameras don't expose CAP_PROP_FPS), set those fields to NULL but keep the rest. Don't fail the worker startup over a metadata read.

> 4. **For cameras whose worker is disabled** (`worker_enabled=false` from P28.5b), the detected_* fields stay as whatever was last detected. That's fine — the UI shows "(last detected ...)" with the timestamp.

> 2. **Stage state computation logic** (lives in worker.py, called by get_stats):
>    ```python
>    def _compute_stage_states(self) -> dict:
>        now = time.time()
>        uptime = now - self.started_at
>
>        # Don't trust judgments in first 60s — too little data
>        if uptime < 60:
>            return {stage: {"state": "unknown", ...} for stage in ["rtsp", "detection", "matching", "attendance"]}
>
>        # RTSP
>        rtsp_age = now - self._last_frame_at if self._last_frame_at else 999
>        rtsp_state = "green" if rtsp_age < 5 else ("amber" if rtsp_age < 30 else "red")
>
>        # Detection
>        det_age = now - self._last_analyzer_cycle_at if self._last_analyzer_cycle_at else 999
>        det_state = "green" if det_age < 30 else ("amber" if det_age < 120 else "red")
>
>        # Matching — conditional red
>        match_age = now - self._last_match_at if self._last_match_at else 999
>        if det_state == "red":
>            match_state = "unknown"  # can't judge matching if detection isn't running
>        elif match_age < 600:
>            match_state = "green"
>        elif match_age < 3600:
>            match_state = "amber"
>        elif self.frames_analyzed_60s > 0:  # detection IS firing — so matcher really is broken
>            match_state = "red"
>        else:
>            match_state = "amber"  # detection idle, can't really judge
>
>        # Attendance — also conditional, query the DB
>        att_age = self._compute_last_attendance_age()  # see below
>        if match_state in ("red", "unknown"):
>            att_state = "unknown"
>        elif att_age < 3600:
>            att_state = "green"
>        elif att_age < 14400:  # 4 hours
>            att_state = "amber"
>        elif self._matches_60s_recent > 0:
>            att_state = "red"
>        else:
>            att_state = "amber"
>
>        return {
>            "rtsp": {"state": rtsp_state, "last_activity_at": ..., "detail": _detail_for_rtsp(...)},
>            ...
>        }
>    ```
>
>    `detail` field is a one-line human-readable string: e.g. "Frames flowing at 15.2 fps" (green), "Last frame 47 seconds ago" (amber), "RTSP disconnected — 3 reconnect attempts" (red).
>
> 3. **`_compute_last_attendance_age()`** — query `attendance_records` for any row whose underlying detection_event came from this camera and was created in the last 4 hours. Cache the result for 30 seconds (don't re-query every get_stats call). Implementation:
>    ```sql
>    SELECT MAX(ar.created_at)
>    FROM attendance_records ar
>    JOIN detection_events de ON de.employee_id = ar.employee_id
>      AND de.captured_at::date = ar.attendance_date
>    WHERE de.camera_id = :camera_id
>      AND ar.created_at > now() - interval '4 hours'
>    ```
>
> 4. **Track `_last_match_at` and `_matches_60s`** — in the matcher integration point (where `detection_events.employee_id` gets populated), call `worker.record_successful_match()`. This sets `_last_match_at = time.time()` and increments a rolling counter.
>
> 5. **Recent errors deque** — `collections.deque(maxlen=5)`. Append a string per error event (RTSP reconnect, decode failure, matcher exception). String format: `"<timestamp>: <category>: <message>"`.

> ### Part 4 — Backend: detector lock instrumentation
>
> Replace the global `_detect_lock` in `backend/hadir/detection/detectors.py` with a `TimedLock` that records held-time:
>
> ```python
> import collections
> import threading
> import time
>
> class TimedLock:
>     """threading.Lock with rolling held-time stats for contention reporting."""
>     def __init__(self):
>         self._lock = threading.Lock()
>         self._held_times: collections.deque = collections.deque(maxlen=600)
>
>     def __enter__(self):
>         self._lock.acquire()
>         self._t_acquired = time.time()
>         return self
>
>     def __exit__(self, *exc):
>         held = time.time() - self._t_acquired
>         self._held_times.append((self._t_acquired, held))
>         self._lock.release()
>
>     def contention_pct_60s(self) -> float:
>         """Percentage of last 60s the lock was held."""
>         cutoff = time.time() - 60
>         relevant = [(t, h) for (t, h) in self._held_times if t >= cutoff]
>         if not relevant:
>             return 0.0
>         total_held = sum(h for _, h in relevant)
>         return min(100.0, total_held / 60.0 * 100)
>
> _detect_lock = TimedLock()
> ```
>
> Caller code (`with _detect_lock:`) doesn't change. Document in `backend/CLAUDE.md` under "Capture instrumentation".

## What this prompt does NOT do

- **No metrics history persistence.** Page shows current state only. Time-series for trend analysis is v1.x.
- **No alerting / paging.** No email or push when a stage goes red. v1.x feature.
- **No ONVIF discovery.** Manual brand/model entry. Auto-discovery is a separate scope item.
- **No multi-process worker model.** All workers run as threads in one Python process. Revisit at 40+ cameras.
- **No restart-with-config-reload.** Capture config changes auto-apply via reconcile loop (P28.5b). Restart is for "stuck worker, clean reset" not config picks-up.

## Why this is the right closing-of-loops phase

When Omran goes live, the failure mode is predictable: a network blip causes one camera to drop, a credentials rotation invalidates RTSP auth, an enrolled employee gets a hat that obscures their face — pipeline silently degrades for that camera. Without this page, recovery requires SSH + log grep + restart, hours later.

With this page:
- Admin sees Office Camera 4 has Matching red. Click → "Detector firing but matcher silent". Realizes face enrollment for the new hire isn't done. Adds photos. Walks past, Matching turns green.
- Total time to diagnose: 30 seconds. Total time to fix: 5 minutes.

That's the difference between "Hadir is buggy" and "Hadir tells me exactly what's wrong" in the eyes of Omran's HR team. Worth the 4-5 hours.
