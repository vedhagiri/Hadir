# P28.5 — Live Capture page (revised: MJPEG + background-capture tap)

**Context:** During pre-Omran validation after P28, the `/live` route renders a v0.1 placeholder reading "Deferred to v1.0". The design archive shows a real-time face detection viewer. This is genuine new scope — not in the BRD as a Must — but it makes the demo dramatically more compelling.

**This prompt supersedes the earlier P28.5 draft.** The architecture changed after reviewing Suresh's Round 2 prototype:
- **Out:** WebSocket + base64-encoded JPEG frames + canvas redraw
- **In:** MJPEG multipart stream (the prototype's pattern) + a frame buffer tap into the existing P8 capture pipeline

The MJPEG pattern is simpler, lower CPU, renders in a plain `<img src=...>` tag, and is already battle-tested in the prototype. We use WebSocket only for the lightweight detection-event JSON stream, not for video.

**Estimated time:** 3.5–4.5 hours (down from the earlier 4–5h estimate — MJPEG saves frontend complexity).

---

## Architectural model

Three concurrent things run in v1.0, with clear ownership:

1. **Always-on background capture (P8 — already built).**
   One OpenCV reader per enabled camera, runs as long as the backend runs, writes detection events to the DB. **Independent of any viewer.** If 20 cameras are enabled, 20 readers run. If zero browsers are open, they keep running. **This prompt does not touch capture worker logic** — only adds a tap.

2. **Per-camera frame buffer (new in this prompt).**
   Each capture worker writes its **latest annotated frame** (with bounding boxes drawn on it) into a small in-memory buffer keyed by camera_id. Just one frame, not a queue — overwrite each iteration. Memory cost: 20 cameras × ~80KB per JPEG ≈ 1.6 MB worst case. Trivial.

3. **Live viewer endpoints (new in this prompt).**
   - `GET /api/cameras/{id}/live.mjpg` — MJPEG multipart stream, reads from the buffer at ~10 fps, serves it. One concurrent viewer per camera per tenant; second viewer of the same camera shares the same buffer (no extra capture work).
   - `GET /api/cameras/{id}/events.ws` (WebSocket) — pushes detection events as they happen, plus heartbeat every 5s. JSON only, no frames.
   - `GET /api/cameras/{id}/live-stats` — rolling counters (last 10 min: known/unknown/total).

The frontend renders the MJPEG in `<img src="/api/cameras/{id}/live.mjpg">`. Bounding boxes are baked into the frame by the backend (same draw logic the prototype uses). The frontend doesn't need to draw boxes itself — it just shows the image and listens to the events WebSocket for the table.

This is intentionally minimal. The earlier "frontend draws SVG boxes from coordinate streams" approach was clever but unnecessary when the backend can render once and many viewers consume.

> ### Part B — Live Capture page (~3.5–4 hours)
>
> #### Architecture
>
> **Do not start new capture workers.** P8 already runs one capture worker per enabled camera. Live Capture is a viewer on top of that pipeline, not a parallel one. The plumbing has three pieces:
>
> 1. **Frame buffer** — in `backend/maugood/capture/frame_buffer.py`, build a thread-safe per-camera latest-frame buffer:
>
>    ```python
>    class FrameBuffer:
>        """Holds the latest annotated frame per camera_id, in memory.
>
>        - One slot per camera. Writes overwrite (never queue).
>        - Reads return a copy of the bytes plus a timestamp.
>        - 20 cameras × ~80KB JPEG ≈ 1.6 MB max. No persistence.
>        - Bounding boxes are drawn into the frame by the capture
>          worker BEFORE writing to this buffer, so viewers can stream
>          straight to <img> without client-side overlay logic.
>        """
>    ```
>
>    Public API: `set(camera_id: int, jpeg_bytes: bytes, ts: float) -> None`, `get(camera_id: int) -> tuple[bytes, float] | None`, `is_fresh(camera_id: int, max_age_s: float = 5.0) -> bool`. Use a `threading.Lock` keyed per camera (a `dict[int, Lock]` initialised lazily).
>
> 2. **Capture worker tap** — modify the existing capture worker from P8 (likely `backend/maugood/capture/reader.py` or `worker.py`) to, after each successful detect+identify cycle: (a) draw the bounding boxes and labels onto the current frame, (b) JPEG-encode at quality 70, (c) write to the FrameBuffer. **Use the SAME drawing primitives the Round 2 prototype uses** — keep visuals consistent. If those primitives don't already exist in v1.0, port them from the prototype's `capture.py` annotation step verbatim. Box colours from design tokens: success-green for identified-above-threshold, warning-amber for unknown/below-threshold.
>
> 3. **Tenant scoping** — the FrameBuffer is process-global, but every read goes through a tenant check first. The handler resolves `camera_id`, confirms it belongs to the requesting session's tenant, then reads. No way for tenant_a to view tenant_b's camera by guessing IDs.
>
> #### Backend endpoints (~90 min)
>
> All endpoints under `backend/maugood/live_capture/router.py`. All require Admin role (HR/Manager scoping is a follow-up).
>
> 1. **`GET /api/cameras/{id}/live.mjpg`** — MJPEG multipart stream of the latest frame from FrameBuffer at ~10 fps.
>
>    Implementation pattern (verbatim from the Round 2 prototype, adapted for tenant scoping):
>
>    ```python
>    async def live_mjpg(camera_id: int, ...):
>        # tenant + role check first
>        # camera lookup, 404 if not in tenant
>        # audit-log subscription start
>
>        boundary = b"--frame"
>        idle_ticks = 0
>
>        async def gen():
>            nonlocal idle_ticks
>            while True:
>                got = frame_buffer.get(camera_id)
>                if got is None or not frame_buffer.is_fresh(camera_id):
>                    idle_ticks += 1
>                    if idle_ticks > 100:  # ~10 seconds no fresh frame
>                        break
>                    await asyncio.sleep(0.1)
>                    continue
>                idle_ticks = 0
>                jpg, _ts = got
>                yield boundary + b"\r\n"
>                yield b"Content-Type: image/jpeg\r\n"
>                yield f"Content-Length: {len(jpg)}\r\n\r\n".encode()
>                yield jpg + b"\r\n"
>                await asyncio.sleep(0.1)  # 10 fps cap
>
>        return StreamingResponse(
>            gen(),
>            media_type="multipart/x-mixed-replace; boundary=frame",
>        )
>    ```
>
>    Two important deltas from the prototype: (a) tenant check before yielding any bytes, (b) audit log on stream open and on stream close (write the close audit in a `finally` block of the generator).
>
> 2. **`GET /api/cameras/{id}/events.ws`** — WebSocket. Authenticates via session cookie. Sends:
>    - `{type: "detection", time, employee_id, employee_name, employee_code, confidence, status}` per new event for this camera
>    - `{type: "heartbeat", server_time, camera_status, fps}` every 5 seconds
>    - `{type: "stats", detections_last_10m, known_count, unknown_count}` every 30 seconds
>
>    No frames in this stream. Pure JSON. Subscribes to a per-tenant pub/sub built on top of the existing capture event flow — if no event bus exists yet, build a minimal one in `backend/maugood/capture/event_bus.py` (in-process, asyncio queues keyed by camera_id). Document in `backend/CLAUDE.md`.
>
> 3. **`GET /api/cameras/{id}/events.csv?hours=1`** — Streams CSV of last hour's detection events for the export button. Same query path as Camera Logs export from P11; just a thin wrapper.
>
> 4. **`GET /api/cameras/{id}/live-stats`** — Returns `{detections_last_10m, known_count, unknown_count, fps, status}` from a SQL query against `detection_events` plus the latest `camera_health_snapshots` row. Polled every 10s by frontend (separate from the WebSocket — keeps stats decoupled from event timing).
>
> #### Concurrency & throttle
>
> - Multiple browsers viewing the **same** camera share the FrameBuffer — no extra capture cost. Allow up to 10 concurrent MJPEG viewers per camera per tenant. Beyond that, return 503. (Higher than my earlier "5" estimate because there's no per-viewer capture cost — only network.)
> - WebSocket: max 10 concurrent event subscribers per camera per tenant, same rationale.
> - If a camera's worker isn't running (because the camera is disabled, or its worker crashed), the FrameBuffer goes stale → MJPEG endpoint sees no fresh frames → returns 503 with `{detail: "camera_offline"}` after a 2-second wait. Frontend handles this by showing the empty/error state.

## Why this differs from the earlier P28.5 prompt

| Earlier prompt | This prompt | Reason |
|---|---|---|
| WebSocket + base64 frames | MJPEG multipart | Simpler, lower CPU, plain `<img>` rendering, prototype-proven |
| Frontend draws SVG boxes | Backend bakes boxes into frame | Many viewers, one render — better economics |
| Spawns capture per subscriber | Reads from existing P8 pipeline buffer | Doesn't double CPU when viewing |
| 5 subscribers per camera | 10 subscribers per camera | Higher because no per-viewer capture cost |
| New event bus invented from scratch | Asyncio pub/sub on top of P8 events | Smaller surface, less to break |
