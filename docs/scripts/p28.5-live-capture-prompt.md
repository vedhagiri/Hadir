> ### Functional spec
>
> The page has three regions:
>
> 1. **Header strip:** Title "Live capture", subtitle showing analyzer status (e.g. "Real-time face detection · motion-skip optimised · analyzer 4.2 fps"). Right-aligned: a Pause/Resume button and a Reconnect button. Pause stops rendering frames but keeps the websocket open. Reconnect tears down and re-opens the streaming connection.
>
> 2. **Main viewer (left, ~2/3 width):** Live video preview from the selected camera with face bounding boxes overlaid. Each box shows the identified employee name and confidence as a label, e.g. `Fatima Al-Kindi · 97%`. Identified faces use the success-green box color from the design tokens; unknown faces use the warning-amber color with label "Unknown · 48%". Below the viewer: a status strip showing camera online state, the (masked) RTSP path as `rtsp://…/cam-01`, and rolling counters: "Detections last 10m: 47 · Known 44 · Unknown 3".
>
> 3. **Camera list (right, ~1/3 width):** Selectable list of all enabled cameras with name, code (CAM-01), current fps, and status badge (Online / Degraded / Offline). Selecting a camera switches the main viewer. There is a "+" button at the top of the list that links to the existing Cameras CRUD page (don't reimplement camera creation here).
>
> 4. **Event stream (full-width below):** A live-updating table of recent detection events for the selected camera. Columns: Time, Camera, Identified (avatar + name + employee code, or "Unknown face"), Confidence (%), Evidence (a small horizontal bar visualising confidence), Status (Identified / Unknown badges). A toggle at the top right reads "Only unknown" — when on, filters to unidentified events. An "Export last hour" button downloads a CSV of the last hour's events.
>
> ### Backend work (~90 minutes)
>
> 1. **Streaming endpoint.** Add `GET /api/cameras/{camera_id}/stream` (WebSocket) that the frontend connects to. The endpoint:
>    - Authenticates via the existing session cookie (Admin role required — same auth pattern as the existing preview endpoint from pilot P7)
>    - Tenant-scoped: verifies the camera_id belongs to the current tenant, 404 otherwise
>    - Subscribes to detection events for that camera
>    - Pushes each new detection event as a JSON message: `{type: "detection", time, bbox, employee_id, employee_name, employee_code, confidence, status}`
>    - Pushes a periodic frame as a JSON message every ~250ms: `{type: "frame", jpeg_b64, timestamp, fps}` — the frame is captured fresh by the existing capture pipeline (not a new capture worker), encoded as JPEG, base64-encoded
>    - Pushes a heartbeat every 5 seconds: `{type: "heartbeat", server_time, camera_status, fps}`
>    - On client disconnect, stops cleanly. No worker leaks.
>
>    **Important:** do NOT spin up a new capture process per WebSocket client. Hook into the existing P8 capture pipeline's event bus. If no event bus exists, build a minimal one in `backend/hadir/capture/event_bus.py` — an in-process pub/sub keyed by camera_id that the existing capture worker publishes to and that this WebSocket subscribes to. Document this in `backend/CLAUDE.md`.
>
> 2. **Stats endpoint.** Add `GET /api/cameras/{camera_id}/live-stats` that returns the rolling counters: `{detections_last_10m, known_count, unknown_count, fps, status}`. Computed from `detection_events` over the last 10 minutes. Used by the header status strip; refreshed every 10 seconds, not on every WebSocket message (don't bloat the message stream).
>
> 3. **Audit:** every WebSocket connection writes an audit entry on open and close (`live_capture_subscribe` / `live_capture_unsubscribe`) with camera_id, actor, duration. Don't audit individual frames or events — that explodes the audit log.
>
> 4. **Throttling:** maximum 5 concurrent live subscribers per tenant per camera. Beyond that, return WebSocket close code 1013 (Try Again Later). This protects CPU during busy demos.
>
> 5. **Tests:** pytest covering: WebSocket connects with valid session, rejects without session, rejects for cross-tenant camera access, receives heartbeat within 6 seconds, frames carry JPEG bytes that decode, detection messages have valid bbox structure.
>
> ### Frontend work (~2.5 hours)
>
> 1. **New file `frontend/src/pages/LiveCapture/LiveCapture.tsx`** (or whatever the project's page directory pattern is — match what exists for Daily Attendance or Camera Logs). Replace whatever currently renders for the `/live` route.
>
> 2. **Hooks:**
>    - `useLiveCameras()` — TanStack Query, polls `/api/cameras` every 30s for the list
>    - `useLiveStream(cameraId)` — manages the WebSocket lifecycle. Returns `{frame, events, heartbeat, status, pause, resume, reconnect}`. Internally maintains a rolling buffer of the last 50 events. Closes on unmount.
>    - `useLiveStats(cameraId)` — TanStack Query, polls `/api/cameras/{id}/live-stats` every 10s
>
> 3. **Video rendering:** the streamed JPEG bytes are drawn to a `<canvas>` element. Bounding boxes are rendered on top of (or alongside, in a sibling layer) the canvas using SVG positioned absolutely. Use the bbox coordinates from the detection messages — they're in the original camera frame's pixel space, so scale them to the canvas dimensions.
>
>    Box color rules:
>    - confidence >= match threshold AND employee_id present → success-green from design tokens
>    - confidence < match threshold OR employee_id null → warning-amber
>
>    Label format: `{name} · {confidence}%` for known, `Unknown · {confidence}%` for unknown. Position label above the box (or below if the box is at the top of the frame).
>
> 4. **Event stream table:** uses the rolling buffer from `useLiveStream`. New events appear at the top with a brief highlight animation (200ms fade). Unknown faces show a "?" avatar; known faces show initials (e.g. "FK" for Fatima Al-Kindi). The "Only unknown" toggle filters the displayed list (does not change subscription).
>
> 5. **Pause/Resume:** Pause stops rendering frames to the canvas but keeps the WebSocket open and keeps appending events to the buffer. The Pause button label and icon flip to "Resume" while paused. Reconnect closes and re-opens the WebSocket; shows a "Reconnecting…" status during.
>
> 6. **Export last hour:** calls a new endpoint `GET /api/cameras/{id}/events.csv?hours=1`. Streams CSV with columns: time, camera_code, employee_code, employee_name, confidence, status. Implementation should reuse the same query path as Camera Logs export from P11.
>
> 7. **Empty / error states:** "Select a camera to begin" when no camera selected. "Camera offline — last seen X ago" when the WebSocket reports offline. "Reconnecting…" with a spinner during transient disconnects.
>
> 8. **Responsive:** the page is desktop-first (BRD NFR-USA-003). On viewports under 1024px, stack the camera list above the viewer and event stream below. Don't make this perfect on mobile — viewing live face detection on a phone is not a real use case.
>
> 9. **Match the design archive exactly.** Spacing, typography (Instrument Serif for the page title, Inter for body, JetBrains Mono for the timestamp overlays), colors via CSS variables only, no Tailwind. If `design-reference/pages.jsx` has a Live Capture component, port it verbatim — don't redesign.
>

> ### Performance budgets
>
> - Frame JPEG quality 70, max 1280×720, base64 overhead acceptable for LAN. Document in code comments that this is tuned for LAN-internal use; if exposed externally we'd switch to MJPEG-over-HTTP or WebRTC.
> - Backend WebSocket message rate per client: max ~10 fps frames + event-driven detections. CPU budget: ≤ 15% of one core per active subscriber on the dev box.
> - Frontend canvas redraw on requestAnimationFrame, not per-message — coalesce frames if they arrive faster than the screen refreshes.
