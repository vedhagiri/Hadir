# P28.5 — Live Capture, capture-control split, and detection config

**Status:** Pre-Omran validation. Pre-P29 cutover. Three sequential sub-phases.

**Source material:** `prototype-reference/backend/` — frozen working code Maugood grew from. Specifically:
- `capture.py` — reader/analyzer split, motion-skip optimisation, per-camera preview JPEG, IoU tracker integration
- `tracker.py` — IoU tracker with max-duration cap
- `detectors.py` — DetectorConfig, two modes, quality scoring, hot re-prep
- `known_people.py` — per-photo embedding cache
- `main.py` — MJPEG endpoint pattern, settings endpoint, job tracker
- Frontend's live preview component

**Architecture truth from reading the prototype:**

The prototype's `CameraWorker` is *already* the right shape — two threads per camera (reader + analyzer), per-camera `latest_jpeg`, motion-skip, configurable knobs. The only thing the prototype's `CaptureManager` does wrong for v1.0 is run **one camera at a time**. Everything else ports cleanly.

So the work is:
- **Bring the `CameraWorker` design from prototype to v1.0** (or align v1.0's existing P8 worker to it)
- **Make `CaptureManager` run N workers concurrently**, keyed by `camera_id`
- **Wrap each access in tenant scoping** — `camera_id` resolves to a tenant; preview/events/control all check that
- **Build the v1.0 frontend Live Capture page** that calls the per-camera MJPEG endpoint
- **Add the worker/display split and the per-camera capture knobs** (P28.5b)
- **Add the system-wide detection/tracker config** (P28.5c)

**Estimated total time:** 10–13 hours across three sub-phases.

> ### Part B — Live Capture page
>
> #### Architectural decisions (read carefully before writing code)
>
> The prototype's `CameraWorker` is the correct design and v1.0's P8 capture pipeline should match it. Specifically:
>
> 1. **Per-worker `latest_jpeg`** — each worker maintains its own preview JPEG, updated by the reader thread on every native frame and overwritten by the analyzer thread when fresh detections are available. **This already exists in the prototype** — port the pattern. There is no need to invent a separate `FrameBuffer` class; the `latest_jpeg` slot on each worker IS the buffer.
>
> 2. **Two threads per camera** — reader runs at native fps (smooth preview), analyzer runs up to 6 fps with motion-skip. Port the entire pattern from `prototype-reference/backend/capture.py:CameraWorker` (~lines 125-562). Adapt to v1.0 conventions (logging, audit, tenant context) but **do not change the threading model or the motion-skip logic.** Those are tuned.
>
> 3. **CaptureManager runs N workers, not 1.** This is the only structural change from the prototype. Replace:
>    ```python
>    self.current_worker: Optional[CameraWorker] = None
>    ```
>    with:
>    ```python
>    self.workers: dict[tuple[int, int], CameraWorker] = {}  # (tenant_id, camera_id) → worker
>    ```
>
>    Workers are added when their camera is enabled (P28.5b separates worker/display, but for now treat existing `enabled` as worker-enable). Removed when disabled or deleted.
>
> 4. **Compare to existing v1.0 P8 capture.** If P8's code already follows this design, fold the prototype's improvements (motion-skip, reader/analyzer split, per-worker preview JPEG) into it. If P8's code is structurally different, **replace it with the prototype's design.** This is a real refactor — the prototype's design is proven; deviating costs us. Document the decision in `docs/phases/P28.5a.md` so it's clear what was changed and why.
>
> 5. **Tenant scoping** — `CaptureManager.get_preview(tenant_id, camera_id)` validates that `(tenant_id, camera_id)` is a real entry in its dict; returns None otherwise. Every endpoint that consumes the preview JPEG resolves the camera through the session's tenant first. **Cross-tenant camera ID guess must return 404, never bytes.** The P5 isolation suite covers this; rerun it after this change.
>
> #### Backend endpoints
>
> All in `backend/maugood/live_capture/router.py`. Admin role required for v1.0 (Manager/HR access deferred).
>
> 1. **`GET /api/cameras/{camera_id}/live.mjpg`** — multipart MJPEG stream of `latest_jpeg` from the worker for that camera. Pattern from `prototype-reference/backend/main.py:/api/stream/preview` lines 263-280, adapted for tenant scoping:
>    ```python
>    boundary = b"--frame"
>    idle_ticks = 0
>    while True:
>        jpeg = capture_manager.get_preview(tenant_id, camera_id)
>        if jpeg is None:
>            idle_ticks += 1
>            if idle_ticks > 100:  # ~5 seconds without preview = give up
>                break
>            await asyncio.sleep(0.05)
>            continue
>        idle_ticks = 0
>        yield boundary + b"\r\n"
>        yield b"Content-Type: image/jpeg\r\n"
>        yield f"Content-Length: {len(jpeg)}\r\n\r\n".encode()
>        yield jpeg + b"\r\n"
>        await asyncio.sleep(0.04)  # ~25 fps cap
>    ```
>
>    Wrap in `try/finally` with audit-log on subscribe + unsubscribe. **Do not audit per-frame** — that floods the audit log.
>
> 2. **`GET /api/cameras/{camera_id}/events.ws`** — WebSocket for live JSON event stream. Sends:
>    - `{type: "detection", time, employee_id, employee_name, employee_code, confidence, status, bbox, track_id}` per new detection
>    - `{type: "track_closed", time, track_id, event_id, faces_saved, duration_sec}` per track retirement
>    - `{type: "heartbeat", server_time, status, fps_reader, fps_analyzer, motion_skipped}` every 5 seconds — pull stats from `worker.get_stats()`
>    - `{type: "stats", detections_last_10m, known_count, unknown_count}` every 30 seconds
>
>    Hooks into the worker's existing event flow. If P8/P9 don't already publish events to a pub/sub, add a minimal asyncio-queue pub/sub in `backend/maugood/capture/event_bus.py`. Worker writes to it after each detection cycle. WebSocket subscribes per `(tenant_id, camera_id)`.
>
> 3. **`GET /api/cameras/{camera_id}/events.csv?hours=1`** — last hour of detection events as CSV. Reuse Camera Logs export query path from P11.
>
> 4. **`GET /api/cameras/{camera_id}/live-stats`** — `{detections_last_10m, known_count, unknown_count, fps_reader, fps_analyzer, status}` from a quick SQL + worker.get_stats(). Polled every 10 seconds by frontend.
>
> #### Concurrency
>
> - Up to 10 concurrent MJPEG viewers per camera per tenant. Beyond → 503. (No per-viewer capture cost since they all read the same `latest_jpeg`.)
> - Up to 10 concurrent WebSocket subscribers per camera per tenant. Same.
> - If the worker for that camera isn't running (camera disabled, RTSP failed, worker crashed) → MJPEG returns 503 `{detail: "camera_unavailable"}` after a 2-second grace.
