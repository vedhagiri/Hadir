# Hadir v1.0 — M2 Test Accounts & Procedures

**Purpose:** Reference document for testing Hadir at the end of P5 and through M2. Lists every test account, what each is for, and the test procedures that exercise them.

**Scope:** v1.0 multi-tenant build, post-P5 (two tenants provisioned, isolation suite green). Updated as new phases land.

**Last updated:** End of P5.

## 3. The cross-tenant honeypot

This is the single most important test setup in P5. Both tenants get an employee with **identical name** but different codes:

- Omran: `OM0099`, full_name `"Test Crossover"`, email `crossover@omran.test`
- Demo Co: `DC0099`, full_name `"Test Crossover"`, email `crossover@demo.test`

If a search by name from inside Omran ever returns the Demo Co row (or vice versa), tenant isolation is broken. The automated test in P5 should already cover this; the manual procedure in Section 6.5 below is your sanity check.

## 4. Password convention (dev only)

`Hadir!2026` for all tenant accounts, `Superadmin!2026` for staff. Reasons:

- Easy to type during demos
- Meets the minimum strength rules from pilot P3 (length, mixed case, digit, symbol)
- Visibly throwaway — no one will mistake them for production credentials
- Identical across roles within a tenant so the only thing you change between sessions is the email

**Production accounts will get strong, unique passwords during P29 cutover, set by the user on first OIDC sign-in or via the password-reset flow. None of these test passwords ship to production.**

## 5. Seed script

Create `backend/scripts/seed_test_accounts.py`. Run it after both tenants are provisioned. The script is idempotent — it skips users that already exist by email.

**Prompt for Claude Code (run as a small follow-on to P5):**

> Create `backend/scripts/seed_test_accounts.py` that seeds the full test account matrix from `docs/testing/v1.0-m2-test-accounts.md`. The script:
>
> 1. Takes `--tenant` (slug) as an arg, or `--all` to run for both `tenant_omran` and `tenant_demo`.
> 2. Reads the account matrix from a Python constant at the top of the file (don't fetch from the markdown — the markdown is the human reference, the constant is the source of truth for the script).
> 3. For each user:
>    - Argon2-hashes the password
>    - Inserts the user with `is_active=true`
>    - Assigns roles via `user_roles` (resolves role IDs by code)
>    - Assigns departments via `user_departments` (resolves department IDs by code; creates the Engineering and Operations departments if they don't exist)
>    - Skips silently if a user with that email already exists in this tenant (idempotent)
> 4. Also seeds the corresponding Employee records (`OM0001`–`OM0006`, plus the `OM0099` honeypot, plus the `DC0001`–`DC0003` and `DC0099` for Demo Co) with names matching the matrix. Email-matches User to Employee where the User email and Employee email align.
> 5. Also seeds a separate Super-Admin section in `public.mts_staff` for `superadmin@mts.test` and `superadmin2@mts.test`.
> 6. Logs a summary at the end: created count, skipped count, errors. No plain-text passwords in the log.
>
> Run it locally to populate both tenants, verify the user counts, commit as `chore: seed M2 test accounts`. Stop and show me the output.

Once the script runs cleanly, the rest of this document assumes those accounts exist.

## 6. Test procedures by capability

These are the manual tests to run after P5 to validate the build. Re-run the relevant ones after every later phase that touches the same area.

### 6.1 Login + role-aware navigation (validates P3, P4, P7-pending)

**Steps:**
1. Open two browser windows side-by-side.
2. In window A: navigate to the Omran login page, log in as `admin@omran.test`.
3. In window B: log in as `admin@demo.test`.
4. Compare sidebars — they should look the same structurally but with different branding (teal vs navy + Lato font).
5. Log out of A, log in as `hr@omran.test`. Sidebar should drop the Admin-only items (Audit Log, System, Cameras, Manager Assignments, Custom Fields, Branding, Authentication settings).
6. Log out, log in as `manager.eng@omran.test`. Sidebar should be smaller still — Approvals, Daily Attendance, Reports, Dashboard.
7. Log out, log in as `employee.eng1@omran.test`. Smallest sidebar — My Attendance, My Requests, Dashboard.

**Expected:**
- Each role sees only its own nav items
- No 403s when clicking any nav item available to that role
- Direct-URL access to a hidden item (e.g. an Employee navigating to `/cameras`) returns 403, not the page

### 6.2 Tenant isolation — manual sanity check (validates P1, P2, P5)

**Steps:**
1. Window A: logged in as `admin@omran.test`.
2. Window B: logged in as `admin@demo.test`.
3. In A, go to Employees. Note the count and the codes (should be `OM*`).
4. In B, go to Employees. Note the count and the codes (should be `DC*`).
5. In A, search for "Crossover". Should return Omran's `OM0099` only.
6. In B, search for "Crossover". Should return Demo Co's `DC0099` only.
7. In A, try to access Demo Co employee directly by guessing IDs. Open the browser dev tools network tab, find an employee detail request like `/api/employees/123`, copy the request, change the ID to one you saw in window B. **Should return 404 or 403, never the row.**
8. Repeat the URL-guess test for departments, audit log entries, attendance records.

**Expected:**
- Counts and codes are tenant-distinct
- Cross-tenant ID guesses fail closed
- No row leakage in any list, search, or detail endpoint

### 6.3 Multi-role user (validates P7 once it lands; for now verify max-role behaviour)

**Pre-P7 (current state):** `dual.role@omran.test` logs in and sees the HR sidebar (HR > Manager in priority).

**Post-P7:** the same user logs in, sees a role chip in the topbar, can switch to Manager and the sidebar updates. Audit log shows the switch.

### 6.4 Multi-department manager (validates BRD FR-AUTH-008)

**Steps:**
1. Log in as `manager.multi@omran.test` (assigned to Engineering AND Operations).
2. Daily Attendance page should show employees from BOTH departments.
3. Compare to `manager.eng@omran.test` (Engineering only) — should see fewer rows.
4. Once approval workflow lands (P15), `manager.multi@omran.test` should receive approval requests from both departments.

### 6.5 Super-Admin cross-tenant access (validates P3)

**Pre-P3:** verify via DB only.
- Confirm `mts_staff` table has both Super-Admin entries.
- Confirm passwords are Argon2 hashes.

**Post-P3:**
1. Navigate to `/super-admin/login` (NOT the tenant login).
2. Log in as `superadmin@mts.test`.
3. Tenant list should show both `tenant_omran` and `tenant_demo` with their employee counts.
4. Click "Access as" on Omran. Red banner should appear at the top reading "Viewing as SuperAdmin — click to exit impersonation". Sidebar should be the Omran admin sidebar.
5. Make a benign change (e.g. edit a department description).
6. Exit impersonation via the banner.
7. "Access as" Demo Co. The benign change from step 5 must NOT appear here.
8. Exit impersonation.
9. Log in as `admin@omran.test` — go to Audit Log. The Super-Admin "Access as" event AND the benign change should both be visible to the tenant Admin (with the actor clearly marked as Super-Admin, not impersonated-as-tenant-admin).
10. Verify the same in `public.super_admin_audit` directly via SQL.

### 6.6 Audit log append-only (validates pilot P2 and continued enforcement)

**Steps (run as the `hadir_app` DB user, not `hadir_admin`):**

```sql
-- Should fail with permission denied:
UPDATE audit_log SET action = 'tampered' WHERE id = 1;

-- Should also fail:
DELETE FROM audit_log WHERE id = 1;

-- Should succeed:
INSERT INTO audit_log (tenant_id, actor_user_id, action, entity_type, created_at)
VALUES (1, 1, 'manual_test', 'test', now());
```

If either of the first two succeed, the grant has been broken — check what migration introduced the regression. This is a P0 bug.

### 6.7 RTSP credential never in plain text (validates pilot P7)

**Steps:**
1. Log in as `admin@omran.test`.
2. Cameras page → add a new camera with URL `rtsp://testuser:supersecret@192.168.1.50:554/stream`.
3. Reload the cameras list. The URL field should be hidden or show `***`.
4. Edit the camera. The URL field should be empty/placeholder, NOT pre-filled with the password.
5. SSH to the dev box, run:
   ```bash
   docker compose logs backend | grep -i 'rtsp://'
   docker compose logs backend | grep 'supersecret'
   psql -d hadir -c "SELECT rtsp_url_encrypted FROM tenant_omran.cameras LIMIT 1;"
   ```
6. The grep commands should return **nothing**. The SQL should return Fernet-encrypted bytes that look like base64, not a URL.

If any of those leaks the password — P0 bug.

### 6.8 Face crops encrypted on disk (validates pilot P6)

**Steps:**
1. Log in as `admin@omran.test`.
2. Upload a photo for `OM0001`.
3. SSH to the dev box, find the file:
   ```bash
   ls -lah /data/faces/1/OM0001/front/
   file /data/faces/1/OM0001/front/*.jpg
   ```
4. `file` should NOT report the file as a JPEG — it should be unrecognised binary (the Fernet ciphertext).
5. Try to open it in an image viewer — should fail.

If the file opens as a normal image — P0 bug, encryption isn't being applied.

### 6.9 Full attendance flow (smoke test, post-P9)

**Steps:**
1. Log in as `admin@omran.test`.
2. Make sure `OM0001` (`employee.eng1@omran.test`) has front-angle photo enrolled.
3. Verify a camera is active in System page.
4. Walk the dev-box camera (or play a recording) showing `OM0001`'s face.
5. Wait ~30 seconds.
6. Camera Logs → see the detection event with `OM0001` identified, confidence > threshold.
7. Wait up to 15 minutes (or trigger manual recompute).
8. Daily Attendance for today → see `OM0001` row with `in_time` set.
9. Log out.
10. Log in as `manager.eng@omran.test` → Daily Attendance → see the same row (engineering scope).
11. Log in as `manager.ops@omran.test` → should NOT see `OM0001` (different department).
12. Log in as `employee.eng1@omran.test` → My Attendance → see own record only.

### 6.10 Approval workflow happy + reject paths (post-P13–P16)

**Pre-P13:** can't run yet.

**Post-P16, full matrix:**

| Path | Submit as | Manager action | HR action | Admin action | Final state |
|---|---|---|---|---|---|
| Manager approves, HR approves | `employee.eng1@omran` | `manager.eng@omran` approves | `hr@omran` approves | — | `hr_approved` |
| Manager approves, HR rejects | `employee.eng1@omran` | `manager.eng@omran` approves | `hr@omran` rejects | — | `hr_rejected` |
| Manager rejects (terminal) | `employee.eng1@omran` | `manager.eng@omran` rejects | (should not appear in HR queue) | — | `manager_rejected` |
| Admin override on HR rejection | `employee.eng1@omran` | `manager.eng@omran` approves | `hr@omran` rejects | `admin@omran` overrides to approve, with mandatory comment | `admin_approved` |
| Employee cancels before manager decision | `employee.eng1@omran` | (none — employee cancels first) | — | — | `cancelled` |

For each path, verify:
- Right people see the request in their inbox at the right time
- Wrong people do not (e.g. `manager.ops@omran` should never see an Engineering employee's request)
- Audit log captures each transition
- Notifications fire (post-P20)
- Attendance recomputes on approval if the request is for a past date (post-P13)

### 6.11 Wrong-department Manager (negative scope test)

**Steps:**
1. `employee.eng1@omran.test` submits an exception request.
2. Log in as `manager.ops@omran.test` (Ops manager, NOT Engineering).
3. The request should NOT appear in their Approvals queue.
4. Open browser dev tools, find the request ID from window A, try to POST `/api/requests/{id}/manager-decide` directly. **Should return 403.**
5. Audit log should record the failed attempt.

### 6.12 Branding isolation (validates P4)

**Steps:**
1. Log in as `admin@omran.test`. UI should be teal.
2. Log in as `admin@demo.test` in another window. UI should be navy with Lato font.
3. Open both windows full-screen, click between them. The accent color, button styles, and font should differ visibly.
4. Open dev tools → inspect the inline `<style>` block for the CSS variables. Omran's `--accent` should be the teal OKLCH; Demo Co's should be navy.
5. Generated reports (P17 onwards) should respect the tenant's branding in the PDF letterhead.

## 7. DB-only verification (when no UI is available)

Some checks are easier or only possible at the DB layer. Use the `hadir_admin` DB user for read-only inspection (don't write).

### 7.1 List tenants

```sql
SELECT id, slug, name, created_at FROM public.tenants;
```

After P5, expect two rows.

### 7.2 List schemas

```sql
SELECT schema_name FROM information_schema.schemata
WHERE schema_name LIKE 'tenant_%' OR schema_name = 'public' OR schema_name = 'main';
```

Multi mode after P5: `tenant_omran`, `tenant_demo`, `public`. Single mode (pilot): `main`, `public`.

### 7.3 Verify user count per tenant

```sql
SELECT 'omran' AS tenant, count(*) FROM tenant_omran.users
UNION ALL
SELECT 'demo' AS tenant, count(*) FROM tenant_demo.users;
```

After running the seed script: 10 for Omran, 7 for Demo Co (matrix above).

### 7.4 Verify password is hashed, not plaintext

```sql
SELECT email, substring(password_hash, 1, 30) FROM tenant_omran.users LIMIT 3;
```

Should start with `$argon2id$v=19$...`. Never with the plaintext password.

### 7.5 Verify role assignments

```sql
SELECT u.email, r.code
FROM tenant_omran.users u
JOIN tenant_omran.user_roles ur ON ur.user_id = u.id
JOIN tenant_omran.roles r ON r.id = ur.role_id
ORDER BY u.email, r.code;
```

Compare against the matrix in Section 2.2.

### 7.6 Verify Super-Admin staff

```sql
SELECT email, created_at FROM public.mts_staff;
```

Two rows after seeding.

### 7.7 Confirm append-only grants on audit_log

```sql
SELECT grantee, privilege_type
FROM information_schema.table_privileges
WHERE table_name = 'audit_log' AND grantee = 'hadir_app';
```

Should show INSERT and SELECT only — no UPDATE, no DELETE.

## 9. Maintenance notes

- **Seed script must stay idempotent.** As more phases add tables, the seed script must keep working without drops. If it ever requires a clean DB to run, that's a regression.
- **Add new test accounts here, not ad-hoc.** Every new account needs a row in Section 2 with a clear purpose. "Test accounts the team made up during a session" rot fast.
- **Production NEVER uses these credentials.** During P29 cutover, the seed script must be excluded from the production deploy. Document this in `docs/deploy-production.md` when P23 lands.
- **Honeypot stays.** Don't delete `OM0099` / `DC0099` — they're load-bearing for the manual isolation check.
